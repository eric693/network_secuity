風險評估概述與法規依據

資訊安全風險評估（Information Security Risk Assessment）是對組織資訊資產所面臨的威脅、脆弱性及其可能造成的損失進行系統性識別、分析和評價的過程。它是制定安全策略、確定投資優先級的核心依據。

為什麼需要風險評估？

企業面臨的網路威脅持續增加，但安全預算始終有限。風險評估回答三個核心問題：我們有哪些資產？、它們面臨什麼威脅？、我們應該優先保護什麼？ 沒有系統性評估，安全投入往往是盲目的。

GB/T 20984-2022 核心要點

2022 年最新修訂版在 2007 年版基礎上，增加了對供應鏈風險和雲計算環境的評估要求，引入了風險場景概念（將威脅與脆弱性組合為具體攻擊場景），並強化了殘餘風險管理要求。同時新增了對數據安全風險（個人信息、重要數據）的評估指引，與《數據安全法》和《個人信息保護法》的要求相銜接。

風險評估流程與角色分工

一個完整的風險評估項目通常歷時 2-6 週，涉及評估方、被評估方多個角色協作。明確的流程和責任分工是項目成功的前提。

風險評估完整流程

階段 1
項目啟動與準備
簽訂服務合同和保密協議；確定評估範圍（系統邊界、業務範圍）；成立評估工作組；制定項目計劃（時間表、里程碑）；收集背景信息（組織架構、業務流程、現有安全制度）。
階段 2
資產調研與識別
通過問卷、訪談、現場調研收集資產清單；對資產進行分類（硬體/軟體/數據/服務/人員）；評估資產重要性（賦值）；建立資產清冊，作為後續評估的基礎。
階段 3
威脅識別
識別可能影響資產的威脅來源（自然災害、人為惡意、系統故障）；評估威脅主體能力（外部黑客、內部人員、供應商）；評估每類威脅的發生可能性（頻率）。
階段 4
脆弱性評估（技術+管理）
技術脆弱性：漏洞掃描、配置基線核查、滲透測試；管理脆弱性：安全制度審查、員工訪談、安全意識評估；評估每個脆弱性的嚴重程度（賦值）。
階段 5
風險計算與分析
根據風險計算模型（資產價值 × 威脅概率 × 脆弱性嚴重度）計算每個風險點的風險值；對照風險等級判定標準劃分高/中/低/極高等級；識別關鍵風險點。
階段 6
風險處置規劃
針對每個風險點提出處置建議（接受/規避/轉移/降低）；制定整改優先級和路線圖；評估整改成本與效益；確認殘餘風險。
階段 7
報告輸出與評審
撰寫風險評估報告（技術報告+管理摘要）；組織評審會議，與客戶確認評估結論；根據客戶反饋修訂報告；正式交付並存檔。

角色分工

評估方（甲方委託的安全公司）
項目負責人：統籌協調、客戶溝通、報告審核；技術評估人員：漏洞掃描、配置核查、滲透測試；文檔分析人員：制度審查、訪談記錄、風險計算；報告撰寫人員：整理發現、撰寫報告、製作圖表。
被評估方（甲方/客戶）
業務負責人：提供業務背景、確認資產重要性、審核評估結果；IT 負責人：提供技術環境信息、協調系統訪問授權、確認技術發現；安全負責人：參與評估過程、確認安全措施現狀；相關業務部門：配合問卷調查和訪談。

資產識別與分類分級

資產（Asset）是對組織有價值的任何事物。準確的資產識別是風險評估的基石——無法識別的資產，其風險也無法被管理。

資產分類體系（GB/T 20984-2022）

資產類型典型示例評估重點
數據資產（最核心）客戶個人信息、財務數據、業務數據、配置文件、源代碼、密鑰憑證機密性要求（是否含個人信息/商業秘密）；完整性要求（篡改影響）；可用性要求（宕機損失）
軟體資產作業系統、資料庫軟體、中間件、業務應用系統、開發工具、安全軟體版本及補丁狀態；授權合規性；開源組件安全
硬體資產伺服器、網路設備（路由器/交換機/防火牆）、終端設備、物聯網設備、存儲設備物理安全、生命週期、冗餘配置
服務資產對外提供的業務系統、API 服務、雲服務、網路連接服務（VPN/互聯網出口）服務可用性 SLA；對外暴露面；依賴關係

物理環境機房、辦公場所、電力供應、空調系統、門禁系統物理訪問控制；環境安全

資產賦值方法

資產賦值從機密性（C）、完整性（I）、可用性（A）三個維度進行評估，每個維度評分 1-5，取最高值作為資產重要性等級。

資產賦值標準（CI A 三維度）
機密性（Confidentiality）賦值標準：
  5 — 最高機密：洩露會導致嚴重法律後果或重大財務損失
      例：銀行客戶完整金融數據、國家秘密
  4 — 秘密：洩露嚴重影響業務，涉及大量個人敏感信息
      例：用戶密碼/身份證號/手機號批量數據
  3 — 內部：洩露對競爭對手有利或影響合作關係
      例：業務策略、合同報價、員工薪資
  2 — 限制：洩露影響較小，僅內部使用
      例：內部操作手冊、IT 資產清單
  1 — 公開：可對外公開的信息

完整性（Integrity）賦值標準：
  5 — 篡改導致直接重大財務損失或人身危害
      例：財務轉帳記錄、醫療記錄、工控系統指令
  4 — 篡改嚴重影響業務決策或導致重大業務中斷
      例：交易記錄、合同文本
  3 — 篡改需要大量資源恢復，影響業務效率
  2 — 篡改有一定影響，但可較快恢復
  1 — 篡改影響很小或可輕易發現恢復

可用性（Availability）賦值標準：
  5 — 7×24 連續運行，中斷 <5 分鐘導致嚴重損失
      例：交易系統、核心業務系統
  4 — 工作時間必須可用，中斷 <1 小時可接受
      例：ERP、OA 系統
  3 — 每天需要數小時可用，中斷 <4 小時可接受
  2 — 不要求連續可用，定期使用即可
  1 — 偶發性使用，長時間中斷也可接受

資產重要性 = MAX(C, I, A)
# 示例：核心業務資料庫  C=5, I=5, A=5 → 重要性等級 5（極高）
# 示例：公告網站         C=1, I=3, A=3 → 重要性等級 3（中等）

威脅識別與脆弱性分析

威脅（Threat）是可能對資產造成損害的潛在事件，脆弱性（Vulnerability）是資產中存在的可被威脅利用的弱點。兩者共同決定風險的可能性。

威脅分類與賦值

威脅類別主要威脅示例頻率賦值參考
網路攻擊外部入侵、DDoS 攻擊、Web 漏洞利用（SQL 注入/XSS）、釣魚郵件、勒索軟件互聯網暴露系統：4-5（高頻）；內部系統：2-3
內部威脅員工誤操作、離職員工資料洩露、特權濫用、數據竊取大型組織：3-4；小型組織：2-3
軟件缺陷已知漏洞未修補、0-day 漏洞、配置錯誤、代碼漏洞取決於修補管理成熟度：1-5
供應鏈風險第三方軟件後門、供應商訪問越權、開源組件漏洞（如 Log4Shell）近年趨勢明顯：3-4
物理安全未授權人員進入機房、設備被盜、硬件篡改一般辦公環境：1-2；高風險行業：3
環境與自然電力中斷、水災、火災、地震、設備故障取決於地理位置和基礎設施：1-3

脆弱性識別方法

01
技術脆弱性掃描 — 使用漏洞掃描工具（Nessus/Qualys/OpenVAS）自動發現已知 CVE 漏洞、系統補丁缺失、服務配置問題。針對 Web 應用使用 AWVS/Burp 進行掃描。評估結果依據 CVSS 評分進行嚴重程度賦值。
02
配置基線核查 — 對照安全基線標準（如 CIS Benchmarks、等保基線）逐項核查系統配置，包括帳號策略、服務開啟狀態、文件權限、審計日誌設置等。記錄每個不符合項作為脆弱性發現。
03
滲透測試驗證 — 對掃描發現的高危漏洞進行手工驗證，確認可利用性。發現掃描工具無法自動識別的邏輯漏洞（越權、業務邏輯缺陷）。提供真實攻擊場景下的脆弱性評估。
04
訪談與問卷 — 對管理人員和技術人員進行結構化訪談，了解安全管理制度執行情況、員工安全意識水平、安全事件歷史。識別管理層面的脆弱性（如無安全培訓、無事件響應流程）。
05
文件審查 — 審查安全相關文件：安全策略、操作規程、系統建設文件、網絡拓撲圖。對比文件要求和實際執行情況，識別制度和執行之間的差距。
小測驗
某銀行核心業務系統資料庫，機密性評分為 5（含大量客戶金融數據），完整性評分為 5（篡改導致直接財務損失），可用性評分為 4（工作時間必須可用）。按 GB/T 20984-2022 的資產賦值方法，該資料庫的資產重要性等級為？
A4（取三個值的平均：(5+5+4)/3 ≈ 4.7，四捨五入）
B4.7（保留一位小數的平均值）
C5（取三個維度中的最高值）
D14（三個維度值相加的總和）
正確！GB/T 20984-2022 規定，資產重要性等級取機密性、完整性、可用性三個維度賦值中的最高值，而非平均值或總和。這樣做的邏輯是：只要資產在任何一個維度上有極高要求，整個資產就應被視為高重要性資產進行保護。此例中 MAX(5,5,4) = 5，該資料庫為最高等級資產，需要最嚴格的安全控制措施。

網路設備安全評估

網路設備（路由器、交換機、防火牆、負載均衡器）是組織網路的基礎設施，其安全配置直接影響整體防護能力。評估時需兼顧技術核查和配置審查兩個層面。

評估檢查項目

評估類別具體檢查項風險說明
帳號管理是否存在默認帳號/密碼；特權帳號數量是否最小化；是否啟用帳號鎖定策略默認憑證是最常見的設備入侵途徑
訪問控制管理訪問是否限制特定 IP；是否禁用不安全管理協議（Telnet、HTTP）；SNMPv1/v2 是否已停用Telnet 明文傳輸、SNMPv1/v2 弱認證可被竊聽
協議安全路由協議是否啟用認證（OSPF MD5/SHA）；BGP 是否配置 MD5 認證；STP 是否啟用 BPDU Guard未認證路由協議可被偽造注入惡意路由
日誌審計是否啟用 Syslog 並集中轉發；日誌是否覆蓋登入/配置變更/異常流量；NTP 時間同步是否正確無日誌則安全事件無法溯源

主機作業系統安全評估

伺服器主機（Windows Server / Linux）是業務系統的運行基礎，主機安全評估需覆蓋系統配置、帳號管理、補丁狀態、服務開放、審計設置等核心項目。

Windows Server 評估重點
帳號安全：本地管理員帳號、禁用 Guest、密碼策略（最小長度 8 位、複雜度要求、最長使用期）；補丁狀態：Windows Update 策略、最近一次更新時間；服務：不必要服務停用（列印假脫機/Telnet/NetBIOS）；審計：安全事件日誌設置、Sysmon 是否部署；UAC/防火牆狀態。
Linux 評估重點
帳號安全：/etc/shadow 密碼策略（minlen/maxdays）、UID=0 帳號數、sudo 配置；SSH 安全：是否禁用 root 直接登入、金鑰認證、已知弱演算法；補丁：未更新套件數量；SUID/SGID 文件；SELinux/AppArmor 狀態；關鍵文件完整性。
Linux 主機安全評估腳本（核查項）
#!/bin/bash
# ─── Linux 主機安全評估核查腳本（需 root 權限）───

echo "===== 1. 帳號安全 ====="
# 查找 UID=0 的非 root 帳號（安全風險！）
awk '-F:' '($3==0){print "UID=0帳號:", $1}' /etc/passwd
# 查找可登入的系統帳號
awk '-F:' '($7!~/nologin|false/){print "可登入帳號:", $1, "shell:", $7}' /etc/passwd
# 密碼策略核查
grep -E 'PASS_MAX_DAYS|PASS_MIN_DAYS|PASS_MIN_LEN|PASS_WARN_AGE' /etc/login.defs

echo "===== 2. SSH 配置 ====="
grep -E 'PermitRootLogin|PasswordAuthentication|PubkeyAuthentication|Protocol|AllowUsers|MaxAuthTries' /etc/ssh/sshd_config
# 應確認：PermitRootLogin no | PasswordAuthentication no（使用金鑰）

echo "===== 3. 補丁狀態 ====="
# CentOS/RHEL
yum check-update --security 2>/dev/null | grep -c 'Security'
# Ubuntu/Debian
apt-get -s upgrade 2>/dev/null | grep -c '^Inst'

echo "===== 4. SUID/SGID 文件 ====="
# 查找不常見的 SUID 文件（可能為後門）
find / -perm -4000 -type f 2>/dev/null | grep -v '/usr/bin\|/usr/sbin\|/bin\|/sbin'

中間件與 Web 應用評估

中間件（Nginx、Apache、Tomcat、Spring Boot）和 Web 應用是對外攻擊面最大的組件，也是風險評估中技術脆弱性最集中的領域。

01
版本與補丁 — 確認 Nginx/Apache/Tomcat 版本，對照 CVE 數據庫（或 Vulhub）查詢已知高危漏洞。重點關注：Nginx 的 HTTP Request Smuggling、Apache Log4j2（Log4Shell）、Tomcat 的反序列化漏洞（CVE-2017-12617）、Spring Framework（Spring4Shell）。
02
敏感信息洩露 — 錯誤頁面是否洩露版本/路徑信息（應配置自訂錯誤頁）；是否存在 .git/.svn 目錄洩露源碼；備份文件洩露（.bak/.zip/.tar.gz）；調試信息是否在生產環境開啟（Django DEBUG=True、PHP display_errors=On）。
03
HTTP 安全頭 — 核查響應頭：X-Frame-Options（防點擊劫持）、Content-Security-Policy（防 XSS）、X-Content-Type-Options: nosniff、Strict-Transport-Security（HSTS）、Set-Cookie 的 Secure/HttpOnly/SameSite 屬性。使用 securityheaders.com 快速評測。
04
OWASP Top 10 掃描 — 使用 AWVS/Burp Suite Pro/OWASP ZAP 對 Web 應用進行掃描，覆蓋：SQL 注入、XSS（反射/存儲/DOM）、不安全的直接對象引用（IDOR）、安全配置錯誤、組件漏洞。重點驗證高危發現的可利用性。
05
API 安全 — API 端點是否有認證授權；是否暴露 Swagger UI 至外網（洩露 API 文檔）；rate limiting 是否配置；CORS 策略是否過於寬鬆（Access-Control-Allow-Origin: *）；GraphQL introspection 是否在生產環境關閉。

風險計算模型：定性與定量

風險計算是將前期收集的資產重要性、威脅頻率、脆弱性嚴重度三項輸入整合，得到量化的風險值。GB/T 20984-2022 提供了定性和定量兩類方法。

GB/T 20984-2022 風險計算公式

風險計算公式與示例
核心公式：風險值 = 資產重要性 × 威脅可能性 × 脆弱性嚴重度

─── 各指標賦值範圍（均為 1-5 分）───
資產重要性（A）：1-5（基於 CIA 三維度最高值）
威脅可能性（T）：1=極低（近5年無事件）
                 2=低（每年偶發）
                 3=中（每季度偶發）
                 4=高（每月可能）
                 5=極高（持續面臨）
脆弱性嚴重度（V）：1=可忽略  2=低  3=中  4=高  5=嚴重
                   （參考 CVSS 評分映射：
                     CVSS 0.1-3.9 → 1-2
                     CVSS 4.0-6.9 → 3
                     CVSS 7.0-8.9 → 4
                     CVSS 9.0-10  → 5）

─── 計算示例 ───
# 場景：互聯網暴露的 Web 服務器，存在 Apache Log4j2 RCE 漏洞（CVE-2021-44228）
A（資產重要性）= 4  # 業務系統，C=3 I=3 A=4
T（威脅可能性）= 5  # Log4Shell 漏洞已有完整 PoC，持續被掃描利用
V（脆弱性嚴重度）= 5  # CVSS 10.0，直接 RCE

風險值 = 4 × 5 × 5 = 100  → 超高風險（立即處置）

# 場景：內網 FTP 服務器，使用默認帳號 anonymous
A（資產重要性）= 3  # 內部文件服務器，C=3 I=2 A=3
T（威脅可能性）= 2  # 內網威脅，概率較低
V（脆弱性嚴重度）= 3  # 允許匿名登入，但訪問有限

風險值 = 3 × 2 × 3 = 18  → 中等風險（計劃整改）

定量方法：年度損失期望（ALE）

ALE 定量風險計算
ALE（Annual Loss Expectancy）= SLE × ARO

SLE（Single Loss Expectancy，單次損失期望）
  = 資產價值（AV）× 暴露因子（EF，百分比）
  暴露因子：若威脅完全實現，資產損失的比例
  例：數據洩露導致客戶流失、罰款，損失 40% 業務價值

ARO（Annualized Rate of Occurrence，年化發生率）
  = 預計每年發生次數
  例：根據行業統計，勒索軟件攻擊成功概率 0.1 次/年

─── 計算示例：勒索軟件攻擊生產資料庫 ───
資產價值（AV）      = ¥5,000,000（資料庫系統及數據商業價值）
暴露因子（EF）      = 70%（加密後業務恢復成本+數據損失）
SLE = ¥5,000,000 × 70% = ¥3,500,000

ARO = 0.2（行業統計：中型金融機構每5年遭受一次成功勒索）
ALE = ¥3,500,000 × 0.2 = ¥700,000/年

# 安全投資決策：若部署 EDR 系統年費 ¥200,000，可將 EF 降低到 20%
# 新 SLE = ¥5,000,000 × 20% = ¥1,000,000
# 新 ALE = ¥1,000,000 × 0.2 = ¥200,000/年
# 安全投資收益 = ¥700,000 - ¥200,000 - ¥200,000（成本）= ¥300,000/年 → 值得投資

殘餘風險評估與接受準則

殘餘風險（Residual Risk）是在實施安全控制措施後仍然存在的風險。沒有任何組織能消除所有風險，因此確定可接受的殘餘風險水平至關重要。

常見誤區

許多組織錯誤地認為「安裝了防火牆/殺毒軟件就沒有風險了」。實際上，任何安全控制只能降低風險，不能消除風險。殘餘風險評估的目的是確認控制措施是否有效，以及剩餘的風險是否在可接受範圍內。如果殘餘風險仍高於可接受水平，需要補充額外的安全措施或作出接受決策（由管理層書面確認接受）。

殘餘風險計算示例
殘餘風險評估流程：

# 初始風險識別
風險場景：外網 Web 應用存在 SQL 注入漏洞
原始風險值：A=4 × T=4 × V=5 = 80（極高風險）

# 已部署安全控制措施
控制 1：WAF（Web 應用防火牆）—— 降低 T（有效攔截自動化掃描）
控制 2：輸入驗證（代碼層修復）—— 降低 V
控制 3：最小權限資料庫帳號 —— 降低 A（即使注入也無法提權）

# 控制措施有效性評估（需現場驗證，而非假設）
WAF 有效性：繞過測試顯示 80% 的 SQL 注入 payload 被攔截
            但 10% 的繞過技術（編碼、HTTP 分塊）仍可繞過

# 殘餘風險計算
控制後 T 降低：4 → 2（WAF 降低威脅可能性）
控制後 V 降低：5 → 3（部分修復，仍有遺漏）
控制後 A 不變：4（資產重要性不因控制而降低）

殘餘風險值 = A=4 × T=2 × V=3 = 24（中等風險）
# 從極高風險降至中等風險，仍需納入整改計劃（3個月內完成代碼根本修復）

滲透測試在風險評估中的定位

滲透測試（Penetration Testing）是風險評估中脆弱性評估環節的重要手段之一，但它不能替代完整的風險評估流程。兩者是互補關係：風險評估提供全面的管理視角，滲透測試提供深入的技術驗證。

維度漏洞掃描滲透測試完整風險評估
範圍廣（全量資產自動掃描）窄（授權範圍內深度測試）最廣（技術+管理+業務全覆蓋）
深度淺（已知漏洞特徵匹配）深（模擬真實攻擊鏈）結構化（按方法論逐層深入）
輸出漏洞清單+CVSS評分攻擊路徑+業務影響證明風險等級+整改優先級+管理建議
週期數小時（自動化）數天至數週數週至數月
主要價值識別已知漏洞全貌驗證真實可利用性、發現邏輯漏洞業務風險量化、安全投資決策支撐
在風險評估中如何使用滲透測試結果

滲透測試的發現可直接作為脆弱性評估的輸入：（1）已被驗證可利用的漏洞，其脆弱性嚴重度應設為最高（5分），因為攻擊可行性已被確認；（2）滲透測試發現的攻擊鏈（如從外網入侵到獲取核心數據庫權限），可作為風險場景直接納入風險計算；（3）滲透測試中未能突破的邊界（如防火牆有效阻擋），可在控制措施有效性評估中作為正面證據。

評估範圍界定與授權文件

在風險評估中開展滲透測試，必須有嚴格的範圍界定和法律授權。任何未授權的測試活動均可能違反《網路安全法》第二十七條，導致法律責任。

風險評估授權文件核心要素
一、服務合同（甲方簽署）
□ 評估服務說明（風險評估+脆弱性評估+滲透測試）
□ 評估範圍清單（IP 段、域名、系統名稱）
□ 授權方式說明（白盒/灰盒/黑盒）
□ 禁止行為列表（如禁止 DoS 測試、禁止真實資料導出）
□ 甲乙雙方法人或授權代表簽字蓋章

二、保密協議（NDA）
□ 評估過程中獲取的信息保密義務
□ 評估報告的使用限制（不得對外披露）
□ 人員保密聲明（所有參與人員簽署）
□ 保密期限和違約責任

三、技術授權書（必備）
□ 明確授權的目標系統/IP/域名清單
  示例：
  授權系統：XXX 集團電商平台
  授權 IP 範圍：
    外網：1.2.3.4-1.2.3.10（Web 服務器集群）
    外網：1.2.3.50（API 網關）
  授權域名：shop.example.com、api.example.com
  授權內網（白盒）：192.168.1.0/24（業務服務器）
  
□ 授權測試時間窗口
  示例：非業務高峰期（工作日 22:00-次日 06:00）
  或：全天（需實時監控，發現異常立即中止）

□ 明確禁止行為
  - 禁止持久化後門植入
  - 禁止對非授權系統測試
  - 禁止真實敏感數據導出（截圖作為証據即可）
  - 禁止任何破壞性操作（不得刪除/修改業務數據）
  - 禁止 DDoS 或壓力測試

□ 緊急聯繫人
  甲方技術聯繫人：姓名、手機（7×24 可達）
  甲方安全負責人：姓名、手機
  發現重大高危漏洞時立即電話通知（不等書面報告）

□ 授權方簽字：甲方信息安全主管或 CTO 簽名+日期

授權書必須由被評估方具有足夠授權的人員簽署（通常是 CTO、CISO 或 CEO）。業務負責人或普通技術人員的簽字通常不構成有效法律授權。測試前應留存授權書原件或原件掃描件。若評估涉及第三方（如 SaaS 雲服務供應商提供的系統），還需額外取得第三方的書面同意。

外網 IP 與 Web 資產評估實戰

外網資產評估是風險評估中暴露面最大、優先級最高的部分。從攻擊者視角對互聯網暴露面進行系統性識別和脆弱性分析。

內網滲透測試評估要點

內網評估通常以白盒或灰盒方式進行，重點評估縱深防禦能力：攻擊者在突破邊界後能獲取多大的訪問權限，以及橫向移動和提權的難度。

01
網絡分段有效性驗證 — 核查 VLAN 和防火牆策略是否真正隔離不同安全區域。從普通辦公網嘗試訪問服務器區（應被阻擋）；測試 DMZ 與內網的隔離是否嚴格；驗證管理網絡是否與業務網絡分離。
02
AD/域環境安全性 — 內網有 Active Directory 域環境時，評估域配置安全性：是否存在 Kerberoasting 攻擊面（具有 SPN 的高權限帳號）、域內密碼策略、域管帳號數量、AS-REP Roasting 風險、是否開放 LDAP 未認證查詢。
03
憑證管理評估 — 掃描是否存在憑證明文傳輸（Telnet/FTP/HTTP 基本認證）；檢查各系統是否使用相同密碼（密碼複用風險）；測試默認憑證（路由器/交換機/資料庫/中間件管理後台）；評估特權帳號管理（PAM）解決方案是否部署。
04
橫向移動路徑評估 — 從一個已落陷的普通主機出發，評估能否橫向移動到其他系統。測試：SMB 文件共享訪問範圍、RDP 可達性、內網服務掃描、是否存在 Pass-the-Hash 機會（Mimikatz 在合法範圍內使用）。記錄完整的橫向移動路徑作為風險場景。
05
關鍵資產可達性 — 從假設的攻擊初始立足點（如辦公網終端），評估到核心資產（核心業務資料庫、支付系統、工控系統）的攻擊路徑。確認是否有足夠的訪問控制阻止攻擊者直達核心資產，識別縱深防禦的薄弱環節。

風險處置策略：接受 / 規避 / 轉移 / 降低

識別和計算風險之後，必須為每個風險制定明確的處置決策。ISO/IEC 27005 和 GB/T 20984-2022 均定義了四種標準風險處置策略，選擇哪種策略取決於風險等級、控制成本和業務需求。

處置策略適用條件典型措施注意事項
降低風險
（Risk Reduction）
最常用		風險高於可接受水平，且有技術或管理手段可有效降低；控制成本低於預期損失修補漏洞、加強訪問控制、部署 WAF/EDR、安全培訓、加密數據、啟用 MFA需驗證控制措施的有效性，並重新評估殘餘風險；避免只做表面控制（如只部署工具，不調整配置）
規避風險
（Risk Avoidance）		風險極高且無法有效控制；業務活動的風險超過其價值；終止或不開展相關業務更合理關閉對外暴露的高危服務；停止使用高風險組件（如使用雲服務替代自建 VPN）；取消高風險業務場景規避往往意味著業務功能的損失，需業務部門決策；注意規避後可能引入新風險
轉移風險
（Risk Transfer）		無法完全消除風險，但可通過合同或保險將損失轉移；對特定類型風險（財務損失）有效購買網絡安全保險（Cyber Insurance）；服務外包合同中明確安全責任；雲服務共擔責任模型轉移不等於消除風險，仍需最低限度的安全控制；保險公司通常要求投保前具備基本安全措施
接受風險
（Risk Acceptance）		風險已在可接受水平（低風險）；進一步控制的成本高於潛在損失；業務必須接受的固有風險文件化記錄接受決策；定期複評（風險可能隨時間變化）；必要時設定觸發條件（超過閾值重新評估）必須由具有授權的管理人員書面確認；高風險不應輕易選擇「接受」

安全加固建議清單編制

加固建議清單是風險評估報告中最具實操價值的部分，需要將技術發現轉化為可執行的整改步驟，並按優先級排序，幫助客戶有效分配有限的資源。

建議清單格式規範

每條加固建議應包含五個要素：問題描述（What）、風險說明（Why）、整改步驟（How）、驗證方法（Verify）、優先級（Priority）。

加固建議示例（MySQL 數據庫）
建議 DB-001：禁止 MySQL root 帳號遠端登入
優先級：高
問題：生產資料庫的 root 帳號允許從任意主機（host='%'）遠端連接。
風險：攻擊者可通過暴力破解或憑證洩露直接以最高權限訪問資料庫，
      獲取、修改或刪除所有業務數據，後果極其嚴重。
整改步驟：
  1. 登入 MySQL：mysql -u root -p
  2. 查詢 root 帳號的 host 設置：
     SELECT user, host FROM mysql.user WHERE user='root';
  3. 刪除允許遠端登入的 root 帳號：
     DROP USER 'root'@'%';
  4. 確保僅保留 root@localhost：
     SELECT user, host FROM mysql.user WHERE user='root';
     -- 應只顯示 root@localhost
  5. 刷新權限：FLUSH PRIVILEGES;
  6. 確認防火牆已封鎖 3306 端口的外部訪問（雙重保護）
驗證方法：
  嘗試從外部 IP 使用 root 帳號連接，應收到「Access denied」錯誤。
  mysql -h [公網IP] -u root -p[密碼]
  預期結果：ERROR 1045 (28000): Access denied...
參考：MySQL 安全指南 §5.1.1

建議 OS-003：啟用 Linux 系統安全審計日誌
優先級：中
問題：目標主機未安裝 auditd 服務，缺乏對特權操作（su、sudo）、
      敏感文件訪問的審計記錄，發生安全事件時無法溯源。
整改步驟：
  1. 安裝 auditd：
     yum install audit -y  # RHEL/CentOS
     apt install auditd -y # Ubuntu/Debian
  2. 啟用並設為開機啟動：
     systemctl enable auditd && systemctl start auditd
  3. 配置關鍵審計規則（/etc/audit/rules.d/99-security.rules）：
     -w /etc/passwd -p wa -k identity_change
     -w /etc/sudoers -p wa -k sudoers_change
     -a always,exit -F arch=b64 -S execve -k exec_commands
  4. 重載規則：augenrules --load
驗證方法：
  ausearch -k identity_change | tail -5（查看是否有審計記錄生成）

風險評估報告撰寫規範

風險評估報告是整個項目的核心交付物，面向兩類讀者：管理層（需要決策支持，關注業務影響和投資建議）和技術團隊（需要可執行的整改步驟）。報告應分層撰寫，滿足不同讀者需求。

報告結構（GB/T 20984-2022 推薦格式）

01
執行摘要（Executive Summary） — 2-3 頁，面向管理層。核心內容：評估背景與目的、主要發現概述（極高/高風險數量）、最重要的 3-5 個風險及業務影響、優先處置建議、整體安全態勢評價。避免技術術語，用業務語言表達。
02
評估概述 — 評估方法論說明（參考標準：GB/T 20984-2022）、評估範圍和邊界、評估時間、評估方式（白盒/灰盒）、評估局限性聲明（本報告反映評估時間點的狀況）。
03
資產識別結果 — 資產清冊（按類型分類）、資產重要性分布圖（高/中/低比例）、關鍵資產列表及其 CIA 賦值說明。
04
威脅與脆弱性分析 — 主要威脅場景描述、脆弱性發現匯總表（按嚴重程度統計：嚴重/高/中/低/信息）、重要脆弱性詳細描述（含截圖或代碼片段作為證據）。
05
風險評估結果 — 風險值計算過程說明、風險分布熱力圖/矩陣圖、風險清單（按等級排序：極高→高→中→低）、每個風險的詳細描述（風險描述、受影響資產、建議處置措施）。
06
安全加固建議 — 按優先級排序的整改建議清單；整改路線圖（時間線，哪些先做、哪些後做）；安全能力建設建議（長期）；必要的安全投資建議。
07
附錄 — 評估工具清單（增加透明度）、評估依據的標準文件清單、關鍵術語說明、人員訪談記錄、相關法規要求對照表。

Python — 風險值自動計算與熱力圖生成
import pandas as pd
import matplotlib.pyplot as plt
import seaborn as sns

# ─── 從 Excel 讀取風險評估數據 ───
df = pd.read_excel('risk_data.xlsx', sheet_name='脆弱性清單')
# 必要欄位：資產名稱, 資產重要性(A), 威脅可能性(T), 脆弱性嚴重度(V)

# ─── 計算風險值 ───
df['風險值'] = df['資產重要性'] * df['威脅可能性'] * df['脆弱性嚴重度']

# ─── 風險等級判定 ───
def risk_level(v):
    if v >= 75: return '極高'
    elif v >= 40: return '高'
    elif v >= 15: return '中'
    else: return '低'

df['風險等級'] = df['風險值'].apply(risk_level)

# ─── 按風險值降序排列（優先處置順序）───
df_sorted = df.sort_values('風險值', ascending=False)
print(df_sorted[['資產名稱', '脆弱性描述', '風險值', '風險等級']].head(20))

# ─── 風險分布統計 ───
risk_counts = df['風險等級'].value_counts()
print("\n風險分布：")
print(f"極高風險：{risk_counts.get('極高', 0)} 項")
print(f"高風險：{risk_counts.get('高', 0)} 項")
print(f"中等風險：{risk_counts.get('中', 0)} 項")
print(f"低風險：{risk_counts.get('低', 0)} 項")

# ─── 生成風險矩陣熱力圖 ───
matrix = df.groupby(['威脅可能性', '脆弱性嚴重度']).size().unstack(fill_value=0)
plt.figure(figsize=(8, 6))
sns.heatmap(matrix, annot=True, fmt='d', cmap='RdYlGn_r', cbar_kws={'label':'風險數量'})
plt.title('風險矩陣熱力圖')
plt.xlabel('脆弱性嚴重度')
plt.ylabel('威脅可能性')
plt.savefig('risk_matrix.png', dpi=150, bbox_inches='tight')
print("熱力圖已保存至 risk_matrix.png")

交付質量控制要點

高質量報告的標誌：（1）每個脆弱性都有截圖或代碼片段作為客觀證據，不依賴工具自動輸出；（2）風險計算有清晰的推理過程，客戶能理解評分依據；（3）整改建議具體可執行（包含具體命令或配置步驟），而非空泛的「加強安全管理」；（4）管理層摘要能讓非技術高管在 10 分鐘內理解核心問題；（5）所有技術術語都有解釋，或在附錄提供術語表。

小測驗
在對某企業的 MySQL 資料庫進行風險評估時，你發現 root 帳號允許從任意 IP 遠端登入，CVSS 評分為 7.2（高危），資料庫存儲大量客戶個人信息（資產重要性=5），評估威脅可能性為 3（中等）。按正確處置流程，以下哪個描述最完整？
A記錄為高危漏洞（CVSS 7.2），建議客戶購買更好的防火牆來降低風險
B計算風險值：5×3×7.2=108，判定為極高風險，建議立即整改
CCVSS 7.2 映射為脆弱性嚴重度 4，風險值=5×3×4=60（高風險），建議1個月內整改：刪除 root@'%' 帳號，僅保留 root@localhost，並驗證防火牆封鎖 3306 端口
D因已有防火牆存在，可將此脆弱性標記為殘餘風險並接受，無需整改
正確！完整的風險評估處置流程應包括三步：（1）按標準映射規則將 CVSS 7.2（高危，7.0-8.9 區間）映射為脆弱性嚴重度 4，而非直接使用 CVSS 原始分數；（2）正確計算風險值：5×3×4 = 60，判定為高風險（40-74 區間），需在 1 個月內整改；（3）提供具體可執行的整改建議：直接操作步驟（DROP USER + FLUSH PRIVILEGES）+ 驗證方法 + 雙重防護（應用層+網絡層）。選項 A 的「購買防火牆」是規避建議而非根本修復；選項 B 直接使用 CVSS 原始分數做乘法，方法錯誤；選項 D 在有明確整改方案的情況下選擇風險接受，決策不當。
