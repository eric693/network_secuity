iOS 架構與安全機制概覽

iOS 安全架構建立在硬體信任根之上，多層機制相互配合。與 Android 相比，iOS 更封閉，安全測試的門檻更高，但漏洞的商業價值也更高。

iOS 分層安全架構

硬體層
Secure Enclave Processor（SEP）
獨立的安全協處理器，負責 Face ID / Touch ID 生物特徵處理、設備加密金鑰管理。SEP 的資料無法被主處理器讀取，即使越獄也無法繞過。
啟動層
安全啟動鏈（Secure Boot Chain）
從 Boot ROM（硬體燒錄，不可修改）開始，每一級引導程式都會驗證下一級的 Apple 數位簽章。任何未簽名的啟動組件都會被拒絕，是越獄需要突破的核心屏障。
核心層
XNU 核心 + 系統完整性保護
基於 Mach / BSD 的混合核心。AMFI（Apple Mobile File Integrity）強制執行代碼簽名，確保每個執行的二進制文件都有 Apple 授權簽名。
系統層
沙盒（Sandbox / Seatbelt）
每個 App 運行在獨立沙盒容器中，只能訪問自己的目錄（~/Library/）。跨 App 通訊必須通過 URL Scheme、Universal Links 或 App Groups 等受控機制。
應用層
App Store 審核 + 代碼簽名
所有 App 必須通過 App Store 審核並使用 Apple 頒發的開發者憑證簽名。企業證書（Enterprise Certificate）可繞過 App Store，曾被惡意 App 濫用。

iOS 核心安全特性速覽

數據保護（Data Protection）
每個文件可設置保護等級（Complete / CompleteUnlessOpen / CompleteUntilFirstUserAuthentication）。設備鎖定後，Complete 等級的文件金鑰被清除，即使硬體提取也無法讀取。
ATS（App Transport Security）
iOS 9+ 強制所有網路連線使用 TLS 1.2+，禁止 HTTP 明文通訊。開發者可在 Info.plist 中明確豁免（NSAllowsArbitraryLoads），但這是安全問題。
Keychain
iOS 系統級加密金鑰庫，用於安全存儲密碼、Token、憑證。不同 App 的 Keychain 項目相互隔離，但同一 App Group 可共享。越獄後 Keychain 可被提取。
ASLR + PIE
地址空間佈局隨機化（ASLR）每次啟動隨機化記憶體佈局，Position Independent Executable（PIE）讓二進制可在任意地址載入。大幅增加記憶體漏洞利用難度。
iOS vs Android 測試差異

iOS 安全測試的最大挑戰是設備依賴性：許多動態分析操作需要越獄設備，而越獄往往依賴特定的 iOS 版本和設備型號。此外，iOS 無法像 Android 那樣輕易修改和重打包 IPA，因為所有二進制文件都需要 Apple 有效簽名。建議準備一台舊款 iPhone（iPhone X 或更早，A11 晶片）並保持在可越獄的 iOS 版本。

IPA 結構與 App 沙盒

IPA（iPhone Package Archive）是 iOS App 的分發格式，本質上是 ZIP 壓縮包。理解其內部結構是靜態分析的基礎。

IPA 內部結構

路徑內容安全關注點
Payload/AppName.app/App 主目錄，包含所有資源整個分析的起點
AppName（二進制）Mach-O 格式可執行文件（可能含多架構 Fat Binary）class-dump / Hopper 逆向的主要目標；注意是否有加密（FairPlay DRM）
Info.plistApp 配置：Bundle ID、版本、URL Scheme、權限說明URL Scheme 聲明、ATS 豁免、危險權限、是否宣告備份排除
Frameworks/App 內嵌的私有 Framework（.framework）可能包含硬編碼金鑰；各 Framework 可單獨逆向
PlugIns/App Extension（Today Widget、Share Extension 等）Extension 可能共享 App Group 數據，是額外攻擊面
_CodeSignature/代碼簽名資訊（CodeResources 文件）篡改任何文件後簽名失效；重打包需要有效開發者憑證

App 沙盒目錄結構

iOS App 沙盒目錄

/var/mobile/Containers/Data/Application/[UUID]/

├── Documents/          # 用戶文件，iTunes / iCloud 備份，用戶可訪問
│   └── user_data.db    # 常見：SQLite 資料庫（重點檢查）
│
├── Library/
│   ├── Preferences/    # NSUserDefaults（.plist 文件），敏感數據常存於此
│   │   └── com.target.app.plist

│   │   └── snapshots/  # App 切換時的截圖！可能截到敏感畫面
│   └── Application Support/  # App 支援文件，備份

# App Bundle（安裝目錄，只讀）
/var/containers/Bundle/Application/[UUID]/AppName.app/
截圖快取洩露

iOS 在 App 進入後台時會自動截圖用於 App 切換器的預覽。如果截圖的瞬間畫面包含密碼、金融資訊等敏感內容，這個截圖會被儲存在 Library/Caches/Snapshots/。防禦方法：在 applicationDidEnterBackground 中隱藏敏感視圖，或使用 ignoreSnapshotOnNextApplicationLaunch。

測試環境：越獄設備與 Checkra1n

checkra1n（硬體漏洞型）
利用 A5–A11 晶片 Boot ROM 漏洞（checkm8），半持久越獄（重啟後需重新執行 checkra1n）。支援 iPhone 5s–iPhone X，iOS 12–14.x。由於是硬體漏洞，Apple 無法修補。
unc0ver / Dopamine
軟體漏洞型越獄，針對特定 iOS 版本的核心漏洞。unc0ver 支援 iOS 11–14.8，Dopamine 針對 iOS 15–16.x。版本依賴性強，Apple 發版後通常修補。
Palera1n
基於 checkm8 的現代越獄工具，支援 iPhone 6–iPhone X 的 iOS 15–16.x。rootless 模式對系統分區改動最小，是目前最受推薦的 A11 及以下設備越獄方案。
Cydia / Sileo
越獄後的套件管理器（類似 apt/brew）。安全測試常用套件：SSH（openssh）、Frida（frida）、debugserver（允許 LLDB 連線）、com.tigisoftware.Filza（文件管理器）。

小測驗
一位測試人員對一個 iOS 銀行 App 進行安全測試，發現即使在越獄設備上，也無法通過 Frida 繞過 Face ID 認證，因為相關處理完全在一個獨立處理器中進行且無法被 Hook。這個處理器是？
AApple Neural Engine（ANE），用於機器學習加速
BSecure Enclave Processor（SEP），獨立的安全協處理器
CImage Signal Processor（ISP），用於攝影機處理
DMotion Coprocessor（M 系列），用於感測器資料
正確！Secure Enclave Processor（SEP）是 Apple A7 晶片（iPhone 5s）以後所有 iPhone 中的獨立安全協處理器。它負責 Face ID / Touch ID 生物特徵資料的處理與比對，以及設備加密金鑰的管理。SEP 與主處理器完全隔離，主處理器只能發送「是否驗證成功」的結果，無法直接存取生物特徵資料。即使在越獄設備上，Frida 也無法 Hook SEP 的操作，這是 iOS 硬體安全的核心設計。

IPA 解密與 class-dump

從 App Store 下載的 App 受 FairPlay DRM 加密保護，直接用 Hopper 或 class-dump 分析只會看到加密的二進制。必須在越獄設備上將 App 從記憶體傾倒（Dump）出來，才能得到可分析的明文二進制。

解密工具

frida-ios-dump
基於 Frida 的自動化 IPA 傾倒工具。在 App 運行時從記憶體提取解密後的二進制，並重新打包為 IPA。這是目前最常用且最可靠的解密方式。
bagbak
另一個基於 Frida 的傾倒工具，語法更簡潔。bagbak --output ./dump com.target.app 即可完成解密和打包。
class-dump / dsdump
從 Objective-C 或 Swift 的 Mach-O 二進制提取類別、方法、屬性的宣告，生成類似頭文件（.h）的輸出。是理解 App 結構的快速方法，但不顯示實作細節。
Clutch（舊版工具）
較舊的傾倒工具，部分新版 iOS 相容性差。已基本被 frida-ios-dump 和 bagbak 取代，但在某些環境下仍可使用。
IPA 解密與 class-dump 使用
# ─── 方法1：bagbak（推薦，最簡單）───
pip3 install bagbak
# 確保 frida-server 在 iOS 設備上運行
bagbak --output ./decrypted/ com.target.app
# 輸出：decrypted/AppName.ipa（已解密，可直接用 jadx / Hopper 分析）

# ─── 方法2：frida-ios-dump ───
git clone https://github.com/AloneMonkey/frida-ios-dump.git
pip3 install -r frida-ios-dump/requirements.txt

# 配置 dump.py 中的連線設定（IP / Port）
# 列出設備上的 App
python3 frida-ios-dump/dump.py -l

# 傾倒指定 App
python3 frida-ios-dump/dump.py com.target.app
# 輸出：AppName.ipa

# ─── 解壓 IPA 並執行 class-dump ───
unzip AppName.ipa -d AppName_extracted/
cd AppName_extracted/Payload/AppName.app/

# class-dump：適用於 Objective-C
class-dump -S -s -H AppName -o ./headers/
# 輸出：每個類別一個 .h 頭文件
ls ./headers/                 # LoginViewController.h, UserManager.h 等
cat ./headers/LoginViewController.h

# dsdump：支援 Objective-C 和 Swift
dsdump --objc --verbose AppName        # Objective-C 詳細輸出
dsdump --swift --verbose AppName       # Swift 類別和方法

Hopper / Ghidra 逆向分析

class-dump 只能看到方法宣告，要了解方法的實作邏輯，需要反組譯（Disassembly）或反編譯（Decompilation）工具。Hopper 是 macOS 上最流行的 iOS 逆向工具，Ghidra 是免費的替代方案。

工具平台特點適用場景
Hopper DisassemblermacOS / LinuxARM64 反組譯 + 偽代碼生成，Objective-C 訊息語義識別，試用版免費但有時間限制iOS 逆向首選；偽代碼可讀性高
Ghidra跨平台（Java）NSA 開源工具，功能全面，插件豐富，無時間限制。ARM64 分析能力強免費替代 Hopper；複雜分析、指令碼自動化
IDA Pro跨平台業界標準，ARM64 分析最強，Hex-Rays 反編譯器輸出品質最高高端漏洞研究；價格昂貴（數千美元）
Binary Ninja跨平台現代化介面，API 豐富，適合腳本化分析，個人版較 IDA 便宜商業替代 IDA；自動化分析
Hopper / Ghidra 逆向工作流
# ─── 使用 Hopper 分析（GUI 工具）───
# 1. File → Open → 選擇解密後的 Mach-O 二進制

# 或使用命令列分析腳本（無 GUI）

  -import AppName \
  -postScript ExtractStrings.py   # 執行後處理腳本

# ─── 定位關鍵函數的通用技巧 ───
# 1. 從字串入手：搜索錯誤訊息字串（如「Invalid password」）
#    → 找到引用該字串的函數 → 這通常就是認證邏輯所在位置
#
# 2. 從 class-dump 頭文件入手：
#    找到 -[LoginViewController verifyPassword:] 等方法

# 3. 從交叉引用（XRef）入手：

Info.plist 安全審計

Info.plist 是 iOS App 的核心設定清單，等同於 Android 的 AndroidManifest.xml。靜態分析的第一步就是仔細審查其中的安全相關設定。

Info.plist — 關鍵安全設定審查
<!-- ─── 1. App Transport Security（ATS）設定 ─── -->
<!-- 危險：完全禁用 ATS，允許任意明文 HTTP 連線 -->
<key>NSAppTransportSecurity</key>
<dict>
  <key>NSAllowsArbitraryLoads</key>
  <true/>  <!-- 高危！允許 HTTP 明文通訊 -->
</dict>

<!-- ─── 2. URL Scheme 聲明（潛在劫持點）─── -->
<key>CFBundleURLTypes</key>
<array>
  <dict>
    <key>CFBundleURLSchemes</key>
    <array>
      <string>myapp</string>  <!-- 任何 App 可搶注此 scheme，劫持回調 -->
    </array>
  </dict>
</array>

<!-- ─── 3. 過度請求權限（超出功能需要）─── -->
<key>NSCameraUsageDescription</key>
<string>用於掃描二維碼</string>
<key>NSMicrophoneUsageDescription</key>
<string>用於語音輸入</string>
<key>NSContactsUsageDescription</key>
<string>用於找到好友</string>
<!-- 一個計算機 App 為何需要麥克風和通訊錄？值得質疑 -->

<!-- ─── 4. 應用程式查詢（窺探已安裝 App）─── -->
<key>LSApplicationQueriesSchemes</key>
<array>
  <string>weixin</string>  <!-- 可查詢是否安裝了微信 -->
  <string>alipay</string>  <!-- 可查詢是否安裝了支付寶（隱私問題）-->
</array>

代碼審計：硬編碼與不安全 API

iOS 靜態分析 — 高危模式搜索
# ─── 在解密後的 IPA 目錄中搜索敏感字串 ───

# 搜索二進制中的字串（硬編碼 Key / URL / 密碼）
strings Payload/AppName.app/AppName | grep -iE "password|secret|apikey|token|http:|192\.168"

# 搜索 plist 資源文件
grep -r "secret\|apikey\|password" Payload/AppName.app/ --include="*.plist"

# ─── 危險 API 識別（class-dump 輸出後搜索）───
# 不安全的隨機數（應使用 SecRandomCopyBytes）
grep -i "arc4random\|rand()" ./headers/*.h

# MD5 使用（不應用於密碼 Hash 或安全目的）
grep -i "CC_MD5\|md5" ./headers/*.h

# 弱加密（DES / 3DES / ECB 模式）
grep -i "kCCAlgorithmDES\|ECBMode" ./headers/*.h

# NSLog 洩露（生產版本應關閉敏感日誌）
grep -i "NSLog\|printf" ./headers/*.h

# UIPasteboard 使用（剪貼板可被其他 App 讀取）
grep -i "UIPasteboard" ./headers/*.h
iOS 危險 API 對照表（Objective-C / Swift）
// ─── 硬編碼憑證（最常見！）───
let apiKey = "sk-live-xxxxxxxxxxxxxxxxxx"    // 硬編碼 API 金鑰
let aesKey = "MySecret16ByteKey"              // 硬編碼加密金鑰

// ─── 明文存儲敏感數據 ───
UserDefaults.standard.set(authToken, forKey: "token")  // 應使用 Keychain
let path = FileManager.default.urls(for: .documentDirectory)
// 在 Documents 中存儲未加密的密碼文件 → 可備份，可被提取

// ─── 不安全的 WebView ───
let config = WKWebViewConfiguration()
// 添加 JS Bridge 暴露原生功能
config.userContentController.add(self, name: "nativeBridge")
// JavaScript 可呼叫：window.webkit.messageHandlers.nativeBridge.postMessage()

// ─── 弱加密算法 ───
// Objective-C：使用 CommonCrypto
CCCrypt(kCCEncrypt, kCCAlgorithmDES,    // DES：56-bit，已不安全
        kCCOptionPKCS7Padding, ...)
CCCrypt(kCCEncrypt, kCCAlgorithmAES,
        kCCOptionECBMode, ...)           // ECB 模式：相同明文輸出相同密文

// ─── 剪貼板洩露 ───
UIPasteboard.general.string = password   // 密碼複製到剪貼板，其他 App 可讀取

// ─── 不安全的 TLS 設定 ───

    didReceive challenge: URLAuthenticationChallenge,
    completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void) {
    completionHandler(.useCredential, URLCredential(trust: challenge.protectionSpace.serverTrust!))
    // 接受任何憑證！相當於禁用 TLS 驗證
}

HTTPS 流量攔截與 SSL Pinning 繞過

iOS HTTPS 流量攔截的基本流程與 Android 相似，但細節有所不同。iOS 14+ 引入了更嚴格的 ATS 強制執行，部分 App 的 SSL Pinning 實作方式也與 Android 不同。

iOS HTTPS 抓包設定流程
# ─── 步驟1：設定 iOS 設備使用 Burp 代理 ───
# 設定 → WiFi → 點選已連線的網路 → 設定代理 → 手動
# 伺服器：攻擊機 IP（如 192.168.1.50）
# 通訊埠：8080

# ─── 步驟2：在 iOS 設備上安裝 Burp CA 憑證 ───
# 在 Safari 訪問 http://burpsuite（自動跳至 http://192.168.1.50:8080）

Frida on iOS：Hook Objective-C 與 Swift

iOS 上的 Frida 使用方式與 Android 類似，但 Hook 語法完全不同：iOS 使用 Objective-C 的訊息傳遞機制（Message Passing），方法呼叫格式為 -[ClassName methodName:arg]。

Frida iOS — Objective-C Hook 語法
// ─── 基本 Objective-C 方法 Hook ───
ObjC.schedule(ObjC.mainQueue, function() {
    var LoginVC = ObjC.classes.LoginViewController;

    // Hook 實例方法 -[LoginViewController verifyPassword:]
    var verifyPassword = LoginVC['- verifyPassword:'];
    Interceptor.attach(verifyPassword.implementation, {
        onEnter: function(args) {
            // args[0] = self, args[1] = _cmd, args[2] = 第一個參數
            var password = new ObjC.Object(args[2]).toString();
            console.log('[+] verifyPassword called with: ' + password);
        },
        onLeave: function(retval) {
            console.log('[+] return value: ' + retval);
            retval.replace(1);  // 強制返回 YES（BOOL 在 ObjC 中是 1）
        }
    });
});

// ─── Hook 類別方法（+[Class method]）───
ObjC.schedule(ObjC.mainQueue, function() {
    var UserManager = ObjC.classes.UserManager;

    // Hook 類別方法 +[UserManager sharedInstance]
    var sharedInstance = UserManager['+ sharedInstance'];
    Interceptor.attach(sharedInstance.implementation, {
        onLeave: function(retval) {
            var instance = new ObjC.Object(retval);
            // 讀取 instance 的屬性：
            console.log('authToken: ' + instance.authToken().toString());
        }
    });
});

// ─── 列出類別的所有方法 ───
ObjC.schedule(ObjC.mainQueue, function() {
    var cls = ObjC.classes.LoginViewController;
    var methods = cls.$ownMethods;
    methods.forEach(function(method) {
        console.log(method);
    });
});

// ─── Hook Swift 方法（需先知道 mangled name）───
// Swift 方法名稱被 mangled，需先找到正確名稱
// 使用 nm 或 rabin2 查找：
// nm AppName | grep -i "verify\|auth"
var swiftFunc = Module.findExportByName(null, '$s9TargetApp12LoginManagerC14verifyPasswordySbSSF');
Interceptor.attach(swiftFunc, {
    onEnter: function(args) { console.log('Swift func called'); },
    onLeave: function(retval) { retval.replace(1); }
});

小測驗
在使用 Frida Hook iOS App 的 Objective-C 方法時，你需要 Hook 方法 -[AuthManager isValidToken:] 並讓它永遠返回 YES。以下哪段代碼是正確的？
AJava.use('AuthManager').isValidToken.implementation = function() { return true; }
BObjC.classes.AuthManager['- isValidToken:'].implementation 加上 onLeave: retval.replace(1)
CModule.findExportByName('AuthManager', 'isValidToken') 加上 Interceptor.attach
DInterceptor.replace(ptr(AuthManager.isValidToken), new NativeCallback(...))
正確！iOS Objective-C 方法的 Frida Hook 必須使用 ObjC API，而不是 Android 的 Java.use()。語法格式是 ObjC.classes.ClassName['- methodName:']，其中「-」代表實例方法，「+」代表類別方法。返回 BOOL YES 需要 retval.replace(1)，因為 Objective-C 的 BOOL 在底層是整數，YES = 1，NO = 0。選項 A 是 Android 的語法；選項 C 適用於 C/C++ 匯出函數，不直接適用於 ObjC 方法。

不安全的數據存儲（Keychain / NSUserDefaults）

iOS 提供了多種數據存儲機制，安全性各不相同。測試人員必須檢查 App 是否將敏感資訊存儲在不適當的位置。

存儲機制安全性常見問題正確用途
Keychain高（硬體加密）使用 kSecAttrAccessible 設定過於寬鬆（如 kSecAttrAccessibleAlways）；越獄後可被 keychain-dumper 提取密碼、Token、私鑰、憑證
NSUserDefaults低（明文 plist）明文存儲 Session Token、用戶偏好設定含敏感業務邏輯僅用於非敏感的 UI 偏好設定
Core Data / SQLite中（依設定）資料庫未加密，敏感欄位明文存儲搭配 SQLCipher 加密；或使用 Data Protection API
文件系統（Documents/）低（可備份）將包含個人資料的文件存入 Documents，iCloud / iTunes 備份可提取設定 NSURLIsExcludedFromBackupKey 排除備份
剪貼板（UIPasteboard）極低密碼/OTP 複製到剪貼板，其他 App 可監聽讀取（iOS 14 以前）iOS 14+ 已有提示，但仍應避免敏感資料進剪貼板
Keychain 安全設定與提取
// ─── Keychain 存取控制等級對比 ───
// 危險：設備未上鎖時也可讀取，且備份後在新設備可讀

// 最安全：需要設備密碼 + 生物辨識，且不可備份
kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly

越獄檢測與繞過策略

iOS App（尤其是金融類）廣泛使用越獄檢測來拒絕在越獄設備上運行，因為越獄環境會削弱沙盒和系統完整性保護。測試人員必須能夠繞過這些檢測。

01
檢測越獄文件是否存在 — 檢查 /Applications/Cydia.app、/bin/bash、/usr/sbin/sshd、/etc/apt 等越獄環境特有路徑
02
嘗試寫入沙盒外路徑 — 正常設備的 App 無法寫入 /private/jailbreak_test.txt；越獄設備可能成功，代表沙盒已被破壞
03
檢測 Cydia URL Scheme — canOpenURL: "cydia://" 若返回 YES，說明 Cydia 已安裝
04
動態函式庫注入檢測 — 檢查進程是否載入了未預期的 dylib（如 FridaGadget.dylib、Substrate.dylib）
05
Integrity Check / AMFI — 驗證 App 本身的代碼簽名完整性，在越獄環境下某些簽名驗證行為會改變
06
IOSurface 沙盒逃逸檢測 — 嘗試通過私有 API 存取受限資源，正常環境會失敗
越獄檢測繞過方法
# ─── 方法1：Objection 一鍵繞過 ───
objection --gadget com.target.app explore
ios jailbreak disable         # Hook 常見越獄檢測函數，統一返回「未越獄」

# ─── 方法2：Liberty Lite / Shadow（Cydia 套件）───
# 在 Cydia 安裝 Liberty Lite 或 Shadow
# 設定 → Liberty → 針對特定 App 啟用隱藏越獄

# ─── 方法3：Frida 手動 Hook 越獄檢測函數 ───
ObjC.schedule(ObjC.mainQueue, function() {
    // Hook fileExistsAtPath:（Cydia.app 路徑檢測）
    var NSFileManager = ObjC.classes.NSFileManager;
    var fileExists = NSFileManager['- fileExistsAtPath:'];
    Interceptor.attach(fileExists.implementation, {
        onEnter: function(args) {
            var path = new ObjC.Object(args[2]).toString();
            if (path.indexOf('Cydia') !== -1 || path.indexOf('jailbreak') !== -1) {
                this.jailbreakPath = true;
            }
        },
        onLeave: function(retval) {
            if (this.jailbreakPath) {
                retval.replace(0);  // fileExists 返回 NO
            }
        }
    });
});

Binary 保護機制分析（PIE / Stack Canary）

分析 iOS App 的二進制保護機制是安全審計的組成部分。缺乏保護的 App 在遭遇記憶體漏洞時更容易被利用。

Binary 保護檢測命令
# ─── 使用 otool 和 checksec 分析 Mach-O 保護機制 ───

# 檢測 PIE（Position Independent Executable）
otool -hv AppName | grep PIE
# 有 PIE 標記 → ASLR 有效；無 PIE → 固定記憶體位址，易於 ROP 攻擊

# 檢測 Stack Canary（堆疊金絲雀，防堆疊溢位）
otool -Iv AppName | grep -i stack_chk
# 找到 ___stack_chk_guard 和 ___stack_chk_fail → Stack Canary 已啟用

# 檢測 ARC（Automatic Reference Counting，防 UAF）
otool -Iv AppName | grep -i objc_release
# 有 _objc_release → 使用 ARC

# 檢測加密（FairPlay DRM）
otool -l AppName | grep -A4 LC_ENCRYPTION_INFO
# cryptsize 不為 0 → 仍有加密，需先解密（frida-ios-dump / bagbak）

# ─── 使用 MobSF 自動分析所有保護機制 ───
# 上傳 IPA 到 MobSF 後，在「Binary Analysis」區域可看到：
# PIE: Yes/No
# Stack Canary: Yes/No
# ARC: Yes/No
# Rpath: Yes/No（含 Rpath 可能有 Dylib Injection 風險）
# Symbols Stripped: Yes/No
# Bitcode: Yes/No

# ─── 使用 rabin2（radare2 套件）全面分析 ───
rabin2 -I AppName          # 顯示所有保護機制
rabin2 -z AppName          # 提取所有字串
rabin2 -i AppName          # 列出所有 import 函數
