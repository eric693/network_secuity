巡檢概述 20 分鐘

安全巡檢定義與服務場景

安全巡檢（Security Inspection）是對組織資訊系統進行週期性、系統化的健康狀態檢查，目標是在既有安全措施下持續驗證安全控制有效性，發現配置偏差、新出現的漏洞及異常狀況。與一次性的風險評估不同，安全巡檢是一種常態化的安全運營活動。

安全巡檢與風險評估的區別

維度安全巡檢風險評估
頻率按週期進行（月度/季度/年度），部分項目每日自動執行通常一年一次，或重大系統上線前後
深度較淺，注重覆蓋面和一致性核查較深，包含深度滲透測試和業務影響分析
目標驗證安全基準是否持續符合，及早發現偏差全面識別、計算和評價風險，支撐管理決策
輸出巡檢報告（合格/不合格清單）、整改工單風險評估報告（風險等級、優先整改建議）
執行方可由內部安全團隊或外部服務商定期執行通常由專業安全機構或持牌評估機構執行

典型服務場景

等保合規維護場景
等保 2.0 要求三級系統每年至少自查一次，兩次測評之間需持續維護安全狀態。安全巡檢作為等保自查的落地手段，確保在下次等保測評前各項指標持續符合要求，避免「應試式」安全建設。
安全運維服務（MSS）
安全服務商（如安全外包 MSS 提供商）為客戶提供週期性安全巡檢服務，替代客戶自建安全運營團隊。月度或季度交付巡檢報告，作為持續安全服務的核心交付物。
重大事件/節假日前保障
重大節假日（春節、國慶）或重大活動（大型晚會、直播、重要業務上線）前，進行專項安全巡檢，確認關鍵系統安全狀態，識別可能導致業務中斷的安全隱患。
系統變更後驗證
系統上線、重大版本升級、網絡架構調整後，進行安全配置巡檢，驗證變更是否引入新的安全隱患，確認既有安全控制措施在新環境中仍然有效。

巡檢流程與項目管理

安全巡檢項目雖週期短（通常 3-10 個工作日），但需嚴格遵循標準流程，確保每次巡檢結果具有可比性，便於趨勢分析。

啟動
計劃確認與授權
與客戶確認本次巡檢範圍（哪些系統、哪些項目）；確認授權文件（巡檢工作單或服務合同中的週期服務條款）；確認巡檢時間窗口（通常避開業務高峰期）；建立本次巡檢的溝通聯繫人。
資產確認
更新資產清單
對比上次巡檢以來的資產變更（新增/下線/配置變更）；更新資產清冊；確認本次巡檢需要覆蓋的所有目標（IP 清單、系統清單、設備清單）；確認必要的訪問憑證和系統賬號。
現場巡檢
按清單逐項核查
策略配置核查（防火牆規則、帳號策略、安全設備策略）；日誌審查（按週期收集並分析異常）；漏洞掃描（對比上次掃描結果，關注新增漏洞）；配置基線核查（對照標準基線逐項驗證）。
結果分析
發現整理與評級
彙總所有巡檢發現；對發現進行分類（配置問題/漏洞/管理問題/異常事件線索）；評定每個發現的嚴重程度（嚴重/高/中/低/建議）；與上次巡檢結果對比（新增問題/已修復問題/持續問題）。
報告輸出
撰寫並交付報告
撰寫巡檢報告（安全狀態趨勢 + 本次發現 + 整改建議）；生成整改工單（責任人、截止日期）；提交給客戶安全負責人；安排報告宣講會（可選，季度或年度巡檢通常需要）。
閉環追蹤
整改驗證
在整改截止日前跟進整改進展；整改完成後進行驗證核查（複查高危發現是否已修復）；更新巡檢問題台帳（標記已修復、延期、接受）；為下次巡檢提供基線對比數據。

小測驗
某組織每季度開展一次安全巡檢，發現本季度防火牆策略中有一條「允許任意來源訪問 22 端口（SSH）至內網所有伺服器」的規則，評估後定為高危發現。此時最適合的處置步驟是？
A立即刪除此防火牆規則，然後記錄在巡檢報告中
B記錄為高危發現並在報告中說明，向客戶安全負責人口頭預警，提供具體整改建議（限制來源 IP），由客戶確認後執行整改，並在整改完成後複驗
C因為只是巡檢服務商，不應修改客戶系統，僅在報告中記錄，等客戶下次聯繫時再處理
D納入下次巡檢觀察，如果下季度仍存在再作高危記錄
正確！安全巡檢中發現的高危問題，服務商的正確做法是：（1）記錄並評級（記錄為高危發現，保存截圖證據）；（2）及時預警（高危問題不應等到報告交付才告知，應口頭或即時通訊立即通知客戶負責人）；（3）提供可執行建議（不只說「有問題」，要給出具體整改步驟）；（4）由客戶授權後執行（巡檢服務商通常不應直接修改客戶生產環境，需獲得客戶明確授權）；（5）驗證整改效果（整改完成後複查確認）。直接修改客戶系統（選項A）在未明確授權的情況下可能違反合同，且可能影響業務。

防火牆策略巡檢

防火牆策略（規則集）是組織邊界防禦的核心，但隨著時間推移，規則集往往累積了大量冗餘、衝突甚至危險的規則。定期的防火牆策略巡檢是安全巡檢中最重要的環節之一。

防火牆策略巡檢核心檢查項

01
「任意到任意」寬鬆規則識別 — 掃描規則集中所有源地址為 any/0.0.0.0/0 且目標地址為 any 的規則，這類規則通常意味著完全不受限制的訪問。重點關注允許從互聯網（外網）直接訪問內網任意目標或任意端口的規則。
02
高危服務端口對外暴露 — 核查以下高危端口是否被允許從互聯網直接訪問：22(SSH)、23(Telnet)、3389(RDP)、445(SMB)、1433(MSSQL)、3306(MySQL)、6379(Redis)、27017(MongoDB)。任何此類直接暴露都應記錄為高危發現。
03
冗餘規則與死規則清理 — 識別長期未被命中的規則（可通過命中計數器判斷：hit count = 0 且規則存在超過 90 天）。這類規則增加維護複雜度，也可能因誤解而保留不安全的訪問。建議標記後由業務確認是否可以刪除。
04
規則順序與衝突 — 防火牆規則按順序匹配，排列靠前的規則優先生效。核查是否存在寬鬆規則在嚴格規則之前（導致嚴格規則永遠不生效）；是否存在允許規則和拒絕規則相互衝突的情況。
05
默認動作核查 — 防火牆的默認策略（default policy）應為 DENY（即沒有明確允許的流量一律拒絕），而非 PERMIT。若默認動作為允許，則所有未被明確拒絕的流量都可通過，安全性大幅降低。
06
NAT 與規則關聯一致性 — 核查 NAT 映射規則與防火牆允許規則是否一致，避免出現「NAT 可以轉發但防火牆沒有對應允許規則」或相反情況，導致訪問控制邏輯不符合設計意圖。
Cisco ASA / iptables 防火牆規則巡檢命令
# ─── Cisco ASA 防火牆策略核查 ───
# 查看所有訪問控制列表
show access-list
show access-list | include any any              ! 識別任意到任意的危險規則
show access-list | include permit any          ! 查找寬泛的 permit any 規則

# 查看命中計數（找出從未被命中的規則）
show access-list | include hitcnt=0             ! hitcnt=0 的規則為死規則

# 查看接口的訪問組（哪個接口應用了哪條 ACL）
show running-config | include access-group

# 查看 NAT 配置
show nat detail

# ─── Linux iptables 規則核查 ───
# 查看所有規則（含計數）
iptables -L -n -v --line-numbers
iptables -L INPUT -n -v --line-numbers    # 入站規則
iptables -L FORWARD -n -v --line-numbers  # 轉發規則

# 核查默認策略（應為 DROP）
iptables -L | grep 'Chain INPUT\|Chain FORWARD\|Chain OUTPUT'
# 預期：Chain INPUT (policy DROP)   ← 安全
# 危險：Chain INPUT (policy ACCEPT) ← 需告警

# ─── 自動化核查腳本（識別高危端口開放）───
# 從外部視角驗證哪些端口實際可到達
for port in 22 23 3389 445 1433 3306 6379 27017 11211 5432; do
  result=$(nc -z -w 2 TARGET_IP $port 2>&1 && echo "OPEN" || echo "CLOSED")
  [ "$result" = "OPEN" ] && echo "[高危] 端口 $port 對外開放！"
done

帳號與訪問控制巡檢

帳號管理是安全巡檢中的高頻發現領域——未及時清理的離職人員帳號、過度授權的服務帳號、未啟用 MFA 的特權帳號，都是高價值的攻擊入口。

帳號安全巡檢全面檢查項

類別巡檢項核查方法合格標準
離職帳號清理已離職人員帳號是否及時停用/刪除對比 HR 系統離職人員名單與 AD/系統帳號清單，比對用戶名/郵箱離職後 24 小時內停用，30 天後刪除或封存
長期未登入帳號超過 90 天未登入的帳號查詢系統登入記錄（AD lastLogon、Linux lastlog）超過 90 天未登入的帳號應禁用並通知業務確認是否可刪除
默認帳號系統/設備默認帳號是否已重命名或禁用核查 Administrator/Guest/admin/root 之外的默認帳號狀態默認管理員帳號重命名或禁用，默認 Guest 帳號禁用
共享帳號是否存在多人共用的帳號訪談 IT 人員，核查是否有「team_admin」等共享帳號特權操作禁止使用共享帳號，每人有獨立帳號（可追溯）
密碼策略密碼複雜度/有效期/歷史策略查看域策略（AD GPO）或系統密碼策略配置最小長度 ≥8，含大小寫+數字+特殊字符，最長 90 天，歷史 5 次
特權帳號 MFA特權帳號（管理員/root）是否啟用多因素認證核查 VPN/堡壘機/雲控制台的 MFA 配置所有特權帳號的遠端訪問必須啟用 MFA
服務帳號應用/服務帳號是否遵循最小權限核查服務帳號的實際授權範圍，與業務所需權限比較服務帳號僅有業務必需的最小權限，不得用於交互式登入
AD 域帳號安全巡檢（PowerShell）
# ─── AD 域帳號安全巡檢 PowerShell 腳本 ───
Import-Module ActiveDirectory

# 1. 長期未登入的帳號（90天）
$cutoff = (Get-Date).AddDays(-90)
Get-ADUser -Filter {LastLogonDate -lt $cutoff -and Enabled -eq $true} `
  -Properties LastLogonDate, Department, Manager |
  Select-Object Name, SamAccountName, LastLogonDate, Department |
  Export-Csv "inactive_users.csv" -Encoding UTF8
Write-Host "長期未登入帳號：" (Get-ADUser -Filter {LastLogonDate -lt $cutoff -and Enabled -eq $true}).Count

# 2. 密碼永不過期的帳號（潛在風險）
Get-ADUser -Filter {PasswordNeverExpires -eq $true -and Enabled -eq $true} `
  -Properties PasswordNeverExpires |
  Select-Object Name, SamAccountName, DistinguishedName |
  Export-Csv "password_never_expires.csv" -Encoding UTF8

# 3. 域管理員帳號清單（人數應嚴格控制）
Get-ADGroupMember -Identity "Domain Admins" |
  Select-Object Name, SamAccountName, objectClass
Write-Host "域管理員數量：" (Get-ADGroupMember "Domain Admins").Count
# 一般組織域管不應超過 5 個，且應有具體業務理由

# 4. 禁用帳號清單（應定期清理）
Get-ADUser -Filter {Enabled -eq $false} -Properties WhenChanged |
  Where-Object {$_.WhenChanged -lt (Get-Date).AddDays(-365)} |
  Select-Object Name, SamAccountName, WhenChanged |
  Export-Csv "old_disabled_accounts.csv" -Encoding UTF8

# 5. 帳號鎖定策略核查
$policy = Get-ADDefaultDomainPasswordPolicy
Write-Host "密碼最小長度：" $policy.MinPasswordLength
Write-Host "密碼最大有效期：" $policy.MaxPasswordAge.Days "天"
Write-Host "帳號鎖定閾值：" $policy.LockoutThreshold "次失敗"
Write-Host "複雜度要求：" $policy.ComplexityEnabled

安全設備策略巡檢（IDS/IPS/WAF）

安全設備若未得到及時維護（特徵庫更新、誤報調優、規則清理），往往會演變成擺設——要麼因誤報太多被業務投訴後靜默，要麼規則太老無法識別新型攻擊。

IDS/IPS 巡檢要點
特徵庫（Signature）版本及最後更新時間（應在 7 天內）；是否有大量低置信度規則觸發（可能是誤報或掃描噪音）；IPS 攔截率統計（過去 30 天攔截事件數量及類型）；高危告警是否有及時響應記錄；Bypass 接口是否有異常（設備宕機是否會旁路）。
WAF 巡檢要點
規則集版本及最後更新時間；防護模式是否為攔截模式（Detection vs Block）；過去 30 天 Top 攻擊類型統計；誤報率核查（是否有合法業務流量被誤攔截）；自訂規則清單核查（是否有業務白名單規則過度寬鬆）；HTTPS 解密配置是否正確（明文流量是否都經過 WAF 檢測）。
防毒/EDR 巡檢要點
病毒庫更新狀態（應在 24 小時內）；終端覆蓋率（多少主機已部署 / 應覆蓋主機總數）；過去 30 天告警統計（檢測數、阻斷數、漏掉數）；是否有未處理的高危告警；失聯主機列表（長時間未上報狀態的終端）。
VPN 巡檢要點
VPN 協議版本（SSL VPN / IPSec，禁用 SSLv3/TLS1.0）；並發連接數是否接近授權上限；異常登入記錄（非工作時間、非常用 IP）；證書有效期核查（是否即將過期）；訪問控制：VPN 用戶能訪問哪些內網資源（是否過於寬鬆）。

備份與災備策略巡檢

備份是最後一道防線，尤其是在勒索軟件攻擊後。但「有備份」不等於「能恢復」——許多組織在真正發生災難時才發現備份已損壞或恢復流程從未測試過。

備份巡檢核查清單
一、備份完整性核查
□ 確認關鍵系統（資料庫/核心業務系統/文件服務）的備份任務是否在規律執行
  - 核查備份作業的最後成功執行時間（應在設定週期內）
  - 核查備份文件大小趨勢（異常縮小可能表示備份損壞）
□ 抽查驗證備份文件完整性
  - 數據庫備份：嘗試還原到測試環境並驗證數據完整性
  - 文件備份：抽查 5-10 個文件確認可正常讀取
□ 備份日誌核查：過去 30 天是否有備份失敗記錄？失敗是否有告警通知？

二、備份策略符合性
□ 是否遵循 3-2-1 備份原則：
    - 3 份備份數據
    - 2 種不同存儲介質（本地磁盤 + NAS 或雲）
    - 1 份異地備份（物理隔離，防範勒索軟件橫向加密備份）
□ 備份保留週期是否符合業務要求（等保/合規通常要求 ≥6 個月日誌、3 個月數據快照）
□ 備份數據是否加密存儲（防止備份介質丟失導致數據洩露）

三、容災恢復（DR）驗證
□ 上次恢復演練時間（應每半年至少演練一次）
□ RTO 目標（Recovery Time Objective）：需多長時間恢復？上次演練是否達標？
□ RPO 目標（Recovery Point Objective）：最多丟失多少數據？備份頻率是否滿足？
□ 恢復文檔是否完整（任何人拿到文檔都能執行恢復，而非依賴關鍵人員）

四、勒索軟件防護特殊要求
□ 備份網絡是否與生產網絡隔離（防止勒索軟件橫向加密在線備份）
□ 是否有不可變備份（Immutable Backup）——即使管理員也無法刪除
□ 備份管理系統的訪問憑證是否與生產系統憑證隔離
□ 備份是否定期做離線副本（磁帶或斷網存儲）

有效的日誌分析前提是完善的日誌收集體系。安全巡檢中的日誌分析環節，首先需要確認日誌收集是否完整，再進行異常行為分析。

巡檢中常見的日誌盲區包括：（1）DNS 查詢日誌未收集（C2 通信常利用 DNS 隧道）；（2）雲平台操作日誌（AWS CloudTrail/阿里雲 ActionTrail）未開啟；（3）容器/K8s 日誌未接入 SIEM；（4）辦公終端日誌覆蓋不全；（5）NTP 時間未同步導致日誌時間戳不一致，無法關聯分析。

系統與網絡日誌分析

系統登入日誌和網絡流量日誌是最核心的安全日誌，能反映帳號入侵嘗試、橫向移動、異常外連等關鍵威脅指標（IoC）。

Linux 系統安全日誌分析命令
# ─── SSH 登入安全分析 ───
# 統計 SSH 登入失敗的 IP（暴力破解溯源）
grep "Failed password" /var/log/auth.log | \
  awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head -20

# 統計每個 IP 的失敗次數（超過 50 次視為暴力破解）
grep "Failed password" /var/log/auth.log | \
  awk '{print $(NF-3)}' | sort | uniq -c | \
  awk '$1 > 50 {print "暴力破解嫌疑:", $1, "次失敗, 來源:", $2}'

# 成功登入記錄（確認是否有非預期的成功登入）
grep "Accepted" /var/log/auth.log | tail -50

# 非工作時間（22:00-06:00）的成功登入（可疑）
grep "Accepted" /var/log/auth.log | \
  awk '$3 ~ /^(2[2-9]|0[0-5]):/ {print "非工作時間登入:", $0}'

# ─── sudo 和特權操作日誌 ───
grep "sudo" /var/log/auth.log | grep "COMMAND" | tail -30
# 特別注意：sudo /bin/bash、sudo su、sudo chmod 等可能表示提權操作

# ─── 異常進程與網絡連接（當前狀態）───
# 非標準端口監聽（高危）
ss -tlnp | grep -v ':22\|:80\|:443\|:25\|:53\|:3306'

# 已建立的外網連接（排查惡意外連）
ss -tnp state established | grep -v '127.0.0.1\|10\.\|192\.168\.\|172\.'
# 對可疑 IP 進行 whois 查詢

# ─── 系統完整性快速核查 ───
# 最近 7 天修改的關鍵系統文件（可能是後門）
find /etc /usr/bin /usr/sbin /bin /sbin -newer /tmp -type f 2>/dev/null | head -20
# 最近創建的 SUID 文件（後門常用）
find / -perm -4000 -newer /tmp -type f 2>/dev/null

Web 訪問日誌與攻擊特徵識別

Web 訪問日誌是識別針對業務系統攻擊行為的關鍵數據源。通過分析 HTTP 請求特徵，可以發現 SQL 注入嘗試、XSS、目錄掃描、漏洞探測等攻擊活動。

Nginx/Apache Web 日誌攻擊特徵分析
# ─── 訪問統計概覽 ───
# 訪問量最大的前 20 IP（確認是否有異常高頻訪問）
awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20

# HTTP 狀態碼統計（大量 4xx = 掃描；大量 5xx = 攻擊或故障）
awk '{print $9}' /var/log/nginx/access.log | sort | uniq -c | sort -rn
# 重點：大量 403（目錄遍歷/權限繞過嘗試）、404（路徑掃描）、500（代碼執行嘗試）

# ─── SQL 注入特徵識別 ───
grep -iE "union.+select|select.+from|insert.+into|drop.+table|or.+1=1|and.+1=1|'.*--|\/\*.*\*\/" \
  /var/log/nginx/access.log | head -20

# ─── 路徑掃描 / 目錄枚舉 ───
# 單個 IP 在短時間大量 404 = 目錄掃描
awk '$9==404 {print $1}' /var/log/nginx/access.log | sort | uniq -c | \
  awk '$1 > 50 {print "掃描嫌疑 IP:", $2, "404次數:", $1}'

# 常見掃描工具 User-Agent 識別
grep -iE "sqlmap|nmap|nikto|masscan|zgrab|dirsearch|gobuster|wfuzz|nessus|openvas|acunetix|awvs" \
  /var/log/nginx/access.log | awk '{print $1, $7}' | sort -u | head -20

# ─── 敏感路徑訪問 ───
grep -E "\.git|\.env|\.aws|phpinfo|wp-admin|/admin|/.config|/backup|\.bak|\.sql" \
  /var/log/nginx/access.log | grep -v " 404 "
# 返回非 404 的敏感路徑訪問需重點調查

# ─── 大量數據外傳（可能的數據洩露）───
# 找出響應體積異常大的請求（可能為數據大量導出）
awk '{if ($10+0 > 1000000) print $1, $7, $10, "bytes"}' /var/log/nginx/access.log | \
  sort -k4 -rn | head -20

# ─── 今日攻擊 IP 彙整（生成黑名單建議）───
today=$(date +%d/%b/%Y)
grep "$today" /var/log/nginx/access.log | \
  grep -iE "union|select|script|eval\(|base64|\.\./" | \
  awk '{print $1}' | sort -u | tee /tmp/attack_ips_today.txt
echo "今日攻擊來源 IP 共" $(wc -l < /tmp/attack_ips_today.txt) "個"

SIEM 平台巡檢告警分析

SIEM（Security Information and Event Management）平台集中收集和關聯分析安全日誌，是安全運營的核心平台。巡檢中需重點確認告警規則有效性和告警響應情況。

SIEM 巡檢核心關注點

01
告警規則覆蓋率 — 確認 SIEM 告警規則是否覆蓋主要威脅場景：暴力破解、帳號異常行為、內網橫向移動、可疑外連、數據大量導出、高危漏洞攻擊。核查是否有關鍵告警規則處於停用狀態（通常因誤報率高被臨時停用後忘記恢復）。
02
告警響應時效 — 抽查過去 30 天的高危告警處理記錄，核查：從告警產生到開始處置的平均時長（MTTD）；是否存在高危告警超過 24 小時未處理的情況；告警關閉的原因是否有記錄（真陽性/誤報/已知例外）。
03
告警疲勞核查 — 統計每日平均告警數量，若告警量過大（如每天數千條），運營人員往往會忽略所有告警。核查高頻誤報告警的規則，建議調優或加入白名單。健康的 SIEM 每天應有可人工審查的告警數量（通常 50-200 條需要關注的告警）。
04
數據接入完整性 — 確認所有應接入的日誌源都已正常接入，且日誌流量在正常範圍內（突然歸零可能表示日誌斷流）。ELK 環境中可通過 Kibana 的 Data Streams 視圖快速確認各來源的日誌接收狀態。
SIEM 巡檢常見發現

在大量巡檢實踐中，最常見的 SIEM 問題包括：（1）某網段日誌未接入，形成監控盲區；（2）所有告警均被標記為「已處理」但無任何處置說明（形式主義關閉告警）；（3）NTP 時間不同步導致多源日誌時間錯位，關聯規則失效；（4）磁盤空間不足導致舊日誌被自動刪除，保留週期不達標；（5）告警規則長期未更新，對新型攻擊手法完全無告警能力。

定期漏洞掃描規劃與執行

定期漏洞掃描是安全巡檢的核心技術手段之一，通過與上次掃描結果對比，可以追蹤漏洞修復進展、發現新增漏洞，並評估整體脆弱性趨勢。

掃描規劃原則

掃描類型頻率覆蓋範圍工具建議
互聯網暴露面掃描每週（自動）所有外網 IP 和域名；專注高危端口和 Web 應用Nuclei（新漏洞快速驗證）、AWVS（Web 深度掃描）、Shodan/Fofa 監控
內網全量掃描每月（自動）所有內網資產；完整端口掃描 + 服務識別Nessus（全面性好）、OpenVAS（開源）
新系統上線前掃描按需（每次上線前）待上線系統及其依賴組件Nessus + 手工驗證高危發現
高危 CVE 專項掃描按需（重大 CVE 發布後 48 小時內）全網相關資產Nuclei 單一模板快速掃描
Nessus 掃描結果導出與對比分析（Python）
import json
import pandas as pd
from datetime import datetime

# ─── 讀取本次和上次掃描結果（Nessus .csv 導出格式）───
# Nessus 導出：掃描任務 → Export → CSV
current = pd.read_csv('scan_2024_Q4.csv')
previous = pd.read_csv('scan_2024_Q3.csv')

# 使用「主機IP + 插件ID」作為唯一標識
current['key'] = current['Host'] + '_' + current['Plugin ID'].astype(str)
previous['key'] = previous['Host'] + '_' + previous['Plugin ID'].astype(str)

current_keys = set(current['key'])
previous_keys = set(previous['key'])

# ─── 趨勢分析 ───
new_vulns = current_keys - previous_keys        # 本次新增漏洞
fixed_vulns = previous_keys - current_keys      # 已修復漏洞（上次有，本次沒有）
persistent = current_keys & previous_keys       # 持續存在的漏洞

print(f"本次掃描漏洞總數：{len(current_keys)}")
print(f"相比上次新增：{len(new_vulns)}")
print(f"已修復：{len(fixed_vulns)}")
print(f"持續存在：{len(persistent)}")

# ─── 本次新增漏洞詳情（重點關注高危/嚴重）───
new_df = current[current['key'].isin(new_vulns)]
critical_new = new_df[new_df['Risk'].isin(['Critical', 'High'])]
print("\n本次新增高危/嚴重漏洞：")
print(critical_new[['Host', 'Risk', 'Name', 'Plugin ID']].to_string())

# ─── 生成巡檢月報數據 ───
risk_dist = current['Risk'].value_counts()
print("\n本次掃描風險分布：")
for risk, count in risk_dist.items():
    print(f"  {risk}: {count}")

補丁管理流程與合規核查

補丁管理（Patch Management）是降低漏洞風險的根本手段，但在實際組織中往往面臨「打補丁可能影響業務穩定性」的阻力。安全巡檢需核查補丁管理流程是否成熟，以及關鍵系統的補丁狀態。

01
補丁評估窗口 — 核查組織是否有明確的補丁評估流程：新 CVE 發布後多少天內完成評估（建議 7 天內）；評估內容是否包含業務影響分析；是否有補丁優先級分類標準（緊急/高/普通，對應不同修補期限）。
02
高危漏洞修補時效 — 核查近期高危 CVE（CVSS ≥ 9.0）的修補時效：是否在 14 天內完成修補；若無法及時修補是否有臨時緩解措施（WAF 規則/網絡隔離）；是否有書面記錄說明延遲原因。等保 2.0 三級以上系統要求高危漏洞 30 天內修復。
03
作業系統補丁狀態 — 對關鍵伺服器進行補丁狀態核查，重點關注：Windows Server 上次更新時間（應在 30 天內）；Linux 系統是否有待安裝的安全更新（yum check-update --security）；是否有 EOL（生命週期結束）的系統在繼續使用（如 Windows Server 2008/CentOS 6）。
04
第三方組件補丁 — 核查業務系統所用的第三方組件（框架/中間件/開源庫）是否有未修復的高危漏洞。重點關注：Apache/Nginx/Tomcat/Spring 框架版本；Java 依賴包（mvn dependency:check）；Python/Node.js 依賴（pip-audit/npm audit）。
補丁狀態核查命令（多系統）
# ─── Linux 補丁核查 ───
# CentOS/RHEL：列出待安裝的安全更新
yum check-update --security
yum updateinfo list security        # 含 CVE 編號的安全更新清單
yum updateinfo summary              # 安全更新概要統計

# Ubuntu/Debian：列出待安裝的安全更新

互聯網暴露面持續監控

暴露面管理（Attack Surface Management，ASM）是近年安全運營的重要趨勢，核心理念是：攻擊者發現你的資產，比你自己更快。定期巡檢中必須包含對互聯網暴露面的系統性核查。

互聯網暴露面自動化監控腳本
#!/bin/bash
# ─── 互聯網暴露面週期巡檢腳本 ───
# 前提：已有上次巡檢的子域名清單 prev_subdomains.txt

DOMAIN="example.com"
DATE=$(date +%Y%m%d)
OUTDIR="/opt/asm_scans/$DATE"
mkdir -p $OUTDIR

# ─── 1. 子域名枚舉（發現新增子域名）───
echo "[*] 枚舉子域名..."
subfinder -d $DOMAIN -silent -all > $OUTDIR/subdomains.txt
# 對比上次結果，找出新增子域名
diff "prev_subdomains.txt" $OUTDIR/subdomains.txt | grep '^>' | \
  awk '{print $2}' > $OUTDIR/new_subdomains.txt
echo "新發現子域名：" $(wc -l < $OUTDIR/new_subdomains.txt) "個"

# ─── 2. 存活探測與端口掃描 ───
echo "[*] 探測存活主機..."
httpx -l $OUTDIR/subdomains.txt -silent -status-code -title -tech-detect \
  -o $OUTDIR/web_alive.txt

# ─── 3. 高危端口直接暴露核查 ───
echo "[*] 核查高危端口..."
while read subdomain; do
  ip=$(dig +short $subdomain | head -1)
  [ -z "$ip" ] && continue
  for port in 22 23 3306 3389 6379 27017 5432 11211; do
    nc -z -w 2 $ip $port 2>/dev/null && \
      echo "[高危] $subdomain ($ip) 端口 $port 對外開放" >> $OUTDIR/exposed_ports.txt
  done
done < $OUTDIR/subdomains.txt

# ─── 4. 新增資產立即用 Nuclei 掃描 ───
if [ -s $OUTDIR/new_subdomains.txt ]; then
  echo "[*] 對新增資產進行漏洞掃描..."
  nuclei -l $OUTDIR/new_subdomains.txt -severity high,critical \
    -o $OUTDIR/new_asset_vulns.txt
fi

# ─── 5. 輸出本次巡檢摘要 ───
echo "======= 暴露面巡檢摘要 ======="
echo "子域名總數：" $(wc -l < $OUTDIR/subdomains.txt)
echo "存活 Web 資產：" $(wc -l < $OUTDIR/web_alive.txt)
echo "新增子域名：" $(wc -l < $OUTDIR/new_subdomains.txt)
[ -f $OUTDIR/exposed_ports.txt ] && echo "高危端口暴露：" $(wc -l < $OUTDIR/exposed_ports.txt) "項"

常態化弱密碼掃描

弱密碼是最常見且危害最大的安全問題之一。在獲得明確授權的前提下，定期對內網服務進行弱密碼探測，是安全巡檢不可或缺的環節。

常見弱密碼類型

空密碼/默認密碼
admin:admin、root:root、test:test、admin:123456；設備廠商默認密碼（如 Cisco 的 admin:cisco，路由器 admin:password）；應用系統默認密碼（如 Tomcat Manager admin:tomcat、phpMyAdmin admin:root）。
組織相關弱密碼
公司名+年份（Company2024）；項目名+序號（Project@001）；域名前綴+常見後綴（example123、example@123）；系統名稱+123（Oracle123、MysqlAdmin1）。這類密碼人工難發現，需定制字典。
多服務弱密碼掃描（授權環境）
# ─── 注意：以下操作必須在書面授權範圍內進行！───

# ─── SSH 弱密碼掃描 ───
hydra -L /tmp/users.txt -P /tmp/passwords.txt -t 4 -T 10 \
  ssh://192.168.1.0/24 -o /tmp/ssh_weak.txt
# -t 4：每目標4個並發（太高可能影響業務）
# -T 10：最大10個目標並發

# ─── MySQL 弱密碼掃描 ───
hydra -L /tmp/db_users.txt -P /tmp/passwords.txt \
  -t 4 mysql://10.0.0.50 -o /tmp/mysql_weak.txt

# ─── Redis 未授權訪問核查（無需密碼）───
for ip in $(nmap -p 6379 --open 192.168.0.0/24 -oG - | grep '/open' | awk '{print $2}'); do
  result=$(redis-cli -h $ip -p 6379 ping 2>/dev/null)
  [ "$result" = "PONG" ] && echo "[高危] Redis 未授權訪問: $ip"
done

# ─── Web 後台弱密碼（Burp Intruder / 腳本）───
# 對登入接口進行有限次嘗試（注意：帳號鎖定閾值，不要超過 5 次/帳號）
for passwd in admin123 Admin@123 "admin@2024" P@ssw0rd Test1234; do
  resp=$(curl -s -o /dev/null -w "%{http_code}" -X POST \
    https://example.com/admin/login \
    -d "username=admin&password=$passwd")
  [ "$resp" != "401" ] && echo "疑似弱密碼: $passwd (HTTP $resp)"
done

# ─── 定制弱密碼字典（基於組織特徵）───
# 使用 CeWL 從組織網站爬取關鍵詞生成字典
cewl -d 2 -m 5 -w /tmp/org_words.txt https://example.com
小測驗
在對某企業進行季度安全巡檢時，日誌分析發現某個外部 IP（1.2.3.4）在過去 6 小時內對 Web 伺服器產生了 3000 次 HTTP 404 錯誤，且 User-Agent 為 dirsearch/0.4.2。這最可能代表什麼威脅，應如何處置？
A服務器故障導致 404，應通知業務負責人檢查應用
B主動目錄掃描（攻擊者使用 dirsearch 工具枚舉 Web 路徑），應記錄 IP 和發現，建議在 WAF 或防火牆封鎖該 IP，並核查是否有任何返回 200 的請求（表示路徑被成功發現）
C搜索引擎爬蟲正常訪問，不需要處置
DDDoS 攻擊，應立即聯繫 ISP 請求流量清洗
正確！dirsearch 是一款知名的 Web 目錄/路徑枚舉工具（攻擊者常用），短時間大量 404 是典型的目錄掃描特徵。正確處置步驟：（1）確認威脅：User-Agent 含 dirsearch + 大量 404 = 確定是目錄掃描；（2）確認是否有成功發現：過濾 User-Agent 為 dirsearch 且返回碼非 404 的請求（如有 200/301，則攻擊者已發現可訪問路徑）；（3）封鎖來源 IP：在 WAF 或防火牆加入黑名單規則；（4）記錄到巡檢報告中，作為外部威脅活動的證據；（5）如有成功返回的敏感路徑，需評估是否需要立即修復或限制訪問。

巡檢發現分類與評級

統一的發現分類和評級標準是確保不同批次巡檢結果可比較、可追蹤的基礎。不同於風險評估的多維度計算，巡檢發現評級更注重直接影響和修復緊迫性。

巡檢發現嚴重程度評級標準

等級定義示例整改期限
嚴重存在直接被利用的高危漏洞，或已有攻擊入侵跡象，可能導致系統完全淪陷或重要數據大規模洩露已知高危 CVE 未修補且有公開 PoC（如 Log4Shell）；發現活躍的惡意進程/後門；核心數據庫 root 帳號可從外網登入立即（24小時內）
高危存在可被利用的安全問題，成功利用可導致重大業務影響，但利用條件相對複雜高危漏洞未及時修補（CVSS 7.0-9.0）；高危服務（SSH/RDP）暴露互聯網；管理員帳號未啟用 MFA；防火牆默認策略為 ACCEPT3-7 天
中危存在安全隱患，在一定條件下可被利用或會為攻擊者提供信息，但直接影響有限中危漏洞未修補；存在長期未登入帳號；SSL/TLS 使用過時協議版本；缺少部分 HTTP 安全頭14-30 天
低危配置不符合最佳實踐，不會直接導致安全事件，但屬於縱深防禦的薄弱環節日誌保留週期不足；備份未加密；部分系統缺少防毒覆蓋；文檔未及時更新下次巡檢前
建議安全加強建議，現狀不算問題，但有改善空間建議部署 MFA（現狀有密碼策略）；建議升級到更新版本；建議完善安全制度根據優先級排期

發現分類維度

技術配置問題
系統/設備安全配置不符合基線要求。特點：有明確的「正確配置」標準，整改操作相對直接。例：SSH 未禁用 root 登入、防火牆規則冗餘、TLS 1.0 未停用。
漏洞風險
已知 CVE 漏洞未及時修補。特點：有 CVE 編號可查詢，CVSS 評分可直接參考，整改方式為升級版本或安裝補丁。
帳號與訪問控制問題
帳號管理不符合最小權限和生命週期管理要求。特點：需要業務確認（不能擅自停用帳號），整改需協調 HR 和業務。
管理制度問題
安全制度缺失或未執行。特點：整改涉及流程和文化，週期較長。例：無安全培訓記錄、無安全事件響應流程、補丁管理無書面流程。
異常事件線索
日誌分析中發現可疑活動，需進一步排查確認是否為真實安全事件。特點：不確定性高，需要應急響應流程介入。
合規差距
不符合等保/法規要求的具體差距。特點：以合規標準條款為參考依據，整改有明確的合規驗收標準。

綜合巡檢報告撰寫規範

巡檢報告是定期服務的核心交付物，其質量直接影響客戶對安全服務的滿意度和信任感。好的巡檢報告應讓客戶清楚地看到：本次發現了什麼問題、和上次相比安全狀況如何變化、下一步應該做什麼。

安全巡檢報告標準結構

01
封面與基本信息 — 報告標題、巡檢週期（第X次季度巡檢）、被巡檢方名稱、巡檢時間範圍、本次巡檢範圍摘要、報告出具日期、服務方蓋章、密級標識（通常為「機密」）。
02
管理層摘要（1頁） — 用非技術語言描述：本次巡檢的整體安全評分（如 A/B/C 等或 85/100）；嚴重/高危發現數量及最重要的 2-3 個問題；與上季度相比安全趨勢（好轉/穩定/惡化）；最高優先整改建議。
03
發現匯總與趨勢分析 — 本次發現統計表（按嚴重程度分類：嚴重/高/中/低/建議各多少條）；上次未整改發現閉環情況（已修復X條，持續存在X條）；本次新增發現X條；安全趨勢圖（按時間軸展示歷次巡檢發現數量變化）。
04
技術發現詳情 — 按嚴重程度排序，每個發現包含：唯一編號、發現標題、嚴重程度、受影響資產、問題描述（技術細節）、風險說明（業務影響）、截圖或代碼片段（客觀證據）、整改建議（具體可執行步驟）、整改期限。
05
整改工單清單 — 所有需要整改的發現生成可操作的工單清單，包含：工單編號、問題描述、嚴重程度、責任人、截止日期、整改建議。此清單應可直接作為工作任務分發給各負責人。
06
上次整改情況複驗 — 逐條核查上次巡檢高危/嚴重發現的整改情況：已整改並複驗通過（附複驗截圖）；已整改但複驗失敗（說明原因）；未整改（說明原因，重新評估風險等級）；延期（新截止日期）。

整改追蹤與閉環管理

安全巡檢的價值不只在發現問題，更在於推動問題得到解決。有效的整改追蹤機制是安全服務持續創造價值的關鍵。

整改閉環管理流程

Day 0
發現記錄與工單創建
巡檢結束後即創建整改工單，明確：問題描述、嚴重程度、指定責任人（IT 負責人或具體系統管理員）、整改截止日期（嚴重：24h；高危：7天；中危：30天）。
整改期
整改執行與進度更新
責任人按整改建議執行修復操作；重要操作前做好回滾準備；整改完成後在工單中更新狀態並說明修復方式；如遇阻礙（業務原因無法按期整改）需及時溝通並申請延期，說明臨時緩解措施。
截止前
整改驗證（複查）
服務方對已整改發現進行技術複驗：執行相同的測試步驟；確認問題確實已修復；記錄複驗截圖作為閉環證據；更新工單狀態為「已驗證關閉」。若複驗失敗，重新開啟工單並協助分析原因。
下次巡檢
閉環狀態更新到下次報告
在下次巡檢報告的「上次整改情況複驗」章節，逐條呈現整改結果：修復率、按期完成率；持續未整改的問題提升一個嚴重等級（如高危升為嚴重）；形成問題台帳的歷史記錄。
整改問題台帳（Excel 模板結構）
安全巡檢問題台帳欄位設計（Excel / 項目管理工具）：

工單編號     │ 格式：YYYYQX-NNN（如 2024Q4-001）
巡檢批次     │ 哪次巡檢發現（如 2024年第4季度）
嚴重程度     │ 嚴重/高危/中危/低危/建議
發現類別     │ 技術配置/漏洞/帳號管理/管理制度/異常線索/合規差距
問題簡述     │ 一句話描述（如：Redis 服務未設置密碼，允許未授權訪問）

CVE 編號     │ 如有（如 CVE-2022-0543）

複驗日期     │ 
複驗人       │ 
備注         │ 延期原因/臨時緩解措施/接受風險說明

巡檢自動化腳本開發

成熟的安全巡檢體系需要將重複性工作自動化，既提高效率，又降低人工失誤風險。下面以實際可用的 Python 自動化巡檢框架為示例。

Python 綜合巡檢框架
import subprocess, datetime, json, csv, os
import paramiko   # SSH 連接庫: pip install paramiko
import pymysql    # MySQL 連接: pip install pymysql

# ─── 配置：巡檢目標清單 ───
CONFIG = {
    'targets': [
        {'ip': '192.168.1.10', 'type': 'linux', 'user': 'auditor', 'key': '/tmp/audit.pem'},
        {'ip': '192.168.1.20', 'type': 'mysql', 'user': 'audit_ro', 'password': '****'},
    ],
    'output_dir': f'/opt/inspections/{datetime.date.today()}'
}

class LinuxInspector:
    def __init__(self, ip, user, key_path):
        self.ip = ip
        self.ssh = paramiko.SSHClient()
        self.ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
        self.ssh.connect(ip, username=user, key_filename=key_path)
        self.findings = []

    def run_cmd(self, cmd):
        _, stdout, _ = self.ssh.exec_command(cmd)
        return stdout.read().decode().strip()

    def check_root_login(self):
        # 核查 SSH 是否禁用 root 直接登入
        val = self.run_cmd("grep -i 'PermitRootLogin' /etc/ssh/sshd_config | grep -v '^#'")
        if 'no' not in val.lower():
            self.findings.append({
                'id': 'SRV-001', 'level': 'high',
                'title': 'SSH 未禁用 root 直接登入',
                'detail': f'PermitRootLogin 當前值: {val or "未配置（默認允許）"}',
                'fix': '設置 PermitRootLogin no 並重啟 sshd'
            })

    def check_uid0_accounts(self):
        # 核查是否有非 root 的 UID=0 帳號
        result = self.run_cmd("awk -F: '($3==0 && $1!=\"root\"){print $1}' /etc/passwd")
        if result:
            self.findings.append({
                'id': 'SRV-002', 'level': 'critical',
                'title': '發現非 root 的 UID=0 帳號（潛在後門）',
                'detail': f'UID=0 帳號: {result}',
                'fix': '立即確認帳號合法性，若為非授權帳號立即刪除'
            })

    def run_all_checks(self):
        for check in [self.check_root_login, self.check_uid0_accounts]:
            try: check()
            except Exception as e:
                print(f"核查失敗 {check.__name__}: {e}")
        self.ssh.close()
        return {'target': self.ip, 'findings': self.findings}

# ─── 輸出報告 ───
def generate_report(all_results, output_dir):
    os.makedirs(output_dir, exist_ok=True)
    # JSON 原始數據
    with open(f'{output_dir}/findings.json', 'w') as f:
        json.dump(all_results, f, ensure_ascii=False, indent=2)
    # CSV 工單導出
    with open(f'{output_dir}/findings.csv', 'w', newline='') as f:
        w = csv.DictWriter(f, ['target','id','level','title','detail','fix'])
        w.writeheader()
        for r in all_results:
            for f_ in r['findings']:
                w.writerow({'target': r['target'], **f_})
    print(f"報告已輸出至 {output_dir}")
