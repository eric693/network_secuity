網路課程概述與通訊概念

基礎概念 預計閱讀時間：15分鐘

為什麼要學網路？

網路是資安工程師最重要的基礎。無論是滲透測試、入侵偵測、還是應急響應，幾乎每一項安全工作都必須建立在對網路的深刻理解之上。如果你不懂 TCP/IP、不知道封包如何傳輸、不理解路由與交換的差異，就很難真正理解攻擊的本質。

核心觀念

資安工程師必須先是一名合格的網路工程師。所有的攻擊流量，都必須透過網路傳遞；所有的防禦機制，也都建立在網路協議之上。

什麼是網路通訊？

網路通訊是指兩台或多台設備之間，透過某種媒介（如電纜、光纖、無線電波）交換資料的過程。要讓這個過程順利進行，需要解決三個核心問題：

如何尋址？
每台設備需要一個唯一的地址（如 IP 地址），這樣資料才知道該發往哪裡，就像郵寄信件需要門牌號碼。
如何路由？
資料在從來源到目的地的過程中，需要經過多個中轉節點，路由器負責決定每一跳的最佳路徑。
如何確保可靠？
網路中的資料傳輸可能發生丟失或錯誤，協議（如 TCP）負責保證資料能完整、正確地到達目的地。
如何轉換資料格式？
不同的應用程式產生不同格式的資料，網路協議定義了如何對資料進行封裝，使其能在網路上統一傳輸。

資安視角：理解網路的重要性

從資安的角度來看，理解網路通訊可以幫助你：

識別異常流量 — 當你看到大量的 ICMP 封包或者不尋常的端口掃描時，能夠快速判斷這是攻擊行為。

理解攻擊原理 — ARP 欺騙、DNS 污染、SYN Flood 等攻擊，本質上都是對網路協議的濫用。

設計防禦策略 — 防火牆規則、ACL 配置、網路分段，都需要對網路架構有深刻理解。

交換機工作原理深度解析

交換機是現代企業網路的核心設備，維護一張 CAM 表（Content Addressable Memory Table），也叫 MAC 地址表。它記錄了每個交換機端口對應的 MAC 地址，讓交換機能精確轉發而不是廣播。

交換機 CAM 表學習過程（完整示例）： 網路環境： 端口1 ── PC_A (MAC: AA:AA:AA:AA:AA:AA, IP: 192.168.1.10) 端口2 ── PC_B (MAC: BB:BB:BB:BB:BB:BB, IP: 192.168.1.20) 端口3 ── PC_C (MAC: CC:CC:CC:CC:CC:CC, IP: 192.168.1.30) ── 初始狀態：CAM 表為空 ── 步驟1：PC_A 發幀給 PC_B（目標 MAC: BB:BB） 交換機收到 → 來源 MAC AA:AA 在端口1 → 寫入 CAM 表 查 CAM 表找 BB:BB → 沒有！ 動作：洪泛廣播（Flood）→ 端口2和端口3都收到 CAM 表：端口1 = AA:AA 步驟2：PC_B 回應 PC_A（目標 MAC: AA:AA） 交換機收到 → 來源 MAC BB:BB 在端口2 → 寫入 CAM 表 查 CAM 表找 AA:AA → 在端口1！ 動作：只從端口1轉發（單播！PC_C 收不到） CAM 表：端口1=AA:AA, 端口2=BB:BB 步驟3：之後 PC_A 再發給 PC_B 查 CAM 表找 BB:BB → 在端口2 動作：只從端口2轉發（精確單播） → PC_C 無法嗅探到 PC_A 和 PC_B 的通信！
攻擊：MAC Flooding（MAC 洪泛）

攻擊工具 macof（Kali Linux 自帶）每秒可發送數千個隨機 MAC 地址的幀，耗盡 CAM 表（通常容量 8,000～16,000 條）。CAM 表滿後，交換機對目標 MAC 未知的幀只能洪泛廣播，效果等同於 Hub——攻擊者就能用 Wireshark 嗅探所有人的流量。

防禦：端口安全（Port Security）——限制每個端口能學習的 MAC 地址數量（通常設為 1 或 3），超過限制則關閉端口（err-disable）並告警。

路由器的路由表

路由器維護的路由表是所有路由決策的依據。每一條路由條目包含：目的網路、子網遮罩、下一跳（Next Hop）、出口介面、度量值（Metric）、來源（靜態/動態協議學習）。

資安工程師的思維框架

當你看到一個企業網路時，應該習慣性地問：「這裡部署了什麼設備？它工作在哪一層？有什麼攻擊面？」例如看到開放 80 端口的服務器前面有 WAF → 你需要考慮 WAF 繞過技術（編碼、分塊傳輸、HTTP 協議模糊）。看到只有包過濾防火牆 → 應用層攻擊可能不被攔截。

小測驗
攻擊者發現目標企業只部署了包過濾防火牆（只過濾 IP/端口），並且允許 TCP 80（HTTP）進入。攻擊者在 HTTP 請求中嵌入 SQL 注入 Payload。防火牆會攔截這個攻擊嗎？
A會攔截，因為 SQL 注入是已知攻擊，所有防火牆都有規則
B不會攔截。包過濾防火牆只看 IP/端口，不解析 HTTP 內容，允許 80 端口就放行所有 HTTP 流量
C不確定，取決於防火牆品牌
D會攔截，因為 SQL 注入包的封包大小異常
正確！包過濾防火牆工作在第三/四層，它只能看到「這是一個 TCP 封包，目標端口 80，來源 IP 合法」→ 放行。它完全不理解 HTTP 協議的內容，更不知道 URL 參數裡有 SQL 注入的 Payload。要防禦應用層攻擊，必須部署 WAF（Web Application Firewall）或 NGFW（下一代防火牆）進行應用層深度包檢測。

網路拓撲結構

網路拓撲描述了網路中各設備的物理或邏輯連接方式。不同的拓撲有不同的特性，直接影響網路的性能、可靠性和安全性。

拓撲類型 結構特點 優點 缺點 常見場景
匯流排型 Bus 所有設備連到一條主幹線 佈線簡單、成本低 主幹線故障全網癱瘓、難排錯 早期以太網，現已淘汰
星型 Star 所有設備連到中央設備 管理方便、單點故障影響小 中央設備成單點故障 現代辦公室最常見
環形 Ring 設備依次首尾相連成環 傳輸效率高 單點故障影響整環 部分骨幹網
樹型 Tree 星型拓撲的層次化擴展 易擴展、層次清晰 上層節點故障影響大 企業多樓層網路
網狀 Mesh 設備間有多條路徑連接 高可靠性、無單點故障 成本高、配置複雜 數據中心、骨幹網

OSI 七層模型深度解析

OSI（Open Systems Interconnection）七層模型是 1984 年 ISO 發布的網路通信標準框架，是網路工程師和資安工程師的共同語言。理解 OSI 不是為了背誦，而是為了在面對任何攻擊或故障時，能快速定位問題發生在哪一層、部署防禦在哪一層。

為什麼 OSI 對資安人員格外重要

所有安全工具的設計、CVE 漏洞的描述、滲透測試的分析，都基於 OSI 框架。當你說「這個漏洞在應用層」和「這個攻擊在傳輸層」，你就能知道：用哪種工具偵測、防火牆規則放在哪、封包捕獲要看哪些欄位。

第一層：物理層（Physical Layer）

物理層是最底層，負責定義如何在物理介質（網線、光纖、無線電波）上傳輸原始的位元（0 和 1）流。它只關心訊號的物理特性，完全不理解資料的含義。

  • PDU 單元：位元（Bit）
  • 代表技術：RJ-45 介面規格、Cat6 網線標準、IEEE 802.3 乙太網物理標準、Wi-Fi 射頻規格
  • 安全威脅：物理竊聽（搭線）、電磁輻射竊聽（Van Eck Phreaking）、硬體植入後門（如供應鏈攻擊中植入竊聽晶片）
  • 防禦：物理安全（機房門禁、攝影機）、使用光纖（比銅纜更難竊聽）、Faraday 屏蔽（軍事設施）

第二層：資料鏈路層（Data Link Layer）

資料鏈路層負責在同一個廣播域（局域網）內的設備之間可靠傳輸幀，使用 MAC 地址定址，並提供基本的錯誤偵測（FCS 幀校驗序列）。分為 LLC（邏輯鏈路控制）和 MAC（媒體訪問控制）兩個子層。

  • PDU 單元：幀（Frame）
  • 代表協議：Ethernet（802.3）、Wi-Fi（802.11）、PPP
  • 代表設備：交換機（Switch）、無線 AP
  • 安全威脅：ARP 欺騙（中間人）、MAC 洪泛攻擊、VLAN Hopping、STP 拓撲攻擊
  • 防禦：動態 ARP 檢測（DAI）、端口安全（Port Security）、BPDU Guard

第三層：網路層（Network Layer）

網路層負責在不同網段之間傳輸資料包，使用 IP 地址進行邏輯定址，並由路由器根據路由表決定每個包的最優路徑（路由決策）。這一層實現了端到端（End-to-End）的通信。

  • PDU 單元：包（Packet）
  • 代表協議：IPv4、IPv6、ICMP（ping）、OSPF、BGP、IGMP
  • 代表設備：路由器（Router）、三層交換機（L3 Switch）
  • 安全威脅：IP 欺騙（Spoofing）、ICMP Flood（Ping of Death）、路由劫持（BGP Hijacking）、Smurf 攻擊
  • 防禦：Ingress/Egress 過濾、uRPF（單播反向路徑轉發）、BGP 認證（BGPsec）
BGP 劫持：互聯網路由的最大威脅

BGP（邊界閘道協議）是互聯網路由的基礎，負責在不同 AS（自治系統）間交換路由信息。BGP 默認信任所有鄰居的路由宣告——這是一個致命的設計缺陷。2018 年，有人宣告了 Google 的 IP 段（8.8.8.8 等），導致全球部分 Google 流量被路由到俄羅斯 Rostelecom 達 74 分鐘。這種攻擊叫做 BGP 劫持，輕則流量被重定向，重則 HTTPS 被中間人攻擊。

第四層：傳輸層（Transport Layer）

傳輸層提供進程到進程（Process-to-Process）的通信服務，用端口號（Port Number 0～65535）區分同一主機上的不同應用進程。TCP 和 UDP 是這層最重要的兩個協議。

小測驗
Heartbleed（CVE-2014-0160）是 OpenSSL 的一個嚴重漏洞，攻擊者可以讀取伺服器記憶體中的私鑰和密碼。它主要影響 OSI 哪一層？為什麼？
A第三層（網路層）——影響了伺服器的 IP 地址通信
B第四層（傳輸層）——TLS 運行在 TCP 之上
C第六層（表示層）——TLS/SSL 協議屬於表示層，負責加密解密
D第七層（應用層）——因為是透過 HTTPS 傳輸觸發的
正確！TLS/SSL 屬於 OSI 第六層（表示層），負責資料格式轉換和加密解密。Heartbleed 是 OpenSSL 在實現 TLS 心跳擴展（RFC 6520）時的邊界未檢查漏洞：攻擊者發送一個聲稱長度為 65535 但實際只有 1 bytes 的心跳請求，伺服器從記憶體中讀取 64KB 並回傳，其中包含私鑰、密碼、Cookie 等敏感內容。即使是 HTTPS（第7層應用），其加密是在第6層完成的，所以這個漏洞本質上是表示層的問題。

資料封裝與解封過程

當應用程式發送資料時，資料需要逐層向下封裝（添加每層的頭部信息），最終以電訊號的形式在物理媒介上傳輸。接收方則逐層向上解封，最終將資料交給應用程式。

封裝過程（發送方）

應用層資料（HTTP Request）："GET /index.html HTTP/1.1" [應用層] → 產生資料（Data） ↓ 加上傳輸層頭部 [傳輸層] → 產生段（Segment）= TCP頭 + 資料 ↓ 加上網路層頭部 [網路層] → 產生包（Packet）= IP頭 + 段 ↓ 加上資料鏈路層頭部尾部 [資料鏈路層] → 產生幀（Frame）= MAC頭 + 包 + FCS ↓ 轉換為物理訊號 [物理層] → 在網線上傳輸比特（0 和 1）

各層封裝的信息

傳輸層 — Segment
添加來源端口、目標端口、序列號等。端口號用於區分同一台主機上的不同服務（如 80=HTTP、443=HTTPS、22=SSH）。
網路層 — Packet
添加來源 IP、目標 IP。IP 地址用於跨網路（跨路由器）定址，實現端到端的路由。
資料鏈路層 — Frame
添加來源 MAC、目標 MAC、FCS（幀校驗序列）。MAC 地址用於同一局域網內的設備定址。
路由器的特殊行為
每經過一個路由器，幀（Frame）的 MAC 地址會重新封裝（更新為下一跳的 MAC），但 IP 地址不變（除非有 NAT）。
Wireshark 抓包知識點

當你用 Wireshark 抓包時，能看到每個封包的完整層次結構：乙太網幀頭部（MAC 地址）、IP 頭部（IP 地址）、TCP/UDP 頭部（端口號）、應用層資料（HTTP 請求等）。這個分層結構對於後面分析惡意流量非常重要。

實際案例：你打開 google.com 發生了什麼？

讓我們把所有知識串聯起來，看看一個完整的 HTTP 請求過程：

步驟一：DNS 解析 — 瀏覽器發現 google.com 是個域名，先查詢本機 hosts 文件，沒有就發 DNS 查詢給設定的 DNS 伺服器，獲取 google.com 的 IP 地址（如 142.250.80.46）。

步驟二：路由選擇 — 你的電腦發現目標 IP 不在本地局域網，資料需要送到默認閘道（路由器）。

步驟三：ARP 解析 — 電腦知道閘道的 IP，但不知道其 MAC 地址，發送 ARP 廣播詢問「誰是 192.168.1.1？」，路由器回應自己的 MAC。

步驟四：封裝與傳送 — 資料逐層封裝，目標 MAC 是路由器 MAC，目標 IP 是 google.com 的 IP，在本地網路傳到路由器。

步驟五：路由轉發 — 路由器剝除幀頭，根據 IP 路由表找到下一跳，重新封裝幀頭，一跳一跳向 google.com 方向轉發。

小測驗
資料從你的電腦傳送到另一個網段的伺服器，在經過路由器時，哪個地址會改變，哪個不會？
A IP 地址改變，MAC 地址不變
B MAC 地址改變（更新為下一跳），IP 地址不變
C IP 地址和 MAC 地址都改變
D IP 地址和 MAC 地址都不變
正確！路由器在第三層（網路層）工作，每經過一個路由器，幀（Frame）的 MAC 地址（來源和目標）都會被更新為對應鏈路上的 MAC 地址，但 IP 頭部的來源 IP 和目標 IP 始終保持不變（除非有 NAT 轉換）。這就是為什麼 IP 地址可以實現端到端的定址。

TCP 三次握手與四次揮手

TCP 是互聯網上最重要的傳輸層協議之一，它提供可靠的、面向連接的資料傳輸。理解 TCP 的工作機制，是理解很多網路攻擊（如 SYN Flood、TCP 劫持）的基礎。

TCP 三次握手（建立連接）

TCP 在傳輸資料之前，必須先通過三次握手建立連接。這個過程確保雙方都有發送和接收能力。

三次握手過程： 客戶端 (Client) 伺服器 (Server) | | | ──── SYN (seq=x) ─────────────────> | 第一次握手： | Client 說：我想連線，我的序號是 x | SYN_SENT 狀態 | | | <─── SYN+ACK (seq=y, ack=x+1) ──── | 第二次握手： | Server 說：好的，我的序號是 y， | SYN_RECEIVED 狀態 | 我確認你的序號 x | | | | ──── ACK (ack=y+1) ───────────────> | 第三次握手： | Client 說：我確認你的序號 y | ESTABLISHED 狀態 | | | ═══════ 連線建立，開始傳輸資料 ══════ |
SYN Flood 攻擊原理

SYN Flood 是一種 DDoS 攻擊，攻擊者發送大量偽造來源 IP 的 SYN 包，伺服器回應 SYN+ACK 後等待最後的 ACK（進入半連接狀態），但因為來源 IP 是偽造的，ACK 永遠不會到達。伺服器的半連接佇列被耗盡，無法響應正常用戶的連接請求。防禦方法：SYN Cookie 技術、減小 SYN 等待超時時間。

TCP 四次揮手（斷開連接）

TCP 連接的斷開需要四次揮手，因為 TCP 連接是全雙工的，兩個方向的連接需要分別關閉。

四次揮手過程： 客戶端 (Client) 伺服器 (Server) | | | ──── FIN ─────────────────────────> | 第一次揮手： | Client 說：我沒資料要發了，想斷連 | FIN_WAIT_1 | | | <─── ACK ───────────────────────── | 第二次揮手： | Server 說：收到，但我可能還有資料要發 | CLOSE_WAIT | | | [Server 繼續發送剩餘資料...] | | | | <─── FIN ───────────────────────── | 第三次揮手： | Server 說：我也沒資料了，也要斷連了 | LAST_ACK | | | ──── ACK ─────────────────────────> | 第四次揮手： | Client 說：收到，等待 2MSL 後關閉 | CLOSED | | | ══════════ 連線斷開 ═══════════════ |
小測驗
為什麼 TCP 建立連接需要三次握手，而不是兩次？
A 因為網路標準規定必須三次
B 三次比兩次更快
C 確保雙方都驗證了對方的發送和接收能力，兩次無法確認客戶端的接收能力
D 為了讓 SYN Flood 攻擊更容易防禦
正確！兩次握手只能確認：(1) 客戶端能發送，伺服器能接收；(2) 伺服器能發送。但無法確認客戶端能接收伺服器的資料。第三次握手（客戶端發 ACK）正是為了讓伺服器確認客戶端有能力接收資料。這也解釋了為什麼三次是最少的可靠次數。

ARP 地址解析協議

ARP（Address Resolution Protocol）負責將 IP 地址解析為 MAC 地址。在同一個局域網內，設備通信需要知道對方的 MAC 地址。ARP 的設計有一個重大的安全缺陷：它是無狀態的，任何人都可以發送假的 ARP 回應，這就是 ARP 欺騙攻擊的基礎。

ARP 工作流程

ARP 解析過程（PC A 想發資料給 PC B）： PC A (192.168.1.10) PC B (192.168.1.20) PC C (192.168.1.30) | | | | ARP Request（廣播） | | | ─── 誰是 192.168.1.20？ ──> | ────────────────────────> | | 我是 AA:BB:CC:11:22:33 | | | | ARP Reply（單播） | | <── 我是 192.168.1.20 | | | MAC: DD:EE:FF:44:55:66 | | | | | PC A 更新 ARP 快取： 192.168.1.20 → DD:EE:FF:44:55:66 PC C 收到廣播，但不會回應（因為問的不是他）
ARP 欺騙攻擊（中間人攻擊）

ARP 協議沒有身份驗證機制！攻擊者可以發送偽造的 ARP 回應，告訴 PC A「我就是閘道（192.168.1.1）的 MAC 是 XX:XX:XX:XX:XX:XX（攻擊者的 MAC）」，同時告訴閘道「PC A（192.168.1.10）的 MAC 是 XX:XX:XX:XX:XX:XX」，這樣所有流量都會先經過攻擊者，攻擊者就能嗅探或篡改流量（中間人攻擊）。防禦方式：動態 ARP 檢測（DAI）、靜態 ARP 條目、使用 HTTPS。

查看和清除 ARP 快取

Windows CMD / PowerShell 
# 查看 ARP 快取表
arp -a

# 清除 ARP 快取（需要管理員權限）
arp -d *

# 添加靜態 ARP 條目（防 ARP 欺騙）
arp -s 192.168.1.1 00-11-22-33-44-55
Linux Terminal 
# 查看 ARP 快取表
arp -n

# 使用 ip 命令（現代 Linux）
ip neigh show

# 清除 ARP 快取
ip neigh flush all

# 使用 arping 工具發送 ARP 請求（需安裝）
arping -I eth0 192.168.1.1

域名解析過程（DNS）

DNS（Domain Name System）是互聯網的「電話簿」，負責將人類可讀的域名（如 google.com）解析為電腦可處理的 IP 地址。DNS 也是滲透測試中重要的信息收集目標。

DNS 層次結構

DNS 域名層次結構： 根域 (.) ├── 頂級域 (TLD)： .com .org .tw .cn │ ├── 二級域： google.com microsoft.com │ │ └── 主機名： www.google.com mail.google.com │ └── ... └── ...

完整的 DNS 查詢過程

用戶訪問 www.example.com 的完整 DNS 解析流程： 瀏覽器 → 本機 hosts 文件 → 本機 DNS 快取 ↓ 都沒有命中 本地 DNS 伺服器（ISP 提供，如 8.8.8.8） ↓ 快取中沒有 根 DNS 伺服器（全球 13 個根服務器集群） ↓ 回應：.com 的 TLD 伺服器地址 .com TLD 伺服器 ↓ 回應：example.com 的授權伺服器地址 example.com 授權 DNS 伺服器 ↓ 回應：www.example.com = 93.184.216.34 本地 DNS 伺服器快取結果 → 回傳給瀏覽器
滲透測試：DNS 信息收集

DNS 是信息收集的重要來源。透過 DNS 查詢可以：(1) 發現子域名（子域名爆破）；(2) 查詢 MX 記錄找郵件伺服器；(3) 查詢 TXT 記錄發現 SPF/DKIM 配置；(4) DNS 區域傳送漏洞（AXFR）可能洩露所有 DNS 記錄。常用工具：nslookup、dig、dnsenum、subfinder。

常用 DNS 查詢命令

nslookup / dig 命令 
# 基本 A 記錄查詢（域名 → IP）
nslookup www.example.com
dig www.example.com A

# 查詢 MX 記錄（郵件伺服器）
dig example.com MX

# 查詢所有記錄類型
dig example.com ANY

# 測試 DNS 區域傳送漏洞
dig axfr @dns.example.com example.com

# 反向解析（IP → 域名）
dig -x 93.184.216.34

IP 地址組成與分類

IP 地址（IPv4）是一個 32 位元的二進制數，通常用四個十進制數表示（如 192.168.1.1）。理解 IP 地址的結構是進行子網劃分和網路規劃的基礎。

IP 地址的組成

IP 地址 = 網路部分 + 主機部分 例：192.168.1.100 / 24 二進制：11000000.10101000.00000001.01100100 ←─────── 網路部分 ────────→←主機→ 192 .168 .1 .100 子網遮罩：/24 = 255.255.255.0 二進制： 11111111.11111111.11111111.00000000 其中 1 的位數對應網路部分，0 的位數對應主機部分

IPv4 地址分類（有類地址）

類別 地址範圍 默認子網遮罩 用途 私有地址範圍
A 類 1.0.0.0 — 126.255.255.255 /8 (255.0.0.0) 大型網路 10.0.0.0/8
B 類 128.0.0.0 — 191.255.255.255 /16 (255.255.0.0) 中型網路 172.16.0.0/12
C 類 192.0.0.0 — 223.255.255.255 /24 (255.255.255.0) 小型網路 192.168.0.0/16
D 類 224.0.0.0 — 239.255.255.255 — 組播（Multicast） —
E 類 240.0.0.0 — 255.255.255.255 — 實驗保留 —
私有地址與 NAT

私有 IP 地址（10.x.x.x、172.16-31.x.x、192.168.x.x）不能直接在互聯網上路由，只能在私有網路內使用。企業和家庭網路使用私有地址，透過 NAT（網路地址轉換）映射到公網 IP 地址進行互聯網通信。這也是為什麼你家裡很多設備可以共用一個公網 IP 的原因。

小測驗
下列哪個 IP 地址是私有 IP 地址？

B 114.114.114.114
C 172.20.10.1

子網劃分詳解

子網劃分是將一個大的 IP 地址空間分割成多個小的子網的技術。在企業網路設計中，子網劃分用於隔離不同部門的流量，在滲透測試中，理解子網劃分有助於分析目標網路的結構。

CIDR 表示法

CIDR（無類別域間路由）使用前綴長度表示子網遮罩：192.168.1.0/24，其中 /24 表示前 24 位是網路部分。

CIDR 前綴 子網遮罩 可用主機數 常見用途
/8 255.0.0.0 16,777,214 大型 ISP、A 類地址
/16 255.255.0.0 65,534 大型企業
/24 255.255.255.0 254 最常見，辦公室/部門
/25 255.255.255.128 126 小型部門
/30 255.255.255.252 2 路由器間的點對點連接
/32 255.255.255.255 1 單台主機路由

子網計算練習

給定網段 192.168.10.0/26，計算以下信息：

已知：192.168.10.0/26 /26 → 子網遮罩：255.255.255.192（二進制末段：11000000） 網路地址（主機位全0）： 192.168.10.0 廣播地址（主機位全1）： 192.168.10.63 可用主機範圍： 192.168.10.1 ~ 192.168.10.62 可用主機數： 2^(32-26) - 2 = 2^6 - 2 = 62 台 計算公式： 可用主機數 = 2^(主機位數) - 2 子網數 = 2^(借用的位數)
滲透測試實用技巧

在滲透測試中，當你發現目標的一個 IP（如 192.168.10.5/24），你應該立刻知道整個子網是 192.168.10.0/24，可以對這個段進行主機發現掃描（nmap -sn 192.168.10.0/24），找到同一子網下的其他主機，這是內網橫向滲透的第一步。

很多企業的網路設備仍然開啟了 Telnet 管理（明文傳輸！）或使用默認帳號密碼（cisco/cisco）。這是滲透測試中常見的發現。在信息收集階段，如果發現 23（Telnet）或 80（HTTP 管理介面）端口開放，應立即進行驗證。

Cisco 設備配置模式

Cisco 路由器和交換機的 CLI 有多個配置模式，了解這些模式是進行設備配置的前提。

Cisco IOS 配置模式層次： Router> ← 用戶 EXEC 模式（只讀，有限命令） Router# enable ← 輸入 enable 進入特權模式 Router# ← 特權 EXEC 模式（可查看所有信息） Router# configure terminal Router(config)# ← 全局配置模式（修改全局設定） Router(config)# interface GigabitEthernet0/0 Router(config-if)# ← 介面配置模式（配置具體介面） Router(config)# router ospf 1 Router(config-router)# ← 路由協議配置模式
Cisco IOS — 基礎配置命令
! 設定主機名稱
Router(config)# hostname R1

! 設定特權模式密碼（加密）
R1(config)# enable secret Str0ngP@ss

! 設定控制台密碼
R1(config)# line console 0
R1(config-line)# password Console123
R1(config-line)# login

! 開啟 SSH（代替 Telnet）
R1(config)# ip domain-name example.com
R1(config)# crypto key generate rsa modulus 2048
R1(config)# ip ssh version 2
R1(config)# line vty 0 4
R1(config-line)# transport input ssh
R1(config-line)# login local

! 保存配置
R1# write memory  或  R1# copy running-config startup-config

靜態路由實戰配置

靜態路由是手動配置的路由條目，告訴路由器「要到達某個網段，下一跳是哪裡」。雖然不像動態路由那樣自動學習，但靜態路由更穩定、更安全，適合小型網路或特定流量的控制。

靜態路由配置場景： [PC A: 192.168.1.10] | [R1] G0/0: 192.168.1.1 G0/1: 10.0.0.1 | [R2] G0/0: 10.0.0.2 G0/1: 192.168.2.1 | [PC B: 192.168.2.10] 問題：PC A 怎麼訪問 PC B？ → R1 需要知道：要到 192.168.2.0/24，下一跳是 10.0.0.2（R2） → R2 需要知道：要到 192.168.1.0/24，下一跳是 10.0.0.1（R1）
Cisco IOS — 靜態路由配置
! R1 的配置
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip address 192.168.1.1 255.255.255.0
R1(config-if)# no shutdown

R1(config)# interface GigabitEthernet0/1
R1(config-if)# ip address 10.0.0.1 255.255.255.252
R1(config-if)# no shutdown

! 添加靜態路由：到 192.168.2.0/24，下一跳 10.0.0.2
R1(config)# ip route 192.168.2.0 255.255.255.0 10.0.0.2

! 查看路由表
R1# show ip route

! 測試連通性
R1# ping 192.168.2.10

動態路由協議概述與分類

動態路由協議讓路由器自動學習網路拓撲，無需手動配置每條路由。當網路拓撲發生變化時（如某條鏈路斷了），路由器能自動收斂找到新的路徑。

分類協議算法適用範圍
距離向量RIP v1/v2Bellman-Ford小型網路（跳數≤15）
鏈路狀態OSPFDijkstra（SPF）企業中大型網路
鏈路狀態IS-ISDijkstraISP 骨幹網
路徑向量BGP路徑向量互聯網 AS 間路由
增強距離向量EIGRPDUALCisco 私有協議

RIP 工作原理與配置

RIP（Routing Information Protocol）是最古老的動態路由協議之一，以「跳數（Hop Count）」作為度量值，最大跳數為 15（16 跳視為不可達）。

Cisco IOS — RIP v2 配置
! 啟用 RIP，版本 2
R1(config)# router rip
R1(config-router)# version 2
R1(config-router)# no auto-summary

! 宣告直連網路
R1(config-router)# network 192.168.1.0
R1(config-router)# network 10.0.0.0

! 查看 RIP 路由
R1# show ip route rip
R1# debug ip rip
RIP 的安全問題

RIP 默認沒有認證機制，攻擊者可以注入偽造的路由信息（路由欺騙），導致流量被重定向到攻擊者控制的設備。RIP v2 支持 MD5 認證，但現代網路已基本被 OSPF 或 EIGRP 取代。

OSPF 原理與特性

OSPF（Open Shortest Path First）是企業網路中最廣泛使用的動態路由協議，使用 Dijkstra 最短路徑算法計算最優路徑。

Router ID
每台 OSPF 路由器的唯一標識，通常使用 Loopback 介面的 IP 地址，格式為 x.x.x.x。
LSDB
鏈路狀態數據庫，保存整個 OSPF 區域的拓撲信息，同一區域的所有路由器 LSDB 相同。
DR/BDR
在廣播網路（如乙太網）上選出指定路由器（DR）和備份指定路由器（BDR），減少 LSA 泛洪。
Area
OSPF 可劃分為多個區域（Area），Area 0 是骨幹區域，所有其他區域必須連接到 Area 0。

OSPF 實戰配置

Cisco IOS — OSPF 基礎配置
! 啟用 OSPF，進程號 1
R1(config)# router ospf 1

! 設定 Router ID
R1(config-router)# router-id 1.1.1.1

! 宣告網路（在 Area 0）
R1(config-router)# network 192.168.1.0 0.0.0.255 area 0
R1(config-router)# network 10.0.0.0 0.0.0.3 area 0

! 查看 OSPF 鄰居關係
R1# show ip ospf neighbor

! 查看 OSPF 路由
R1# show ip route ospf

NAT 概念與靜態 NAT

NAT（Network Address Translation）解決了 IPv4 地址不足的問題，允許多台私有 IP 設備共用少量公網 IP 地址進行互聯網通信。

靜態 NAT
一對一映射：一個私有 IP ↔ 一個公網 IP。通常用於需要從外網訪問的伺服器（如 Web 伺服器）。
動態 NAT
多對多映射：從公網 IP 池中動態分配。連接斷開後公網 IP 歸還池中。
PAT（NAT Overload）
多對一映射：多台私有設備共用一個公網 IP，用不同的端口號區分。家用路由器最常用。
資安影響
NAT 提供了一定程度的隱藏，但不是防火牆！不要依賴 NAT 作為安全措施，仍需配置 ACL 和防火牆規則。

動態 NAT 與 PAT 配置

Cisco IOS — PAT 配置（最常用）
! 定義內部網路 ACL
R1(config)# access-list 1 permit 192.168.1.0 0.0.0.255

! 配置 PAT（使用外部介面 IP）
R1(config)# ip nat inside source list 1 interface GigabitEthernet0/1 overload

! 標記內部和外部介面
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip nat inside

R1(config)# interface GigabitEthernet0/1
R1(config-if)# ip nat outside

! 查看 NAT 轉換表
R1# show ip nat translations
R1# show ip nat statistics

NAT 排錯與 TCP 負載均衡

NAT 配置常見問題排查步驟：確認介面是否正確標記 inside/outside、確認 ACL 是否匹配正確的源地址、確認路由表是否有到外部的默認路由。

Cisco IOS — NAT 排錯命令
! 查看 NAT 轉換（是否有條目生成）
R1# show ip nat translations

! 查看 NAT 統計（看 hits 和 misses）
R1# show ip nat statistics

! 開啟 NAT 調試（謹慎使用）
R1# debug ip nat

! 清除 NAT 轉換表
R1# clear ip nat translation *

二層交換基礎與 VLAN 概述

VLAN（Virtual Local Area Network）虛擬局域網，是在一台物理交換機上劃分出多個邏輯上獨立的廣播域。不同 VLAN 的設備默認無法直接通信，需要透過路由器或三層交換機進行跨 VLAN 路由。

VLAN 的安全價值

VLAN 是企業網路安全設計的重要工具。例如：將財務部門和研發部門放在不同 VLAN，即使有一台設備被攻陷，攻擊者也無法直接橫向移動到其他部門。這正是「網路分段（Network Segmentation）」安全策略的基礎。

VLAN 配置與跨 VLAN 路由

Cisco IOS — VLAN 基礎配置
! 創建 VLAN
SW1(config)# vlan 10
SW1(config-vlan)# name Finance
SW1(config)# vlan 20
SW1(config-vlan)# name Engineering

! 將端口分配到 VLAN（Access 端口）
SW1(config)# interface GigabitEthernet0/1
SW1(config-if)# switchport mode access
SW1(config-if)# switchport access vlan 10

! 配置 Trunk 端口（連接路由器或其他交換機）
SW1(config)# interface GigabitEthernet0/24
SW1(config-if)# switchport mode trunk
SW1(config-if)# switchport trunk allowed vlan 10,20

! 查看 VLAN 信息
SW1# show vlan brief

訪問控制列表（ACL）配置

ACL（Access Control List）是路由器上的流量過濾工具，根據定義的規則允許或拒絕特定的網路流量。ACL 是實現基本網路安全策略的重要手段。

Cisco IOS — ACL 配置
! 標準 ACL（只基於源 IP 過濾）
R1(config)# access-list 10 permit 192.168.1.0 0.0.0.255
R1(config)# access-list 10 deny any

! 擴展 ACL（基於源/目標 IP、協議、端口）
R1(config)# ip access-list extended BLOCK_TELNET
R1(config-ext-nacl)# deny tcp any any eq 23
R1(config-ext-nacl)# deny tcp any any eq 21
R1(config-ext-nacl)# permit ip any any

! 將 ACL 應用到介面
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip access-group BLOCK_TELNET in

! 查看 ACL
R1# show access-lists
