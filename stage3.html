資安行業概況與就業方向

行業入門 15 分鐘

資安行業現況

根據 2024 年統計，全球網路安全人才缺口已超過 350 萬人，且持續增長。在中國，隨著《網路安全法》《數據安全法》《個人信息保護法》陸續實施，以及「等保 2.0」全面落地，企業對資安合規的需求呈爆發式增長。資安不再是可有可無的選項，而是企業的法律義務。

資安工程師的主要方向

01
滲透測試工程師 — 模擬攻擊者對系統進行授權的安全測試，找出漏洞並出具報告。需要最廣泛的技術知識，對應本課程的核心 Stage 4。
02
安全運營工程師（SOC） — 負責監控安全事件、分析告警、應急響應。使用 SIEM 工具（ELK/Splunk）進行日誌分析，對應 Stage 9。
03
等保/合規工程師 — 協助企業完成等級保護測評、ISO27001 認證等合規工作，需求量大，對應 Stage 10-14。
04
安全開發工程師 — 開發安全工具、WAF、掃描器等，需要紮實的程式能力（Python/Go），對應 Stage 15。
05
移動安全工程師 — 對 Android/iOS 應用進行安全分析，屬於稀缺方向，對應 Stage 5-6。
06
雲安全工程師 — 新興方向，負責雲原生環境（K8S、容器）的安全評估，對應 Stage 8。
學習路線建議

正確策略：先夯實基礎（Stage 1-3）→ 深入 Web 滲透核心（Stage 4，最重要，佔 2 個月）→ 根據興趣選擇專項方向（Stage 5-15）。不要跳過基礎直接學工具，知其然不知其所以然，遇到新場景就會束手無策。

網路安全法律法規

在開始學習任何攻擊技術之前，必須先了解相關法律法規。未經授權的滲透測試是違法行為，即使初衷是「幫助對方找漏洞」，沒有書面授權依然可能面臨刑事責任。

法律紅線

《刑法》第 285 條（非法侵入計算機信息系統罪）和第 286 條（破壞計算機信息系統罪）明確規定，未經授權侵入他人計算機系統，最高可判處 7 年有期徒刑。永遠只在授權範圍內進行測試，永遠先取得書面授權協議（SOW）。

三大核心法律

《網路安全法》
2017年6月施行
確立網路安全基本制度框架，包括等級保護、關鍵信息基礎設施保護、個人信息保護等。規定網絡運營者的安全義務。
《數據安全法》
2021年9月施行
建立數據分類分級保護制度。違規最高罰款 200 萬元，情節嚴重可罰 1000 萬元並吊銷執照。
《個人信息保護法》
2021年11月施行
中國版「GDPR」，規範個人信息的收集、處理、存儲、使用。違規最高罰款為上一年度營業額的 5%。

合法滲透的四個前提

01
書面授權協議（SOW） — 明確測試範圍、時間、方法，由甲方授權負責人簽名蓋章。沒有這份文件，任何測試都是非法的。
02
範圍限定 — 測試只能在授權的 IP 範圍、域名、系統內進行，絕對不能橫向擴展到授權範圍之外。
03
不破壞原則 — 即使是授權的測試，也不應造成業務中斷、數據損壞或洩露。生產環境需格外謹慎。
04
負責任的漏洞披露 — 發現的漏洞只能報告給委託方，不得公開或利用。可透過 CNVD/CVE 進行規範披露。
SRC 平台：合法練習真實漏洞的最佳途徑

各大廠商（騰訊、阿里、百度、字節等）均設有 SRC（安全應急響應中心），允許安全研究者在規定範圍內提交漏洞，給予現金或積分獎勵。這是初學者合法練習真實漏洞挖掘並建立個人資安履歷的最佳方式。

小測驗
你發現了某電商平台存在 SQL 注入漏洞。以下哪個做法是合法且負責任的？
A利用漏洞導出部分用戶數據，截圖後向網站索要賞金
B在社群媒體公開發布漏洞細節，讓更多人知道
C不實際利用漏洞，透過 SRC 平台或郵件向企業安全團隊負責任地披露
D保留漏洞，等以後有需要時再使用
正確！這是「負責任的漏洞披露（Responsible Disclosure）」。正確做法：不實際利用漏洞竊取數據（犯罪），通過正規渠道通知企業，給予合理修復時間（通常 90 天）。選項 A 是敲詐勒索，B 可能導致漏洞被惡意利用，D 留作後用同樣違法。

其他重要漏洞類型

XSS（跨站腳本攻擊）
攻擊者在網頁中注入惡意 JavaScript，在受害者瀏覽器中執行，可竊取 Cookie、釣魚等。分反射型、存儲型、DOM 型。
CSRF（跨站請求偽造）
利用受害者已登入的身份，在不知情情況下發送惡意請求。常用於轉帳、修改密碼等操作。
文件上傳漏洞
伺服器未正確驗證上傳文件類型，攻擊者上傳 PHP Webshell 獲取伺服器控制權。是最直接的 GetShell 方法之一。
反序列化漏洞
應用程式在還原序列化數據時未驗證來源，攻擊者構造惡意數據實現 RCE（遠端代碼執行）。Log4j 屬此類。

攻防實驗環境搭建

在學習任何攻防技術之前，需要先搭建安全的實驗環境。切勿在真實公網環境或未獲授權的系統上練習任何攻擊技術。

CTF、護網行動與漏洞庫

CTF（Capture The Flag）

CTF 是一種資安競賽形式，參賽者透過解決安全挑戰獲取「Flag」（特定格式的字符串）。CTF 是學習和驗證技術能力的重要途徑，也是建立個人技術名聲的方式。

Jeopardy（解題模式）
最常見的 CTF 形式。各題目獨立，解出即得分。題型：Web、Pwn（二進制）、Reverse（逆向）、Crypto（密碼學）、Misc（雜項）。
Attack-Defense（攻防模式）
每個隊伍同時作為攻擊方和防守方，在保護自身服務的同時攻擊其他隊伍，更接近真實攻防場景。
國內練習平台
BUUCTF、攻防世界（xctf.org.cn）、Bugku、NSSCTF、CTFHub——推薦從 Web 類題目入手。
國際平台
CTFtime.org（賽事日曆）、PicoCTF（初學者友好）、pwn.college（系統安全深度學習）。

護網行動（HW / 攻防演練）

護網行動是國家組織的大規模攻防演練，每年定期舉行。企業（藍隊/防守方）需保護系統，授權的攻擊隊（紅隊）進行真實攻擊測試。護網期間資安人才需求激增。

護網崗位

藍隊（防守）：安全監控、告警分析、應急響應、溯源反制。需熟悉安全設備（WAF/IDS/IPS）操作和日誌分析。
紅隊（攻擊）：需通過資質審查，使用真實攻擊技術對授權目標進行測試，需豐富的滲透經驗。

為什麼資安工程師要學 HTML？

很多初學者疑惑：「我是要學滲透測試的，為什麼要學前端開發？」答案很簡單：你無法攻擊你不理解的東西。

01
XSS 攻擊的基礎 — XSS（跨站腳本攻擊）本質上是在 HTML 頁面中注入惡意 JavaScript。不懂 HTML 結構和 JS 執行機制，就無法理解為什麼 <script>alert(1)</script> 能彈窗，更別說繞過 WAF 過濾了。
02
表單與參數分析 — 所有 Web 漏洞（SQL 注入、CSRF、邏輯漏洞等）都涉及 HTTP 請求參數。理解 HTML 表單如何構造請求，能快速識別所有輸入點。
03
閱讀頁面源代碼 — 信息收集階段，查看網頁源代碼（View Source / Ctrl+U）可以發現隱藏參數、注釋中的敏感信息、使用的 JS 框架和版本等。
04
釣魚頁面與社工攻擊 — 製作仿冒真實網站的釣魚頁面需要基礎的 HTML/CSS 能力。這也是為什麼理解前端能幫助你識別和防禦釣魚攻擊。
學習目標

本章節的目標不是讓你成為前端工程師，而是達到「能讀懂 HTML/JS 代碼，理解基本結構，識別潛在安全問題」的水準。實際滲透時你不需要自己寫頁面，但需要能分析和修改已有代碼。

HTML 標籤、元素與表單

HTML（HyperText Markup Language）使用標籤（Tag）描述網頁結構。每個標籤由 <開始標籤>、內容、</結束標籤> 組成。

HTML — 基礎結構
<!-- HTML 基礎結構 -->
<!DOCTYPE html>
<html lang="zh-TW">
<head>
  <meta charset="UTF-8">
  <title>頁面標題</title>
</head>
<body>
  <h1>標題</h1>
  <p>段落文字</p>
  <a href="https://example.com">超連結</a>
  <img src="image.jpg" alt="圖片描述">
  <div id="container" class="box">區塊容器</div>
</body>
</html>

HTML 表單（資安核心關注點）

表單是 Web 應用接收用戶輸入的主要方式，也是絕大多數 Web 漏洞的入口點。每個 <input> 字段都是潛在的注入點。

HTML 表單 — 資安分析視角
<form action="/login" method="POST">

  <!-- text：明文輸入，直接注入點 -->
  <input type="text"     name="username">

  <!-- password：介面不顯示明文，但 Burp Suite 抓包完全可見 -->
  <input type="password" name="password">

  <!-- hidden：用戶看不到，但用 Burp 攔截後可任意修改！漏洞高發區 -->
  <input type="hidden"   name="role" value="user">

  <!-- file：文件上傳漏洞的入口 -->
  <input type="file"     name="avatar">

  <!-- select：下拉選單，前端限制可被 Burp 繞過 -->
  <select name="dept">
    <option value="1">技術部</option>
    <option value="2">人事部</option>
  </select>

  <button type="submit">登入</button>
</form>
資安視角：hidden 字段的陷阱

type="hidden" 的字段對普通用戶不可見，但使用 Burp Suite 攔截請求後可以任意修改。如果後端直接信任這個字段（如把 role=admin 當作權限依據），就會產生越權漏洞。永遠不要在客戶端做安全決策，所有驗證必須在服務端進行。

小測驗
一個登入表單使用了 <input type="hidden" name="is_admin" value="0">，後端直接讀取這個值判斷管理員權限。這個設計存在什麼安全問題？
A沒有問題，hidden 字段用戶看不到
B用戶可以用 Burp Suite 攔截並修改這個字段為 1，獲得管理員權限
C問題在於應該使用 GET 方法而非 POST
D問題在於沒有使用 HTTPS
正確！這是「參數篡改」漏洞。雖然 hidden 字段在瀏覽器介面不可見，但使用 Burp Suite 攔截 HTTP 請求後，可以輕易修改任何請求參數。攻擊者只需將 is_admin=0 改為 is_admin=1 提交，就能獲取管理員權限。正確做法：權限驗證必須在服務端基於 Session/Token 進行，不能信任任何來自客戶端的輸入。

JavaScript 基礎語法

JavaScript 是 Web 的「行為層」，讓頁面具有動態交互能力。從資安角度，理解 JS 是分析 XSS 攻擊、繞過前端校驗、分析混淆代碼的基礎。

JavaScript — 基礎語法
// ─── 變數宣告 ───
var   x  = 10;      // 函數作用域（舊式，避免使用）
let   y  = 20;      // 塊作用域（推薦）
const PI = 3.14;   // 常數，不可重新賦值

// ─── 資料類型 ───
let str  = "Hello";            // 字串
let num  = 42;                  // 數字
let bool = true;               // 布林
let arr  = [1, 2, 3];           // 陣列
let obj  = { name: "Alice", age: 25 }; // 物件

// ─── 函數 ───
function add(a, b) { return a + b; }
const square = n => n * n;     // 箭頭函數

// ─── DOM 操作（XSS 攻擊的核心）───
document.getElementById('id').innerHTML = '<b>修改DOM</b>';
document.cookie;                        // 讀取全部 Cookie → XSS 竊取目標
document.location = 'http://evil.com'; // 重定向 → 釣魚攻擊

XSS 攻擊鏈

完整的 XSS Cookie 竊取流程：
1. 攻擊者在存儲型 XSS 漏洞處注入惡意 JS → 2. 受害者訪問含惡意 JS 的頁面 → 3. 受害者瀏覽器執行 JS，讀取 document.cookie → 4. Cookie 被發送到攻擊者的服務器 → 5. 攻擊者用竊取的 Session Cookie 登入受害者帳號（Session 劫持）

文件包含漏洞（LFI / RFI）

PHP 操作 MySQL 資料庫

PHP 操作資料庫有三種方式：mysql_* 函數（已廢棄，危險）、mysqli 擴展、PDO（推薦）。理解這三種方式有助於在代碼審計時識別 SQL 注入漏洞。

PHP — PDO 安全連線與查詢
<?php
// ─── PDO 連線（推薦的安全方式）───
try {
    $pdo = new PDO(
        'mysql:host=localhost;dbname=mydb;charset=utf8',
        'dbuser',
        'dbpass'
    );
    $pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);
} catch (PDOException $e) {
    // 記錄錯誤，但不顯示給用戶
    error_log($e->getMessage());
    die("資料庫連線失敗");
}

// ─── 預處理語句（防 SQL 注入的正確做法）───
$stmt = $pdo->prepare("SELECT * FROM users WHERE id = :id");
$stmt->bindParam(':id', $_GET['id'], PDO::PARAM_INT);
$stmt->execute();
$user = $stmt->fetch(PDO::FETCH_ASSOC);

// ─── 不安全的字串拼接（漏洞示例！）───
$id    = $_GET['id'];   // 用戶輸入：1 OR 1=1
$query = "SELECT * FROM users WHERE id = $id";
// 最終 SQL：SELECT * FROM users WHERE id = 1 OR 1=1 → 洩露所有用戶！

// ─── INSERT 插入（同樣使用預處理）───
$stmt = $pdo->prepare("INSERT INTO logs (user, action) VALUES (?, ?)");
$stmt->execute([$_SESSION['user'], 'login']);
?>
預處理語句為什麼能防止 SQL 注入

預處理語句（Prepared Statement）將 SQL 結構和數據分開傳輸。SQL 模板先發送給資料庫編譯，再單獨傳入參數值。資料庫已經知道哪裡是「代碼」、哪裡是「數據」，因此無論參數包含什麼特殊字符，都只會被當作普通數據處理，不可能改變 SQL 的邏輯結構。這是防 SQL 注入最根本的方法。

SQL 注入的原理

SQL 注入發生在 Web 應用將用戶輸入未經過濾地拼接進 SQL 語句時。攻擊者通過構造特殊的輸入，改變 SQL 語句的邏輯，實現越權查詢、繞過認證、提取數據甚至執行系統命令。

SQL 注入原理示例
-- 正常查詢（用戶輸入 id=1）
SELECT * FROM users WHERE id = 1;

-- 注入攻擊（用戶輸入 id=1 OR 1=1）
SELECT * FROM users WHERE id = 1 OR 1=1;
-- 結果：返回所有用戶記錄！（因為 1=1 恆為真）

-- 繞過登入驗證（用戶名輸入 admin'--）
SELECT * FROM users WHERE username='admin'--' AND password='xxx';
-- -- 是 MySQL 注釋符，後面的 AND password=... 被注釋掉
-- 結果：不驗證密碼直接登入 admin！

-- UNION 注入（提取其他表的數據）
-- 用戶輸入 id=1 UNION SELECT username,password FROM users--
SELECT name, desc FROM products WHERE id=1 
UNION SELECT username, password FROM users--;
-- 結果：商品查詢結果中混入了用戶表的帳號密碼！

-- 通過 information_schema 獲取所有表名
SELECT table_name FROM information_schema.tables
WHERE table_schema=database() LIMIT 1,1;

類型特點說明
聯合查詢注入有回顯使用 UNION SELECT 在頁面顯示位提取數據，最常見、最直接
錯誤注入有錯誤回顯利用資料庫報錯信息中包含的數據，如 extractvalue()、updatexml()

小測驗
一個登入頁面在用戶名欄位輸入 admin' OR '1'='1，密碼隨便填，結果成功登入了。這說明這個應用存在什麼漏洞？
AXSS 漏洞，因為輸入了特殊字符
B暴力破解成功，說明密碼太簡單
CSQL 注入漏洞，用戶名中的 SQL 語句改變了查詢邏輯，繞過了密碼驗證
DSession 固定攻擊，Session 被竊取
正確！這是 SQL 注入繞過認證的經典場景。原本的 SQL 查詢可能是 SELECT * FROM users WHERE username='$u' AND password='$p'，注入 admin' OR '1'='1 後變成 WHERE username='admin' OR '1'='1' AND password='...'。由於 OR 優先級的關係，條件變成 admin' OR TRUE，查詢返回 admin 用戶，跳過了密碼驗證。

MySQL 注入常用查詢組合
-- 第1步：獲取當前資料庫名
SELECT database();

-- 第2步：獲取所有資料表名（group_concat 一次列出）
SELECT group_concat(table_name) FROM information_schema.tables
WHERE table_schema=database();

-- 第3步：獲取指定表的所有欄位名
SELECT group_concat(column_name) FROM information_schema.columns
WHERE table_name='users';

-- 第4步：提取數據
SELECT group_concat(username, '~', password) FROM users;

-- 布林盲注示例（逐字元猜測 database() 的第1個字）
SELECT * FROM products WHERE id=1 
  AND substr(database(),1,1)='m';   -- 返回正常則第1個字母是 m

-- 時間盲注示例
SELECT * FROM products WHERE id=1 
  AND if(substr(database(),1,1)='m', sleep(3), 0);
-- 若延遲 3 秒則第1個字母是 m
