報告架構與讀者定位

滲透測試報告是整個測試項目最重要的交付物。即使技術發現再精彩，若報告寫得晦澀難懂、結構混亂，客戶將無法採取行動修復問題。優秀的報告需要同時服務兩類讀者：對技術細節感興趣的技術人員，以及只關心業務風險的管理層。

標準報告結構

第 1 節
封面頁 & 文件控制
項目名稱、委託方、執行機構、測試日期、報告版本號、保密等級（CONFIDENTIAL）、版本歷史（作者、審閱者、日期）。建立文件責任鏈。
第 2 節
執行摘要（Executive Summary）
1–2 頁，面向 CEO/CISO/Board 等管理層。概括整體安全狀態、最嚴重的 3–5 個問題、業務風險影響，以及核心改進建議。不含技術細節，用業務語言表達風險。
第 3 節
測試範疇與方法論
白盒/灰盒/黑盒說明、測試時間窗、授權範圍（IP 範圍、域名、應用列表）、明確排除範圍、使用的方法論框架（PTES/OWASP/NIST SP 800-115）、測試人員資訊。
第 4 節
發現彙總（Findings Summary）
所有漏洞的列表，按嚴重程度排序（Critical → High → Medium → Low → Informational）。通常以表格或圖表形式呈現漏洞分佈，方便管理層快速掌握全貌。
第 5 節
詳細技術發現
每個漏洞的完整技術記錄：標題、嚴重程度、CVSS 評分、受影響系統、漏洞描述、利用步驟（PoC）、截圖/日誌證據、修復建議（短期/長期）、參考資料（CVE/CWE/OWASP）。
附錄
附錄（Appendices）
完整的掃描輸出、工具清單與版本、測試日誌（活動時間軸）、原始 Exploit 代碼（選擇性提供）、術語表。技術細節不影響主報告可讀性但方便技術複查。
讀者分層原則

每份報告都有兩類核心讀者，必須同時服務好：管理層（CEO/CFO/CISO）——只讀執行摘要，需要用業務語言說明「如果不修復，公司會面臨什麼損失」；技術團隊（開發者/系統管理員）——閱讀詳細技術發現，需要準確的重現步驟和可執行的修復指引。永遠先問：「這一段寫給誰看？他們最需要知道什麼？」

執行摘要（Executive Summary）寫法

執行摘要是報告中最具戰略價值的部分，也是最難寫好的部分。它必須讓一個沒有技術背景的高管，在 5 分鐘內理解公司面臨的安全風險，並願意授權修復預算。

執行摘要黃金結構

01
測試概述（1 段） — 說明測試目的、範圍和時間。例：「本次針對 XYZ 公司電子商務平台的黑盒滲透測試，於 2024 年 11 月 18–22 日進行，涵蓋 3 個對外 Web 應用程式及相關 API。」
02
整體安全評估（1–2 段） — 用 1–5 級量表或顏色（紅/橙/黃/綠）給出整體評級，配合簡短說明。例：「整體安全狀態評級為高風險（紅色）。測試人員在未授權的情況下，成功訪問了包含 150 萬筆客戶個人資料的資料庫，並取得了生產伺服器的管理員權限。」
03
關鍵發現（3–5 個，按業務影響排序） — 不用漏洞術語，改用業務影響語言。不要寫「發現 SQL Injection」，要寫「攻擊者可在 5 分鐘內竊取所有客戶的信用卡資訊，估計監管罰款風險超過 HK$200 萬」。
04
修復優先順序建議（表格或清單） — 分三個時間段：立即行動（24–48 小時，針對 Critical）、短期（2 週內，針對 High）、中長期（1–3 個月，針對 Medium/Low）。讓管理層清楚行動計劃。
05
正面發現（選擇性） — 提及組織做得好的地方（如「密碼策略符合行業標準」「WAF 有效攔截了自動化掃描工具」）。讓報告看起來更均衡，而非純粹批評。
執行摘要段落範本（節錄）
【整體評估】

本次評估中，測試團隊成功滲透 XYZ 公司核心電子商務平台，取得包含
150 萬筆客戶姓名、地址及加密支付資訊的資料庫存取權限，並取得生產
環境伺服器的最高管理員控制權。此等級別的訪問使攻擊者能夠在不留痕
跡的情況下，長期監控或竊取客戶資料，甚至部署勒索軟件。

整體安全狀態：高風險（評分 4/5）

【關鍵發現（依業務影響排序）】

  1. 客戶資料庫完全暴露（嚴重）
     — 攻擊者可在 5 分鐘內提取所有客戶 PII
     — 監管合規風險：PDPO 第 64 條罰款最高 HK$50,000 及刑責
     — 建議：立即修復（24 小時內）

  2. 生產伺服器管理員控制權（嚴重）
     — 可部署勒索軟件或後門，業務中斷風險極高
     — 估計業務停擺損失：每日 HK$120 萬
     — 建議：立即修復（24 小時內）

  3. 員工帳號弱密碼（高）
     — 32% 的員工帳號可在 2 小時內被暴力破解
     — 任何已入侵的內部帳號均可橫向移動至核心系統
     — 建議：短期修復（2 週內）

【修復建議】

  立即（48 小時）：修補 SQL Injection 漏洞；撤銷洩露的管理員憑證
  短期（2 週）：強制所有帳號啟用雙因素認證；更新密碼策略
  中期（1 個月）：導入 WAF 規則；修復所有 High 級別漏洞

漏洞描述與風險評級：CVSS v4

CVSS（Common Vulnerability Scoring System）v4.0 是 2023 年 11 月發布的最新版本漏洞嚴重程度評分標準，用於標準化漏洞影響評估，讓客戶能夠跨不同漏洞進行比較和優先排序。

CVSS 評分嚴重程度顏色修復 SLA 建議典型範例
9.0 – 10.0Critical（嚴重）深紅立即（24–48 小時）無認證 RCE、SQL Injection 直達資料庫、預設管理員憑證
7.0 – 8.9High（高）橙紅短期（7–14 天）需認證 RCE、敏感資料洩露（無加密）、SSRF 訪問內網
4.0 – 6.9Medium（中）橙黃中期（30–60 天）CSRF、反射型 XSS、目錄遍歷（無敏感文件）、資訊洩露
0.1 – 3.9Low（低）黃綠長期（90 天內）版本資訊洩露、弱 Cookie 設定、過期 TLS 版本（TLS 1.1）
0.0Informational（資訊）灰藍最佳實踐建議未使用 HTTPS Strict-Transport-Security、缺少 X-Frame-Options 等

標準漏洞發現描述模板

漏洞發現記錄模板（SQL Injection 範例）
漏洞標題： 用戶搜索功能 SQL Injection（SQLi）— 可未授權讀取資料庫

嚴重程度： Critical（嚴重）
CVSS v4 評分： 9.8
CVSS 向量： CVSS:4.0/AV:N/AC:L/AT:N/PR:N/UI:N/VC:H/VI:H/VA:H/SC:N/SI:N/SA:N

受影響系統：
  - https://shop.example.com/search?q=
  - 後端資料庫：MySQL 8.0.32（shop_db）

漏洞描述：
  搜索功能的 q 參數未對用戶輸入進行適當的過濾或參數化處理，
  導致攻擊者可以注入任意 SQL 語句。測試人員利用此漏洞，在無需
  任何認證的情況下，成功提取了 customers 表中的 1,521,408 筆
  用戶記錄，包含姓名、電子郵件、電話號碼及加密後的支付卡資料。

重現步驟（PoC）：
  1. 訪問 https://shop.example.com/search
  2. 在搜索框輸入以下 Payload：
     ' UNION SELECT null,username,password,email,null FROM customers--
  3. 搜索結果頁面顯示客戶帳號資訊

  驗證命令（sqlmap）：
  sqlmap -u "https://shop.example.com/search?q=test" --dbs --batch

業務影響：
  - 直接：所有客戶個人資料（PII）完全洩露
  - 合規：違反 PDPO（個人資料（私隱）條例）第 4 原則
  - 財務：潛在監管罰款 + 客戶賠償 + 聲譽損失

修復建議（短期）：
  立即改用參數化查詢（Prepared Statements）：
  $stmt = $pdo->prepare("SELECT * FROM products WHERE name LIKE ?");
  $stmt->execute(["%$search%"]);

修復建議（長期）：
  - 全面代碼審計，確保所有資料庫查詢均使用 ORM 或參數化語句
  - 部署 WAF 規則攔截 SQL Injection 嘗試
  - 實施最小權限原則，資料庫帳號只給予必要的讀寫權限

參考資料：
  - CWE-89: Improper Neutralization of Special Elements in SQL Commands
  - OWASP Top 10 2021 A03: Injection
  - OWASP Testing Guide v4.2, Section 4.7.5

修復建議與複測追蹤

優質的修復建議是報告差異化的關鍵。不僅告訴客戶「有什麼問題」，更要清楚說明「如何修復」、「修復後如何驗證」，以及追蹤修復進度的機制。

短期修復（Immediate Fix）
針對 Critical/High 漏洞。提供具體的代碼或配置修改範例，讓開發者能立即行動。例如：提供修復後的代碼片段、需要更改的配置行、需要更新的套件版本號。避免籠統說「修復漏洞」或「加強安全」。
長期建議（Strategic Fix）
根本原因的解決方案和系統性改進，例如：引入 SAST 工具到 CI/CD 流程、建立安全代碼審查規範、導入開發者安全培訓計劃。這部分體現了顧問價值，超越單純的「打補丁」。
驗證方法（Verification）
告訴客戶「如何確認已修復」：提供重現測試的具體步驟，或提供簡單的驗證腳本。讓客戶的內部技術團隊在修復後能自行驗證，不依賴外部顧問。
複測（Retest / Verification Test）
在報告交付 2–4 週後，提供免費或付費的複測服務，驗證所有 Critical/High 漏洞已被正確修復，並出具複測報告（Retest Report）。這是建立長期客戶關係的重要環節。
修復建議的品質差異

差的建議：「建議修復 SQL Injection 漏洞，加強輸入驗證。」
好的建議：「將 searchProducts($q) 函式第 23 行的字串拼接改為參數化查詢（見下方代碼範例）；同時在 MySQL 設定中為 app_user 帳號撤銷 DROP 和 CREATE 權限，使其符合最小權限原則。建議在 PHP 7.4+ 環境使用 PDO 參數化查詢，修復後執行附錄 B 的驗證腳本確認修復有效。」

報告品質把關清單

在提交報告前，用以下清單進行最終品質審查。一份專業的報告是公司形象和技術公信力的直接體現。

01
敏感資訊脫敏 — 截圖中的真實密碼、個人資料、信用卡號碼已遮蔽（馬賽克/替換為 ****）；PoC 代碼中不包含可直接部署的 Weaponized Exploit；客戶環境的 IP 地址在必要時替換為示意性地址。
02
技術準確性 — 所有漏洞已由第二名測試人員（Peer Review）驗證可重現；CVSS 評分已核對計算；受影響版本和系統資訊準確；CVE/CWE 引用正確。
03
完整的證據鏈 — 每個漏洞都有截圖或日誌作為證據；截圖包含時間戳和 URL；重現步驟清晰，另一名技術人員能獨立重現；Payload 已記錄在案。
04
讀者一致性 — 執行摘要使用業務語言，無縮寫術語（或有術語表）；技術部分使用精確的技術語言；全報告術語一致（不要交替使用「漏洞」「安全弱點」「Issue」指代同一概念）。
05
修復建議可執行性 — 每個發現都有具體、可操作的修復建議（不是籠統的「改善安全」）；Critical/High 發現有代碼範例；提供了驗證步驟；標明了修復的優先順序和時間要求。
06
格式與呈現 — 文件有頁碼、目錄、版本號；所有截圖標注清晰（說明截圖內容的標注文字）；嚴重程度顏色標記一致；PDF 已加密（開啟需要密碼）並設定不可複製/列印限制。
小測驗
你正在撰寫針對一家零售企業的滲透測試報告，發現了一個 SQL Injection 漏洞，可以提取所有客戶資料。在執行摘要中，以下哪種表述方式最適合？
A「在 /search 端點的 q 參數中發現了 UNION-based SQL Injection，CVSS 評分 9.8，CWE-89，攻擊者可使用 sqlmap 提取 customers 表。」
B「測試人員在 5 分鐘內無需任何帳號即可下載所有 150 萬筆客戶個人資料，違反 PDPO 合規要求，潛在監管及訴訟風險逾 HK$500 萬。」
C「在搜索功能中發現嚴重漏洞，建議立即修復，避免資料洩露。」
D「發現 SQL Injection 漏洞，這是 OWASP Top 10 的 A03 注入類別，屬於最常見的 Web 安全問題之一，已有成熟的修復方案。」
正確！執行摘要的對象是管理層，他們不需要技術術語（選項 A 的 SQL Injection / UNION-based / CWE 這些），也不需要教育他們 OWASP 是什麼（選項 D）。他們需要知道的是：業務影響有多大（150 萬筆客戶資料）、時間維度（5 分鐘）、合規和財務風險（PDPO 罰款）。選項 C 太籠統，無法讓管理層感受到危急程度，也無法推動他們批准修復預算。

認證全景圖：入門到頂尖

資安認證是職涯發展的加速器，但選錯認證會浪費大量時間和金錢。以下是依職涯階段整理的認證路徑圖，幫助你做出最有效率的選擇。

CompTIA Security+ 入門
資安領域最廣泛認可的入門認證。涵蓋網路安全、威脅管理、密碼學等基礎知識。許多企業和政府機構要求 IT 人員持有此證。適合轉行者作為第一張認證。
考試費 ~US$392 備考 2–3 個月 無前提
CEH（Certified Ethical Hacker） 入門-中級
EC-Council 出品的倫理黑客認證。知名度高，但在業界實戰評價兩極——知識涵蓋廣但偏理論，多選題形式。對政府/軍事/合規型工作有一定加分，實戰黑客社群評價較低。
考試費 ~US$950 備考 2–3 個月 2年資安工作經驗或培訓
OSCP（Offensive Security Certified Professional） 中級（最推薦）
業界最受尊重的實戰滲透測試認證。24 小時全實機考試（非多選題），需在真實環境中獲取多台機器的 proof.txt。通過率僅 ~45%。幾乎所有頂尖滲透測試職位的敲門磚。
課程+考試 ~US$1,499 備考 3–6 個月 需有滲透測試基礎
CRTO（Certified Red Team Operator） 中級
Zero-Point Security 出品的 Cobalt Strike 紅隊認證，重點是 Active Directory 攻擊和真實 C2 框架使用。48 小時考試，需完成 Active Directory 完整攻擊鏈。性價比極高，課程品質優秀。
課程+考試 ~GBP 449 備考 1–2 個月 建議先有 AD 基礎
OSEP / OSED / OSMR 高級
Offensive Security 的進階認證系列：OSEP（逃避性技術和高級滲透）、OSED（Windows Exploit 開發）、OSMR（macOS 研究員）。每張均為實機 48 小時考試，難度極高，是 OSCP 之後的進階路徑。
各 ~US$1,499 備考 3–6 個月 需先通過 OSCP
CISSP 管理層
ISC² 出品的資安管理認證，適合資安主管/CISO 職位。涵蓋 8 大安全領域（CBK），偏向策略和管理而非技術實戰。要求 5 年相關工作經驗，是晉升管理層的標準認證。
考試費 ~US$749 備考 3–6 個月 需 5 年工作經驗
GXPN / GREM（GIAC 系列） 高級
SANS/GIAC 的高階認證：GXPN（利用開發，含 Exploit Dev）、GREM（逆向工程惡意軟件）。SANS 課程品質頂尖但費用昂貴（US$7,000–9,000），適合有公司培訓預算的資深人員。
課程 US$7,000+ 備考 2–3 個月 需資深技術背景
CREST / TIGER Scheme（英港澳政府認可） 頂級合規
英國/香港政府機構採購滲透測試服務的強制要求。CREST 認可的測試人員和公司才能承接政府、金融、關鍵基礎設施合約。香港金融管理局（HKMA）相關要求下常被指定。
考試費 GBP 200–600 需大量實戰經驗 香港/英國政府合約必備

OSCP 考試策略與備考計劃

OSCP（Offensive Security Certified Professional）是業界最受認可的實戰滲透認證。考試為 24 小時不間斷實機滲透，需要在 Offensive Security 提供的網路中獲取指定數量的 proof.txt 文件。

考試結構（截至 2024 年格式）

目標分值要求
Active Directory 完整攻擊鏈40 分需完整攻擊鏈（3 台機器：Client + DC）；全拿或全不給，不能部分得分
獨立靶機 × 3各 20 分（共 60 分）每台機器各有 local.txt（10分）和 proof.txt（10分）；需截圖含 IP + whoami

  1. 先完整攻破 AD 鏈（40 分保底）
  2. 再攻兩台獨立機（+40 分，共 80 分，及格）
  3. 剩餘時間嘗試第三台獨立機（+20 分）

備考核心建議

OSCP 考核的是「方法論」而非「工具技巧」。許多人考試失敗不是因為不會漏洞利用，而是因為資訊收集不夠系統化（漏掃某個端口）或卡在一台機器太久（超過 2 小時沒進展應立即換機器）。建立並堅守你的枚舉 Checklist，比掌握更多工具更重要。

CEH / CISSP / CompTIA 認證定位

不同認證在不同職業場景有不同的價值，選擇時需要考慮目標職位和雇主偏好，而非純粹追求「最難」的認證。

01
CompTIA Security+：進入資安領域的「通行証」。不強調任何特定方向，適合轉行者、IT 人員轉資安，以及政府/軍事/合規型組織（美國 DoD 8570 要求）。香港方面，部分跨國銀行和顧問公司將其視為 Junior 職位的基本要求之一。
02
CompTIA PenTest+：定位介於 Security+ 和 OSCP 之間，但業界認可度遠不如 OSCP。若預算有限且無法立即投入 OSCP，可作為過渡，但長遠建議仍以 OSCP 為目標。
03
CEH（Certified Ethical Hacker）：適合政府採購、合規需求，或某些需要「認可認證清單」的企業客戶。在漏洞賞金和滲透測試社群評價兩極，但在中東、東南亞、印度的企業和政府市場仍有相當認可度。
04
CISSP：資安管理層的「必需品」，適合目標晉升至 CISO/資安主管的人員。不要把它和技術滲透認證比較，它是不同職位要求不同認證的典型範例。5 年工作經驗要求使其成為職涯中期後的目標。
05
CISM（Certified Information Security Manager）：ISACA 出品，與 CISSP 類似但更聚焦於安全管理和資訊風險治理，在金融業（香港銀行業常見）和企業資安主管職位有良好認可度。
認證選擇策略

若目標是滲透測試技術崗位：OSCP 是核心，其他認證是補充。若目標是政府/金融合規崗位：CREST + CISSP/CISM 組合最有效。若目標是管理晉升：CISSP 或 CISM + MBA 路線。不要為了認證而認證——雇主更看重真實的 HTB/CTF 成績單和 GitHub 代碼庫，遠勝於一張不具備實戰能力的認證卡。

滲透測試工程師職涯路徑

滲透測試是資安領域中技術要求最高、薪資增長最快的方向之一。以下是從零開始到資深的典型職涯發展路徑，以及每個階段的里程碑。

Year 0
學習基礎 & 建立作品集
完成 SEC_TRACK 課程；TryHackMe/HTB Starting Point；取得 CompTIA Security+ 或 eJPT；在 GitHub 發布工具腳本；開始 CTF 參賽（建議：picoCTF → CTFtime 新手比賽）。
Year 1
初級滲透測試工程師（Junior Pentester）
HTB Pro Hacker 排名；備考並通過 OSCP（最重要里程碑）；加入顧問公司或資安部門，在資深工程師指導下執行測試；撰寫第一份正式報告；薪資 HK$22,000–30,000。
Year 2–3
中級滲透測試工程師（Pentester）
獨立執行完整滲透測試項目；發現並負責任披露至少 1 個 CVE；取得 CRTO 或 CRTE；開始主導客戶溝通和匯報；薪資 HK$32,000–50,000；選擇深化方向（Web/AD/移動/IoT）。
Year 4–6
資深滲透測試工程師（Senior Pentester）
領導多人測試團隊；開發自定義工具；在 DEF CON/BlackHat 發表演講或投稿；多個 CVE；OSEP 或 GXPN；年薪 HK$600,000–900,000；可選擇專職漏洞研究或轉向管理。
Year 7+
資安主管 / 獨立顧問 / 創業
滲透測試總監（Director of Penetration Testing）；創立自己的資安顧問公司；成為專職漏洞研究員（Bug Bounty / Vuln Research 收入）；CREST 認可測試人員；CISSP；年薪 HK$900,000+。

漏洞賞金獵人（Bug Hunter）

漏洞賞金（Bug Bounty）是滲透測試工程師的重要副業或全職發展方向。頂尖的 Bug Hunter 年收入可達數十萬美元，但入門門檻比想像中高——大多數人在開始的幾個月內找不到任何有效漏洞。

主要平台
HackerOne：最大的 Bug Bounty 平台，雇主包含 Twitter/X、GitHub、美國國防部等。Bugcrowd：企業客戶為主，包括 Tesla、Mastercard。Intigriti：歐洲主要平台。Synack：邀請制，通過審核的研究者有保障性收入。各廠商自辦計劃：Google、Apple、Microsoft、Meta 有最高賞金潛力。
起步策略
不要從 Google 或 Apple 的計劃開始（競爭極激烈）。選擇範圍廣、新上線的計劃（新程式漏洞密度高）；聚焦你最熟悉的漏洞類型（先用 IDOR 或 BAC 建立信心）；閱讀其他人的 HackerOne Disclosed Reports 學習思路；先確保報告品質而非數量。
高賞金漏洞類型
RCE（遠端代碼執行）：通常 $5,000–$100,000+；SSRF 導致雲端後設資料洩露：$5,000–$30,000；認證繞過/帳號接管（ATO）：$1,000–$15,000；SQL Injection（在範圍內）：$500–$10,000；IDOR 訪問他人私有資料：$500–$5,000。
現實期望管理
大多數 Bug Hunter 前 6 個月收入接近零——這是正常的學習曲線，而非失敗。年收入超過 US$10,000 的 Hunter 約佔所有活躍 Hunter 的 5%。年收入超過 US$100,000 的屬於頂尖（約 1%）。建議先作為副業（不依賴其收入），再逐步提升。

作品集建立：GitHub / Blog / CVE

在資安領域，可見度（Visibility）是職涯發展的關鍵。雇主越來越重視候選人的「公開足跡」——GitHub 代碼庫、技術部落格、CTF 戰績、CVE 記錄。這些比任何認證更能直接展示你的實際能力。

01
GitHub 作品集 — 保持至少 5–10 個高品質公開倉庫：自己開發的滲透測試工具（即使是小工具）、CTF 解題 Writeup 倉庫、課程學習筆記、自動化腳本。README 要詳細（工具用途/安裝/使用範例）。GitHub Contribution Graph 越綠越好——展示持續學習的習慣。
02
技術部落格 — 部落格是最強力的個人品牌工具。每篇文章可以是：CTF 題目詳細 Writeup（含解題思路）、漏洞研究筆記（即使是別人發現的 CVE 的分析）、工具使用心得、概念深入解析。平台選擇：Medium（曝光度高）、GitHub Pages（可自定域名）、Ghost（專業外觀）。不需要每天更新——每月 1–2 篇高品質文章效果優於每天發短貼。
03
CVE 發現與披露 — 擁有一個自己發現的 CVE 是資安履歷上最有力的加分項之一。入門策略：從你日常使用的開源工具入手進行代碼審計；或在授權的 Bug Bounty 計劃中發現漏洞（部分平台會協助申請 CVE）。目標：在入行後 2–3 年內取得第一個 CVE。
04
CTF 戰績記錄 — 在 CTFtime.org 有公開戰績；HTB Profile 和 Rank（Elite Hacker 以上）；TryHackMe 排名。這些是對技術能力的客觀第三方佐證，比自吹自擂的履歷更有說服力。
05
社群貢獻 — 在 Reddit（r/netsec、r/HowToHack）、Discord 資安社群（HTB、TryHackMe、PWNBUF）中回答問題；在 OWASP 本地分部或資安 Meetup 發表演講（即使是 10 分鐘的閃電演講）；為開源工具提交 PR。可見度和人際網路對找工作有非常實際的幫助。

面試準備：技術題與行為題

資安面試通常分為技術面和行為面兩個部分。技術面測試你的實際能力，行為面評估你的職業態度和道德標準。

常見技術面試題（含建議回答方向）

滲透測試面試題精選（答題框架）
Q1: 「如果你在測試一個 Web 應用程式，你的第一步是什麼？」
答題要點：展示方法論，而非直接跳到工具
→ 授權確認 → 被動偵察（WHOIS/DNS/Shodan）→ 主動偵察（子域名/端口掃描）
→ 技術棧識別 → 功能性測試（認證/授權/業務邏輯）→ 自動化掃描
→ 手動漏洞驗證 → PoC 構建 → 文件記錄

Q2: 「解釋 OWASP Top 10 的 A01:2021 失效存取控制」
答題要點：定義 + 真實範例 + 修復建議三段式
→ 定義：認證用戶能夠訪問不應被授權的資源或操作
→ 範例：水平越權（IDOR）、強制瀏覽（Forced Browsing）
        垂直越權（普通用戶訪問管理員功能）
→ 修復：在服務端對每個請求重新驗證權限（不依賴客戶端狀態）
        實施最小權限原則；記錄並警報異常訪問嘗試

Q3: 「你如何在不觸發 AV/EDR 的情況下執行 Payload？」
答題要點：展示深度但也展示謹慎
→ 技術層面：PE 載入器（Shellcode Loader）、進程注入（Process Injection）
              AMSI Bypass、Living-off-the-Land（LOLBins）
→ 關鍵補充：「在實際滲透測試中，我們通常先與客戶確認是否在測試範圍內
              包含 AV/EDR 繞過；一些客戶希望測試防禦能力，另一些客戶
              只希望假設已繞過 AV 的場景。」
→ 展示：知道邊界和授權的重要性

Q4: 「SQL Injection 的類型有哪些，你如何在沒有工具的情況下測試？」
→ 類型：基於錯誤（Error-based）、盲注（Blind: Boolean/Time-based）
        UNION-based、帶外（OOB / DNS）
→ 手動測試：
  1. 加入單引號 ' 看是否報錯
  2. OR 1=1-- 測試 Boolean 邏輯
  3. ' OR SLEEP(5)-- 測試時間盲注（回應延遲 5 秒）
  4. ' UNION SELECT NULL,NULL-- 測試欄位數

Q5: 「你在測試中意外發現超出授權範圍的嚴重漏洞，你怎麼做？」
答題要點：這是職業道德測試題，標準答案非常重要
→ 立即停止對該範圍的測試
→ 記錄發現（含時間戳），但不繼續利用
→ 立即通知客戶聯絡人（電話 + 電郵）
→ 在獲得明確書面擴大授權前不繼續
→ 在報告中如實記錄此發現和行動

常見行為面試題

01
「你如何保持技術的持續更新？」 — 具體提及：定期閱讀的 Blog/Twitter 帳號（Project Zero、PortSwigger Research）、參加的 CTF 比賽、最近完成的 HTB/TryHackMe 機器、訂閱的安全週報（tl;dr sec、This Week in Security）。

「描述一次你在工作中犯錯的經歷以及如何處理」 — 用 STAR 框架（Situation/Task/Action/Result）回答。展示自我認知、主動溝通錯誤、從中學習的成熟度。資安職位最怕面試者掩飾錯誤或沒有自知之明。
03
「如果客戶要求你測試一個不在授權清單內的系統？」 — 標準答案：請客戶修訂書面授權後才能繼續。這類問題測試你的職業道德底線——雇主需要確認你不會因客戶施壓而越界操作。
小測驗
在滲透測試面試中，面試官問：「你在對一個 Web 應用程式進行授權測試時，意外發現了另一個不在授權範圍內的子域名有嚴重漏洞。你會怎麼做？」以下哪個回答最符合職業標準？
A「我會先快速確認漏洞的嚴重程度，取得 proof.txt 作為證明，然後再通知客戶。」
B「我會等到報告提交時一併告知客戶，以免打亂測試流程。」
C「立即停止對範圍外系統的任何操作，僅記錄發現（不利用），立即電話通知客戶緊急聯絡人，並等待書面授權後才決定下一步。」
D「子域名屬於同一個客戶，我會繼續測試以給他們更完整的安全報告，這對客戶更有價值。」
正確！選項 C 是唯一符合職業操守和法律要求的答案。未授權的系統測試在絕大多數司法管轄區是違法行為——即使是同一個客戶的資產，在書面授權明確前也不能進行。選項 A 的「先利用再通知」和選項 D 的「因為是同一客戶所以繼續」是最危險的錯誤，可能導致刑事責任。選項 B 的「等到報告時再說」在發現嚴重漏洞的情況下是不可接受的延誤——客戶有權利在漏洞被修復前知道其暴露情況。面試官問這類問題，是在篩選是否會因客戶壓力或追求更好成果而放棄道德底線的候選人。

CTF 參賽策略與資源地圖

CTF（Capture The Flag）比賽是學習資安技術最有趣、最有效率的方式，同時也是建立技術聲望和人脈的重要途徑。有策略地選擇比賽和組建隊伍，可以最大化學習效率。

資安倫理與職業操守

技術能力決定你能做什麼，職業道德決定你成為什麼樣的資安從業者。資安是少數幾個能力越強、潛在危害越大的領域，因此職業倫理不是選擇題，而是從業的基本條件。

01
授權原則（永遠是第一條） — 只在明確書面授權的範圍內執行任何測試或研究。口頭授權、電郵許可、「我以為在範圍內」均不構成合法授權。沒有書面 SOW（Statement of Work）或 ROE（Rules of Engagement），就是未授權訪問。在大多數地區，這是刑事罪行，不是民事糾紛。
02
最小必要原則 — 在完成測試目標的前提下，使用最小的侵入性。不需要 exfiltrate（外洩）真實資料來證明 SQL Injection 存在——截取部分資料示意即可。不要在生產系統上執行可能導致服務中斷的 DoS 型 Exploit，除非客戶明確授權並接受風險。
03
資料保護義務 — 測試過程中接觸到的客戶數據（用戶資料、財務資訊、商業機密）必須嚴格保密。測試完成後應安全刪除（不保留客戶敏感數據副本）。報告中的截圖需對敏感資訊進行適當遮蔽。違反保密義務不僅有法律責任，也會永久損害個人和公司聲譽。
04
責任披露（Responsible Disclosure） — 在 Bug Bounty 或漏洞研究中發現的漏洞，必須遵守責任披露原則：聯繫廠商，給予修復時間（通常 90 天），再公開披露。不得在修復前以威脅要求金錢報酬（這是勒索，不是 Bug Bounty），也不得將漏洞出售給第三方用於攻擊。
05
拒絕灰色用途 — 拒絕接受任何要求對真實第三方（非授權測試目標）實施攻擊的工作，無論報酬多高。包括：「幫我看看我前公司的系統有沒有漏洞」、「幫我測試競爭對手的網站」、「這只是研究目的」。這些委托不會因意圖而改變其非法性質。
06
能力邊界的誠實 — 不接受超出自己能力範圍的項目，或明確說明技術限制。在報告中如實呈現測試的局限性（「本次測試為黑盒，未測試內部網路」）。資安顧問的信譽建立在誠實上，誇大測試範圍或能力最終會傷害自己和客戶。
香港適用法律參考

香港的《電腦罪行條例》（第 200 章）第 27 條（未經授權訪問電腦程式或數據）和第 29 條（未經授權損害電腦）均為刑事罪行，最高刑罰分別為監禁 2 年和 5 年。英美兩地的 Computer Fraud and Abuse Act（美）和 Computer Misuse Act（英）同樣適用於針對其境內系統的攻擊，即使攻擊者身處香港。「我以為是公開系統」或「我只是在測試」均不構成辯護理由。合規的書面授權是唯一保護。
